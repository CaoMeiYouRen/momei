# 国际化系统 (Internationalization)

## 1. 概述 (Overview)

墨梅博客是一个“原生国际化”驱动的平台。国际化深入到了 UI 界面、内容管理、后端逻辑以及邮件通知中心。

## 2. 前端国际化 (Frontend i18n)

### 2.1 架构设计
基于 **Vue-i18n** 与 **PrimeVue**。
- **动态切换**: 支持无刷新切换语言，通过 `$t()` 进行业务文案展示。
- **组件同步**: PrimeVue 的内置文案（如日历、分页）通过全局插件自动与系统语言同步。
- **配置中心化**: 当前已引入 `Locale Registry`，由 `i18n/config/locale-registry.ts` 统一维护启用语言、默认语言、路由前缀与语言就绪度。
- **模块化消息加载**: 当前前端翻译资源已从单文件模式迁移为 `i18n/locales/<locale>/<module>.json` 多文件结构，并通过 Nuxt i18n `lazy + files[]` 方式按模块装配。

## 3. 内容国际化 (Content i18n)

- **语言标识**: 文章、分类、标签实体军带有 `language` 字段。
- **翻译关联**: 同源的多种语言文章通过 `translationId` 进行逻辑聚合。

## 4. 后端全量国际化 (Backend i18n)

在第六阶段，系统实现了基于 Nitro 钩子的后端全量国际化支持：

### 4.1 自动语言感知
后端工具类 `server/utils/i18n.ts` 能够自动从请求中提取 `Accept-Language` 或特定 Header。

### 4.2 服务端消息装配

- 服务端翻译加载器当前与前端共享同一套 Locale Registry。
- `server/utils/i18n.ts` 会按模块合并读取 `common`、`components`、`pages`、`feed` 等消息文件，并对非默认语言进行默认语言回退装配。

### 4.3 核心覆盖领域
- **RSS/Atom Feed**: 根据请求参数 (`/feed/en-US.xml`) 自动生成对应语言的 XML。
- **邮件系统**: 邮件模板（如注册确认、评论回复）根据用户的 `preferredLanguage` 自动切换翻译。
- **错误码与响应**: 后端统一返回的错误消息支持多语言。

## 5. 开发规范
- **禁止硬编码**: 禁止在代码中使用中/英文字符串，必须通过 `i18n/locales/` 下的 JSON 文件定义。
- **强一致性**: 前后端翻译 Key 尽量保持命名空间一致，方便维护。

## 6. 第十阶段语言扩展设计 (Phase 10 Language Expansion)

### 6.1 本轮目标

- 在现有 `Locale Registry` 基础上新增 `zh-TW` 与 `ko-KR`。
- 首轮覆盖公开页面、认证、核心设置、Demo 与 Onboarding 关键链路。
- 允许用户在前端切换新语言，但在翻译治理与邮件/SEO 回归完成前，不进入搜索引擎索引与 sitemap。

### 6.2 新语言接入策略

本轮不直接把新增语言提升为 `seo-ready`，而是采用“先 `ui-ready`，后 SEO 放量”的策略。

| Locale | routePrefix | readiness | fallbackChain | switchable | indexable | sitemapEnabled | feedEnabled |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| `zh-CN` | `` | `seo-ready` | `zh-CN` | 是 | 是 | 是 | 是 |
| `en-US` | `/en-US` | `seo-ready` | `en-US -> zh-CN` | 是 | 是 | 是 | 是 |
| `zh-TW` | `/zh-TW` | `ui-ready` | `zh-TW -> zh-CN -> en-US` | 是 | 否 | 否 | 否 |
| `ko-KR` | `/ko-KR` | `ui-ready` | `ko-KR -> en-US -> zh-CN` | 是 | 否 | 否 | 否 |

说明：

- `zh-TW` 与 `zh-CN` 语义最接近，缺词优先回退到简体中文。
- `ko-KR` 的 UI 与产品提示缺词优先回退到英文，再回退到简体中文，减少机器感过强的中文兜底。
- `ui-ready` 语言仍会参与前端切换、页面渲染、PrimeVue 组件内置文案同步与服务端翻译装配。
- `ui-ready` 语言不会被 `usePageSeo()` 视为可索引语种，也不会进入 sitemap 与 feed alternates。

### 6.3 首轮覆盖边界

首轮必须补齐以下模块：

- 核心模块：`common`、`components`、`public`、`settings`、`legal`
- 关键链路模块：`auth`、`home`、`demo`
- 按需保障：`installation`、`admin`、`feed` 至少保持 key parity，避免运行时缺词

链路要求：

- [登录页](../../../pages/login.vue) 的 `auth` 模块必须在首屏预加载后即可完整显示按钮与表单文案。
- [语言切换器](../../../components/language-switcher.vue) 必须能稳定展示 `zh-TW` 与 `ko-KR` 的显示名称。
- [Demo / Onboarding](../../../composables/use-onboarding.ts) 使用的 `demo.*`、`common.*` 词条必须在新增语言下完整可用。
- [PrimeVue i18n 插件](../../../plugins/primevue-i18n.ts) 必须同步对应 locale，避免组件内置文案回退到英文。

### 6.4 翻译治理与发布门禁

新增语言从 `ui-ready` 升级到 `seo-ready` 前，必须同时满足以下条件：

1. `i18n/locales/<locale>/` 下 11 个模块全部完成 key parity。
2. `scripts/i18n/audit-locale-keys.mjs --fail-on-missing` 无缺词报错。
3. 前端关键链路通过定向测试：locale registry、模块加载、登录页、Onboarding。
4. 服务端邮件 locale 已具备同等字段覆盖，避免 UI 已翻译但邮件仍回退到默认语言。
5. `usePageSeo`、sitemap、feed 输出已确认不会泄露 `ui-ready` 语种到索引面。

### 6.5 受影响文件清单

- `i18n/config/locale-registry.ts`: 扩展语言元数据、别名与回退链。
- `plugins/primevue-i18n.ts`: 补齐 `zh-TW`、`ko-KR` 的 PrimeVue locale 映射。
- `i18n/locales/zh-TW/*.json` 与 `i18n/locales/ko-KR/*.json`: 新语言词条。
- `scripts/i18n/audit-locale-keys.mjs`: 继续作为 parity 审计门禁。
- `server/utils/email/locales/index.ts`: 邮件语言白名单需与语言准入策略保持一致。
