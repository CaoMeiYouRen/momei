# AI 初始化 / 配置问答助手评估

## 1. 概述

本文档用于评估“是否为新部署者提供 AI 初始化 / 配置问答助手”，并冻结本轮可接受的范围、事实源复用方式、安全边界、失败回退和下一阶段产品判断。

本评估只回答以下问题：

- 是否值得继续推进最小原型。
- 原型最多能回答什么、绝不能做什么。
- 现有安装向导、系统设置、只读问答能力与文档知识源能否直接复用。
- 若进入下一轮实现，最小可行路径应如何收敛。

本评估不直接承诺站内全功能 AI 助手，也不引入新的配置模型、第二套知识库或新的写配置通道。

## 2. 现有事实源与可复用资产

当前仓库已经具备做“启用式、只读、解释型”问答助手的主要基础设施，无需从零设计新的配置事实源。

### 2.1 安装向导与初始化分层

- 安装向导当前已经拆分为健康检查、数据库、站点配置、管理员账号、预览、完成六步流程。
- 安装向导的最小站点配置模型已经由 `InstallationSiteConfigModel` 固化，包含站点标题、描述、关键词、URL、版权和默认语言等核心字段。
- 可延后增强项已经由 `InstallationExtraConfigModel` 固化，覆盖 AI、邮件、存储、统计等配置。
- `getInstallationChecklist()` 已经区分“现在就该做”和“安装后再做”的事项层级，可直接复用为问答助手的回答分层依据。
- 安装向导已有环境变量回填映射，说明当前表单与设置键之间已经存在稳定映射，不需要额外维护一套“助手专用字段表”。

### 2.2 系统设置元信息与锁定语义

- 系统设置当前统一遵循 `ENV -> DB -> Default` 的三层读取模型。
- `/api/admin/settings` 已返回 `source`、`isLocked`、`envKey`、`lockReason`、`requiresRestart` 等解释层字段。
- `FORCED_ENV_LOCKED_KEYS` 已冻结启动期只读字段，当前至少包括 `SITE_URL`、第三方登录和验证码相关键。
- `INTERNAL_ONLY_ENV_KEYS` 与 `INTERNAL_ONLY_SETTING_KEYS` 已定义完全不应暴露给前端问答层的敏感项。
- `resolveInstallationEnvLockMessage()` 已把锁定原因映射为用户可理解文案，可直接复用为“为什么这里不能改”的回答模板。

### 2.3 现有问答与文档知识源

- `qa-assistant` 已被定义为只读检索和证据化回答角色，适合作为底层知识检索能力复用。
- README、快速开始、部署指南、变量映射、系统能力设计文档已覆盖新部署者高频问题。
- 安装向导与后台设置当前已有完整的多语言提示、handoff 文案和失败反馈，不需要为助手额外维护另一套解释文本。

## 3. 评估结论

结论：建议进入下一轮“最小原型”实现，但范围必须严格收敛为只读解释型助手，不建议当前直接立项为可写配置或全站通用 AI 助手。

判断依据如下：

1. 当前事实源已经足够支撑“解释配置、指路排障、区分必填与延后项”的核心目标，复用成本低于新建一套安装帮助系统。
2. 新部署者最常见的痛点是“看不懂字段、分不清先后、遇到锁定项不知下一步”，这些都能通过现有安装向导、设置元信息与文档直接回答。
3. 一旦助手具备直接写配置、代填敏感字段或越权调用后台接口的能力，风险会迅速从“解释错误”升级为“配置破坏、权限穿透、秘密泄露”，当前不适合跨这条边界。

因此，本轮产品判断为：

- 允许进入下一轮正式实现的，仅限“启用前引导 + 配置解释 + 文档跳转 + 排障分流”的最小原型。
- 不建议进入下一轮正式实现的，包括“自动写 `.env`”、“代管管理员初始化”、“跨模块自由问答”、“全站 AI Copilot 化配置中心”。

## 4. 范围冻结

### 4.1 允许回答的范围

原型阶段只允许回答以下问题：

- 当前部署路径下哪些配置是核心必填项，哪些可以安装后补齐。
- 某个字段的用途、来源、是否锁定、为何锁定、是否需要重启才能生效。
- 某类错误或校验失败更可能属于哪一层：环境变量缺失、配置冲突、平台不支持、输入格式不合法。
- 本地开发、Vercel、Docker、自托管 Node 四类高频路径下，下一步最应该去哪个页面、哪个步骤或哪份文档。
- 当前安装阶段应继续走安装向导，还是转入后台系统设置。

### 4.2 原型阶段禁止的范围

原型阶段明确禁止以下能力：

- 直接写入 `.env`、宿主机环境变量、密钥文件或任何部署平台 Secret。
- 直接调用写接口保存敏感配置，或绕过当前页面权限边界替用户修改系统设置。
- 回答与初始化 / 配置无关的全站通用问题，避免把本能力扩写为站内万能问答入口。
- 暴露 `INTERNAL_ONLY` 字段、脱敏前密钥值、数据库连接串、私钥或其他后端私有配置。
- 将 Cloudflare Workers / Pages 误导为当前已支持的应用主体部署路径。

### 4.3 原型阶段的可写范围判断

本轮评估结论为：默认不开放 AI 侧写配置。

如果后续一定要评估“助手代填”能力，必须同时满足以下条件：

1. 只允许写入已有安装向导或后台设置页已经允许用户手工保存的字段。
2. 只允许写入当前未锁定、非敏感、非内部独占的字段。
3. 必须通过显式确认链路，让用户在最终提交前看到逐字段 diff。
4. 实际写入仍必须复用现有安装 API 或后台设置 API，不允许增加新的秘密写入路径。

在满足这些条件之前，产品口径统一为“只读解释，不代操作”。

## 5. 安全边界

### 5.1 配置与秘密边界

- `INTERNAL_ONLY_ENV_KEYS` 和 `INTERNAL_ONLY_SETTING_KEYS` 对助手保持不可见，不回答具体值，不暴露存在性细节以外的信息。
- `FORCED_ENV_LOCKED_KEYS` 只允许解释“为什么不能在这里改”，不允许提供伪造的可写建议。
- 涉及密钥、密码、Token、Secret 的问题，只能回答字段用途、配置入口和脱敏后的状态，不能回显真实内容。

### 5.2 权限与越权边界

- 助手必须绑定当前页面和当前角色语境，安装页只回答安装页上下文，后台设置页只回答设置页上下文。
- 未登录或未到后台权限的用户，不能通过问答层获得后台私有配置、审计信息或管理入口。
- 任何需要“保存、覆盖、删除、重置”的动作都必须退回现有表单与 API，不允许由问答层自行触发。

### 5.3 错误回答与失败回退

- 当答案无法从现有设置元信息、安装向导状态或文档知识源可靠推出时，必须显式降级为“无法确认，请查看对应文档或手工检查当前字段”。
- 当用户问题超出初始化 / 配置范围时，必须回退到现有文档链接或明确提示不在当前助手职责内。
- 当平台路径与现有研究结论冲突时，以当前文档事实源为准，禁止助手做主观猜测。

## 6. 最小交互原型

### 6.1 建议入口

建议只保留两个入口：

1. 安装向导预览 / 完成页的“还有配置疑问？”入口。
2. 后台系统设置页的“解释当前配置”入口。

不建议在首页、编辑器或全站 Header 提前铺开统一问答入口。

### 6.2 建议交互结构

单轮回答统一收敛为以下结构：

1. 当前问题的直接结论。
2. 该问题关联的字段或步骤。
3. 若字段受锁定，解释锁定原因和下一步。
4. 若当前可以先跳过，明确说明“建议安装后再补”。
5. 给出唯一推荐的下一步动作和对应文档入口。

### 6.3 推荐的首批问题模版

首批只覆盖高频问题：

- 我现在最少还需要配置哪些项才能启动。
- 为什么这个字段是只读的。
- 这个字段应该在安装向导配置还是去后台配置。
- Docker / Vercel / 本地开发分别优先看哪一组配置。
- 当前报错更像是缺变量、格式错误，还是平台不支持。

## 7. 实现复用建议

若进入下一轮最小原型，优先复用以下资产：

- 安装阶段分层与字段模型：`utils/shared/installation-settings.ts`、`utils/shared/installation-checklist.ts`
- 安装状态与回填：`composables/use-installation-wizard.ts`、`server/services/installation.ts`
- 设置解释层与锁定规则：`server/services/setting.constants.ts`、`server/services/setting.ts`
- 锁定文案与错误反馈：`utils/shared/installation-env-setting.ts`、`utils/shared/request-feedback.ts`
- 只读知识检索约束：`.github/skills/qa-assistant/SKILL.md`
- 文档知识源：`docs/guide/quick-start.md`、`docs/guide/deploy.md`、`docs/guide/variables.md`、`docs/design/modules/system.md`、`docs/design/modules/system-config-unification.md`

这些资产已经覆盖字段定义、锁定规则、用户文案、错误归因和只读问答边界。下一轮实现应只增加“编排与展示层”，不新增第二套配置知识。

## 8. 风险分级

| 风险项 | 等级 | 说明 | 当前处置 |
| :--- | :--- | :--- | :--- |
| 助手回答过时或与当前事实源漂移 | 中 | 文档、设置元信息和安装模型若不同步，容易出现错误指路 | 只允许复用现有事实源，不单独维护助手知识副本 |
| 将只读解释误扩成可写代理 | 高 | 一旦允许隐式保存，风险立即升级到权限与配置破坏 | 原型阶段禁止 AI 侧写配置 |
| 泄露敏感或内部独占配置 | 高 | 安装与设置链路包含密钥、Secret、数据库连接等敏感项 | 继续遵守 `INTERNAL_ONLY` 与脱敏边界 |
| 把平台边界解释错误 | 中 | 尤其是 Cloudflare 路径和启动期锁定项 | 冲突时统一回到现有部署文档与研究文档 |
| 入口过早扩张导致用户误以为是全站 Copilot | 中 | 会稀释初始化助手的职责，提升维护成本 | 限制在安装页和设置页两个入口 |

## 9. 下一阶段准入条件

仅当满足以下条件时，才建议把本评估推进为正式开发任务：

1. 明确坚持“初始化 / 配置解释”这一单一职责，不扩成全站问答。
2. 所有回答都能从现有安装向导、设置元信息或文档事实源生成，不新增独立知识库存档。
3. 原型阶段不触碰敏感写入，不新增新的权限模型。
4. 至少能覆盖本地开发、Vercel、Docker、自托管 Node 四类路径，并继续把 Cloudflare 作为边界提示处理。
5. 在 UI 文案与交互上明确区分“建议下一步”与“系统已自动完成”，避免误导用户。

如果未来产品目标升级为“AI 直接代填配置、跨页排障、管理后台统一 Copilot”，则必须重新立项，并先补一份独立的安全与权限设计文档，不能沿用本评估直接扩写。

## 10. 结论摘要

本轮评估建议保留并推进一个只读、启用式、解释型的 AI 初始化 / 配置助手最小原型。它应被视为安装向导、系统设置解释层和现有文档知识源之间的编排器，而不是新的配置中心或新的知识事实源。

当前最稳妥的产品路径是：

- 先做“解释和指路”。
- 暂不做“代填和代改”。
- 把安全边界锁死在现有权限和配置模型之内。

在这个范围内推进，ROI 与可控性都成立；超出这个范围，则应重新评估而不是在当前结论上继续叠加能力。
