# 分类与标签聚合页设计 (Taxonomy Aggregation)

## 1. 概述

为了增强文章的可发现性和相关性，本模块旨在实现分类 (Category) 和标签 (Tag) 的详情展示页。用户点击文章详情或卡片中的分类/标签时，将跳转到对应的聚合路径，展示所有关联的文章。

## 2. 路由结构

采用基于 `slug` 的动态路由，确保 URL 具有良好的 SEO。

-   **分类页**: `/categories/[slug]`
    -   示例: `/categories/javascript`
-   **标签页**: `/tags/[slug]`
    -   示例: `/tags/nuxt`

## 3. 页面逻辑

### 3.1 数据获取

-   **页面初始化**:
    -   基于 `useRoute().params.slug` 识别目标分类或标签。
    -   调用后端 API 获取该分类/标签的元数据（名称、描述等）。
    -   调用文章列表 API 进行过滤查询。
-   **文章筛选 API**:
    -   请求路径: `GET /api/posts`
    -   查询参数:
        -   `category`: 分类 slug
        -   `tag`: 标签 slug (单数，用于聚合页)
        -   `language`: 当前语言 (i18n 隔离)

### 3.2 UI 展示

-   **头部 (Header)**:
    -   显示当前分类/标签的名称。
    -   显示该分类下的文章总数。
    -   (可选) 显示分类的描述信息。
-   **列表内容**:
    -   复用 `article-card` 组件。
    -   支持分页加载。
-   **空状态**:
    -   若无相关文章，展示友好提示。

## 4. 交互增强

-   **详情页**: 在 `pages/posts/[id].vue` 中，将面包屑中的分类名称、底部的标签 Chip 转换为 `nuxt-link`。
-   **列表页**: 在 `components/article-card.vue` 中，将分类和标签设为可点击状态。

## 5. SEO 优化

-   动态设置页面 Title: `[分类/标签名] - [站点名称]`。
-   Meta Description 使用分类描述或占位符。
-   遵循多语言前缀路由 (`/zh/categories/...`)。

## 6. 文章翻译标签绑定策略

### 6.1 背景与问题

-   文章编辑器在翻译场景下不能只保留 `string[]` 形式的标签名；一旦丢失源标签的 `translationId`、`slug` 或 `id`，保存时就只能退化为“按名称创建标签”。
-   这种退化会造成两个直接后果：
    -   同一篇文章的不同语言版本可能生成不同的标签集合，无法稳定回到同一翻译簇。
    -   目标语言缺少现成标签时，后端会把当前名称当作全新标签创建，最终形成孤立标签。

### 6.2 目标策略

-   文章翻译工作流必须基于源文章标签生成目标语言标签，而不是在目标文章保存时重新猜测。
-   编辑器保存请求需要同时提交“展示用标签名”和“标签翻译绑定信息”，后者至少应携带以下候选信息之一：
    -   源标签或目标标签的 `translationId`
    -   源标签的 `slug`
    -   源标签的 `id`
-   服务层在处理标签时应遵循以下优先级：
    1.  按翻译簇 (`translationId -> slug -> id`) 查找目标语言标签。
    2.  命中则直接复用该标签。
    3.  未命中则使用目标语言标签名创建新标签，并写入相同的翻译簇标识。
    4.  用户手动新增且没有翻译绑定的标签，继续按普通标签处理，不强制加入现有翻译簇。

### 6.3 保存契约

-   `POST /api/posts` 与 `PUT /api/posts/:id` 在保留原有 `tags: string[]` 的同时，新增 `tagBindings` 字段用于传输标签翻译绑定信息。
-   `tags` 负责维持当前编辑器 UI 与现有自动完成组件的兼容性。
-   `tagBindings` 作为精确关联的事实源，仅在服务层解析，不直接持久化到文章实体。

```typescript
interface PostTagBindingInput {
    name: string
    translationId?: string | null
    sourceTagSlug?: string | null
    sourceTagId?: string | null
}
```

### 6.4 编辑器行为约束

-   当翻译工作流从源文章导入标签时，应为每个标签生成对应的 `tagBindings` 项。
-   若目标语言已经存在同簇标签，编辑器展示该目标语言标签名，并保留对应翻译簇。
-   若目标语言尚无同簇标签，编辑器可先生成目标语言标签名，但在真正保存前不得丢失其源标签翻译簇信息。
-   用户手动删除标签时，关联的 `tagBindings` 也必须同步删除；用户手动输入一个全新标签时，只创建没有翻译簇的普通绑定项。

### 6.5 风险边界

-   目标语言若已存在“同名但不同翻译簇”的标签，服务层不能仅凭名称静默重绑到错误翻译簇。
-   本轮优化的最低保证是“翻译场景优先按翻译簇复用或创建标签”；对于历史上已经形成的同名冲突标签，后续仍需通过后台治理工具逐步修复。
