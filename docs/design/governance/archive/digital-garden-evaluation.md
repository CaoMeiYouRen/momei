# Digital Garden / 知识花园模式探索评估

> 评估日期: 2026-06-08  
> 评估对象: backlog #10「Digital Garden / 知识花园模式」  
> 评估范围: go/no-go（仅评估，不进入本阶段实现）

## 1. 评估结论

**结论：No-Go（当前阶段不进入实现，保留为后续 P2 候选）**

当前代码基线下，Digital Garden 的最小可用能力（双向链接 + 非时序发现）可以设计，但若在本阶段直接落地，会同时触发数据模型、路由信息架构与前端依赖三条改造面，侵入度明显高于本阶段「评估态」预期。建议先完成轻量设计冻结与低成本 PoC，再决定是否上收到后续阶段实现。

---

## 2. 现状基线（代码事实）

1. **数据模型现状**
   - `Post` 已有统一 `metadata: json` 容器，可承载扩展字段。  
   - 标签关系已通过 `post_tags_tag_posts` 关联表实现，项目已有关系型建模与查询基座。

2. **内容发现现状**
   - 公开导航与页面结构以 `posts / categories / tags / archives` 为主，核心发现链路仍是时序 + taxonomy。  
   - 文章详情页已有“相关推荐”，但来源是分类/标签相似度，不是引用图谱或双向链接关系。

3. **依赖现状**
   - 当前依赖中未引入 Cytoscape、Sigma、vis-network、graphology 等常见知识图谱库。

---

## 3. 核心评估维度

### 3.1 双向链接存储模型（JSON 字段 vs 关联表）

| 方案 | 落地方式 | 查询成本（正向/反向） | 性能与维护预估 | 结论 |
| :-- | :-- | :-- | :-- | :-- |
| JSON 字段 | 在 `post.metadata` 写入 `outboundLinks` / `backlinks` | 正向低，反向高（通常需扫描或额外同步） | 初期改动小，但反向查询与一致性治理成本高；跨数据库方言下索引策略不稳定 | 仅适合 PoC |
| 关联表 | 新增 `post_links`（`source_post_id`, `target_post_id`, `kind`）+ 索引 | 正向/反向都可索引查询 | 建模一次后可持续扩展（反向链接、图谱、统计）；与现有 TypeORM 关系模式一致 | **推荐长期方案** |

**判断**:
- 若目标只做“单篇文内引用提示”的短期试验，JSON 可快速验证。
- 若目标包含“被引用列表 + 图谱 + 后续排序策略”，应直接使用关联表，避免二次迁移。

### 3.2 非时序导航对路由 / IA 的侵入度

当前主导航和公开入口围绕时序与分类组织。引入 Digital Garden 需要至少新增：

1. 独立入口（如 `/garden` 或等价页面）与导航位调整。  
2. 文章详情的双向链接区块（与现有“相关推荐”并存或替换规则）。  
3. i18n、SEO（站点地图/内部链接权重）与发现策略联动。

**侵入度评估：中高**。  
这不是单点 UI 增强，而是内容发现模型改造，会影响导航信息密度与首屏入口策略。

### 3.3 知识图谱可视化依赖与 bundle 增量

当前仓库未内建图谱库。若直接上图谱能力，通常需要新增图谱渲染与布局相关依赖，并处理：

1. 客户端仅加载（SSR/水合边界）。  
2. 大图数据分片与渐进渲染。  
3. 移动端交互与性能兜底。

**风险评估：中高**。  
在未先验证“文本级双向链接是否有真实使用价值”前，直接引入图谱渲染依赖，ROI 偏低且会抬升前端维护与性能负担。

## 4. 决策与建议路径

### 4.1 当前阶段决策

- **No-Go（实现层）**：本阶段不做双向链接、成熟度标记和知识图谱落地。  
- **Go（设计层）**：允许进入下一轮“轻量设计冻结 + PoC”。

### 4.2 建议的后续最小切片（若后续上收）

1. **Phase A（低风险 PoC）**  
   - 仅做文内链接解析 + 单篇“被引用”文本列表（无图谱）。  
   - 数据可临时放在 JSON 字段，验证真实使用频率与编辑成本。

2. **Phase B（可持续建模）**  
   - 若 PoC 数据成立，迁移到 `post_links` 关联表并补索引。  
   - 统一正向/反向查询 API，与现有详情页读模型整合。

3. **Phase C（可视化增强，可选）**  
   - 在完成 A/B 且有明确收益后，再评估图谱可视化依赖接入。  
   - 要求先完成 bundle 预算与交互降级方案再进入实现。
