# Phase 38 执行计划

**时间表**: 2026-05-18 ~ 约 1 - 2 周
**目标**: 在第三十七阶段完成 Windows 本地性能、测试有效性、结构复用、ESLint 窄切片与 Postgres 长窗口复核后，继续以“0 个新功能 + 5 个优化”的受控组合推进下一阶段，优先修补第三方分发标签尾注与预览一致性，并围绕测试有效性第二轮、Postgres 单路径收敛、结构复用第二轮与 AI provider / 测试桩类型债继续做窄切片治理。

## 准入结论

五条主线均来自 [backlog.md](../../plan/backlog.md) 的长期主线或短期候选，总数控制在 `5` 项内，符合当前阶段容量约束。第三十七阶段已完成归档，因此本文件现作为第三十八阶段的正式执行计划，供 [roadmap.md](../../plan/roadmap.md) / [todo.md](../../plan/todo.md) 与实现阶段对照使用。

## 执行约束

- 第三方分发修补只冻结 `B 站 / Memos` 两个渠道，不顺手扩写到 WechatSync、邮件或其他导出器。
- Postgres 下一轮只允许在“剩余显式 `initializeDB()` 调用点审计”和“公开热点读链路继续瘦身”中二选一；基于当前预算压力，本轮默认优先后者。
- 测试有效性、结构复用与 ESLint / 类型债都必须继续维持“窄范围 + 可验证 + 可回退”的节奏，不得借机膨胀为全仓治理。
- 若任一主线需要改动跨模块契约或新增治理专题边界，必须先补对应设计文档再进入实现。

## 主线 1：第三方分发标签尾注与预览一致性修补（P1）

**执行范围**:

1. 仅覆盖 `B 站` 与 `Memos` 两个渠道的标签尾注与预览一致性问题。
2. 优先复用 [post-distribution-template.ts](../../../../server/services/post-distribution-template.ts) 现有物料包入口，收敛“标签标准化 + 尾注拼装 + 预览展示”到同一 helper，而不是分别热修预览层和投递层。
3. 校准“标签去空格再输出”的唯一口径，避免预览与实际投递对同一标签集合生成不同结果。

**非目标**:

- 不扩写到 WechatSync、邮件、Hexo 仓库同步或其他分发渠道。
- 不重做整套分发 UI，只修补物料构造与预览 / 投递一致性。

**验收标准**:

- `B 站` 预览、`B 站` 实际同步 payload 与 `Memos` 预览三处输出在标签尾注上保持一致。
- 预览构造与实际分发复用同一条标签标准化 / 尾注拼装入口。
- 至少补齐一组预览层断言和一组分发 / 导出层断言，证明修补不是 UI 层假转绿。

**最小验证矩阵**:

- 定向 Vitest：分发物料 helper / template 测试。
- 定向 Vitest：`post-distribution` 服务层测试或等价 payload 测试。
- 必要时补一组后台分发预览组件测试。

## 主线 2：测试有效性第二轮切片（P0）

**执行范围**:

1. 优先补组件层 direct TTS 失败映射，直接沿 [use-tts-volcengine-direct.test.ts](../../../../composables/use-tts-volcengine-direct.test.ts) 与相关对话框测试继续扩断言，而不是新开一套测试面。
2. 补页面级 auth degradation 断言，优先选择已有页面测试基座的认证页或公开页装配链路。
3. 从 [public.get.test.ts](../../../../tests/server/api/settings/public.get.test.ts) 与 friend-links 现有测试基座中二选一，补一组 `503` 或等价失败口径断言，避免高风险链路只剩服务端成功路径。

**非目标**:

- 不回到低价值 coverage 铺量。
- 不把 snapshot 或纯存在性断言当作主要成果。

**验收标准**:

- 至少完成 `3` 组高风险失败 / 边界断言：组件层 direct TTS、页面级 auth degradation、`settings public` 或 `friend-links` 失败口径。
- 其中至少 `1` 组明确覆盖用户可见错误映射，而不是只断言内部异常被抛出。
- 本轮新增入口被记录为可复用的定向回归矩阵，并注明未覆盖边界。

**最小验证矩阵**:

- 定向 Vitest：受影响 composable / page / API 测试文件。
- 受影响文件类型检查。

## 主线 3：Postgres 公开热点读链路继续瘦身（P0）

**执行范围**:

1. 下一轮只选择“公开热点读链路继续瘦身”，不并行开启剩余显式 `initializeDB()` 调用点审计。
2. 优先围绕 `settings public`、`friend-links`、`posts home`、`search` 等已知公共热读路径，继续做最小字段集、短 TTL、请求去重或装配裁剪中的一项或两项最小收敛。
3. 结合预算事实复核：免费额度 `5GB / 17` 天已消耗约 `301MB/天`，高于约 `170MB/天` 的安全预算；`90 CU-hrs / 17` 天约为 `5.29 CU-hrs/天`，对应约 `21` 小时 `0.25 CU` 活跃窗口。

**非目标**:

- 不并行推进新的请求入口初始化治理。
- 不扩写为全站数据库重构或全量 SQL 清洗工程。

**验收标准**:

- 给出一组新的 `pg_stat_statements` 或等价 live sample，对照说明目标公开读路径的 calls、rows、mean time 或网络体量存在下降趋势。
- 新切片至少回答“为什么当前超预算更像公开热读问题，而不是初始化误触问题”。
- 若本轮无法证明收益，应停止扩写并把剩余 `initializeDB()` 审计保留为下一候选，而不是双线并行。

**最小验证矩阵**:

- 定向 API 测试。
- 受影响文件类型检查。
- 一组 live sample 或本地等价观测记录。

## 主线 5：ESLint / 类型债下一轮窄切片（P1）

**执行范围**:

1. 继续坚持“单规则、单文件或双文件”的窄切片。
2. 优先在 AI provider 聚合层（如 [server/utils/ai/index.ts](../../../../server/utils/ai/index.ts) 及近邻 provider 实现）与高 ROI 测试桩历史断言之间二选一推进。
3. 进入实现前先冻结命中清单、替代写法、回滚边界与最小验证矩阵，不回到全仓 `any` 清零。

**非目标**:

- 不并行开启 `no-unsafe-*` 或全目录规则收紧。
- 不把测试桩清理扩大成整仓 mock 体系重写。

**验收标准**:

- 定向 ESLint、定向测试与类型检查通过。
- 残余债务与下一轮候选有明确记录。
- 若选择 AI provider 聚合层，需优先收敛 provider 名称与配置解析处的 `as any` / `catch any`；若选择测试桩，则需优先清理高频 `event as any` / `repo as any` 历史断言。

**最小验证矩阵**:

- 定向 ESLint。
- 定向 Vitest。
- 受影响文件类型检查。
