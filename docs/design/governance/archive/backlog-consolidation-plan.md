# Backlog 长期主线瘦身与层级重构方案

**时间表**: 2026-05-17 ~ 约 1 天（纯规划文档变更，不涉及代码实现）
**目标**: 对 `docs/plan/backlog.md` 中当前 12 条长期主线做两件事——①将 4 条文档类任务合并为 1 条，消除意图重复；②将"周期性回归"从普通长期主线提级为独立的周期性验证层，修正其层级错位。

## 背景与问题诊断

### 当前状态

`docs/plan/backlog.md` 中长期主线共 **12 条**，按性质可分为以下五类：

| 类别 | 序号 | 任务 |
|:---|:---|:---|
| 代码质量 | #1 | 测试覆盖率与有效性治理 |
| 代码质量 | #2 | ESLint / 类型债与规则收紧治理 |
| 代码质量 | #3 | 结构复用治理 |
| 代码质量 | #5 | 存量代码注释治理与注释漂移收敛 |
| 运行时/性能 | #6 | Postgres 查询与数据库治理 |
| 运行时/性能 | #12 | Windows 本地 Dev / Build 性能治理 |
| i18n | #10 | 国际化运行时加载与文案复用治理 |
| **文档** | **#7** | **模块设计与专项治理文档收敛** |
| **文档** | **#8** | **回归记录独立归档与深度分层** |
| **文档** | **#9** | **路线图 / Todo 深度归档治理** |
| **文档** | **#11** | **文档翻译 freshness 清偿与翻译治理** |
| **元任务** | **#4** | **周期性回归与阶段收口执行治理** |

### 问题一：四条文档类任务存在严重意图重叠

| 任务 | 治理对象 | 治理动作 |
|:---|:---|:---|
| #7 模块设计与专项治理文档收敛 | `docs/design/` 目录 | 分层边界维护 + 漂移清理 |
| #8 回归记录独立归档与深度分层 | `docs/reports/regression/` | 分文件归档 + 分层入口 |
| #9 路线图 / Todo 深度归档治理 | `docs/plan/` | 行数阈值触发 + 区间分片 |
| #11 文档翻译 freshness 清偿 | `docs/i18n/*/` | freshness 同步 + tier 分层 |

四条任务治理的对象不同，但**治理动作的本质完全相同**：都是"维护文档分层结构 + 设置膨胀阈值 + 定期清理漂移 + 保持 freshness"。它们在当前 backlog 中各自独立维护一张卡片，导致：

- 每条约 30-50 行的卡片正文存在结构性重复（都有"目标/状态/当前状态/最近上收阶段/下一次切片方向"五字段，且当前状态中有大量交叉引用的阶段结论）。
- 四条任务经常被同一个阶段同时上收为不同主线（例如第三十阶段同时上收了 #3、#5、#7、#8、#9、#11 中的多条），但它们作为单独条目的语义区别在执行侧并不显著——执行者面对的是"改文档目录/改文档内容/改翻译"，而不会感知"这是文档治理主线 A 还是文档治理主线 B"。
- 新阶段做候选上收评估时，需要逐一核对四条任务的状态和方向，评估成本与任务数量成正比。

### 问题二：回归任务层级错位

#4 "周期性回归与阶段收口执行治理" 与其他 11 条长期主线存在**根本性的性质差异**：

| 其他长期主线 | 回归任务 (#4) |
|:---|:---|
| 有明确的"修复目标"（提 coverage、收紧 ESLint、压重复率） | 没有"修复目标"——它**验证**其他主线是否回退 |
| 通过代码/文档改动推进 | 通过执行固定命令组合（`regression:weekly` 等）推进 |
| 按阶段切片上收 | 按固定节奏执行（周级/发版前/阶段收口） |
| 可以暂停/观察/关闭 | 不能暂停——它是整个治理体系的"健康检查" |

回归任务的本质是一个**周期性验证层**，而不是一个应该与其他治理任务并列排队的普通任务。把它放在长期主线列表中，制造了三个实际坏处：

1. **模糊了它的特殊地位**：执行者在候选上收评估时，会把"要不要收回归切片"和"要不要收 ESLint 切片"放在同一个维度做取舍。
2. **隐藏了它对其他任务的依赖**：回归任务发现的问题（例如 coverage 下降、ESLint warning 反弹）本质上是其他长期主线的漂移信号，应该回灌到对应主线，而不是由回归任务自己去修。
3. **阻碍了回归记录管理的归位**：#8（回归记录归档）本质上是回归验证层的配套工件管理——回归跑了 → 结果写到 `current.md` → 膨胀了 → 滚动归档。这条任务的自然主人是回归层，不是文档治理层。当前因为回归层被压在普通任务列表中，所以它连自己的配套管理都带不起来。

## 方案设计

### 一、文档类任务合并

将 #7、#9、#11 **三条**合并为一条新的长期主线：

> **文档事实源、翻译与分层归档治理**

合并后的卡片覆盖原三条的全部治理面：

- **分层边界维护**（原 #7）：`docs/design/modules/` vs `docs/design/governance/` 的分层边界；模块总设计只保留稳定入口，专项治理只保留增量结论。
- **膨胀阈值触发**（原 #9）：`roadmap.md`、`todo-archive.md`、`backlog.md` 等规划文档的行数阈值监控与区间分片归档。
- **翻译 freshness**（原 #11）：`docs:check:source-of-truth` 的通过状态维护；翻译 tier 分层（必须同步 / 允许摘要 / source-only）的持续执行。
- **漂移清理**（原三条共有）：过时设计文档的归档或删除；残留任务口吻的清理；旧引用路径的修复。

**合并前后对比**：

| 合并前 | 合并后 | 变化 |
|:---|:---|:---|
| #7：模块设计与专项治理文档收敛（30 行） | 合并为一条"文档事实源、翻译与分层归档治理"（约 60 行） | -2 条独立卡片 |
| #9：路线图 / Todo 深度归档治理（30 行） | | |
| #11：文档翻译 freshness 清偿（30 行） | | |
| 合计 ~90 行，3 条卡片 | 合计 ~60 行，1 条卡片 | 维护成本降低，语义更集中 |

**#8（回归记录独立归档）的去向**：#8 不合并到文档治理线，而是随 #4 一起归入回归验证层（见下一节）。理由：回归记录的归档是回归执行的直接产物管理，不是独立的文档治理事项。

### 二、回归任务提级

将 #4 从长期主线列表中移出，在 `backlog.md` 中新增一个与"长期主线任务"同级的独立区块：

> **周期性回归验证层**

#### 定位

回归验证层不是"一个任务"，而是长期主线任务的**健康检查层**。它不产生直接的"改进"，只验证"没有回退"。它的存在理由与其他长期主线不同——不是因为"这里有债要修"，而是因为"这里的债修完之后，必须有人持续检查有没有新的债"。

#### 层级关系

```
┌─────────────────────────────────────────┐
│         周期性回归验证层 (regression)      │
│  周级 / 发版前 / 阶段收口  三条固定入口     │
│  验证 → 发现漂移 → 路由到对应主线          │
└──────────────┬──────────────────────────┘
               │ 验证覆盖
    ┌──────────┼──────────┬──────────┐
    ▼          ▼          ▼          ▼
 测试治理   ESLint治理  结构复用    i18n治理  ...
 (长期主线1) (长期主线2) (长期主线3) (长期主线N)
```

#### 与长期主线的交互协议

1. **回归层只检查，不修复**。当 `regression:weekly` 发现 coverage 下降时，不自己补测试——而是标记到"测试覆盖率治理"长期主线的下一次切片候选。
2. **回归层有固定节奏，不参与阶段切片排队**。其他长期主线按"评估 → 上收 → 执行 → 归档"的阶段节奏推进；回归层按"周级 / 发版前 / 阶段收口"的日历节奏执行。两者互不抢占阶段容量。
3. **回归层是 blocker 唯一来源**。当回归验证发现 blocker 级问题（例如 `phase-close` 回归中 strict 检查失败），它有权阻断当前阶段的归档，而不是"作为建议记录然后继续"。

#### 在原 #4 基础上增强的内容

| 原 #4 内容 | 新回归层 |
|:---|:---|
| 三条固定入口（weekly/pre-release/phase-close） | 保留，并明确标注为"不可跳过"的硬入口 |
| 回归执行后的证据留痕 | 保留，并归入"回归记录管理"（原 #8 内容整体移入） |
| 下一轮优化方向 | 改为"当前回归覆盖矩阵"——展示每条长期主线的当前回归覆盖率 |
| 无 | 新增：漂移路由规则——回归发现的具体问题应路由到哪条长期主线 |

#### 纳入 #8（回归记录归档管理）

回归记录归档不再作为独立的长期任务，而是回归验证层的配套工件管理：

- 每次回归执行后，结果写入 `docs/reports/regression/current.md`。
- 当 `current.md` 触发膨胀阈值（500-700 行 warning，700+ 行 blocker），由回归层负责触发滚动归档。
- 滚动归档的执行方式：将旧记录整体迁移到 `docs/reports/regression/archive/`，主窗口仅保留近线记录。

```

│   ├── #3 结构复用治理
│   ├── #4 存量代码注释治理与注释漂移收敛
│   ├── #5 Postgres 查询与数据库治理
│   ├── #6 国际化运行时加载与文案复用治理
│   ├── #7 文档事实源、翻译与分层归档治理  ← 合并自原 #7+#9+#11
│   └── #8 Windows 本地 Dev / Build 性能治理
│

│
```

### 四、执行约束

- 本次改动**仅涉及规划文档**（`backlog.md` + 新增设计文档），不修改任何代码或脚本。
- 合并后的文档治理线卡片必须保留原三条被合并任务的关键里程碑（最近上收阶段、当前状态），确保历史可追溯。
- 回归层提级后，原 #4 卡片从长期主线区块移除，新回归层区块以 "当前正在执行什么检查、覆盖了哪些主线" 为核心信息，不再维护"下一个切片方向"。
- 本次不改动 `planning.md` 已有的回归调度规则（三条固定入口及其命令组合已在该文件中完整定义），回归层区块只做引用和覆盖率说明。

## 范围与非目标

### 执行范围

1.  将 `backlog.md` 中 4 条文档类长期主线合并为 1 条。
2.  将回归任务从长期主线列表移出，提升为独立的"周期性回归验证层"区块。
3.  同步更新 `backlog.md` 的维护规则说明，反映新的层级结构。
4.  可选：同步更新 `docs/standards/planning.md` 的编排建议（如果其内容需要对准新的层级结构）。

### 非目标

-   不修改回归调度的三条固定入口及其命令组合（仍在 `planning.md` 中维护）。
-   不新增或删除任何回归检查项。
-   不修改 `roadmap.md`、`todo.md` 或 `todo-archive.md`。
-   不修改 `docs/design/governance/` 中已有的文档治理专项（本方案只改 backlog 层的归类，不动具体治理策略的正文）。
