# 文章分享系统设计文档 (Post Sharing System)

## 1. 概述

本文档用于收敛“文章页一键分享与图标系统修复”中与分享系统相关的设计结论，聚焦以下三个问题：

1. 哪些公开页面应该提供分享入口。
2. 各平台应采用“直接拼接分享链接”还是“复制链接 / 二维码 / 手动发布”的形式。
3. 分享时的链接口径应该如何与当前 locale、SEO canonical 与公开访问边界保持一致。

为避免职责重叠，作者社交链接、打赏配置与文章页赞助展示仍以 [商业化与社交集成设计文档](../modules/commercial.md) 为主；本文只描述“页面分享”能力，不重复作者社交资料展示逻辑。

## 2. 设计目标

本阶段的分享系统应满足以下目标：

- 在文章详情页提供统一、可发现的一键分享入口。
- 在移动端优先复用浏览器原生分享能力，减少复杂交互成本。
- 对支持 Web Share 或稳定分享 URL 的平台提供直接分享。
- 对缺少稳定网页分享接口的平台，统一降级到“复制链接 / 复制文案 / 展示二维码 / 打开站内文章链接”的可用方案。
- 分享链接与 SEO canonical 保持一致，避免把草稿参数、后台参数或无意义 hash 片段带入分享结果。
- 保持与现有图标系统、i18n、SEO 和文章页结构兼容，尽量减少额外维护面。

## 3. 范围建议

### 3.1 当前阶段建议纳入的分享页面

| 优先级 | 页面 | 路径口径 | 是否推荐在本阶段支持 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| P0 | 文章详情页 | `/posts/:slug-or-id` 的 locale 版本 | 是 | ROI 最高，天然具备标题、摘要、封面和 canonical 链接 |
| P1 | 分类页 | `/categories/:slug` | 可后续补充 | 适合分享专题聚合入口，但转化价值低于文章详情 |
| P1 | 标签页 | `/tags/:slug` | 可后续补充 | 与分类页类似，建议在组件稳定后复用 |
| P1 | 归档页 / 文章列表页 | `/archives`、`/posts` | 可后续补充 | 更适合“复制链接”而不是平台定向分享 |
| P2 | About / 友链页 | `/about`、`/friend-links` | 低优先级 | 有价值，但不应阻塞文章页主链路 |

### 3.2 当前阶段不建议提供分享入口的页面

以下页面默认不进入分享系统的公开能力范围：

- 后台页、登录页、注册页、安装页。
- 草稿页、受密码保护页、仅登录可见页、订阅可见页。
- 含一次性查询参数、调试参数或预览态参数的链接。

结论：**本阶段先把文章详情页做好，再把分享能力抽象为可复用组件，后续再扩面到 taxonomy / list / about 等公开页面。**

## 4. 分享载荷设计

统一的分享载荷建议收敛为：

```ts
interface SharePayload {
  pageKind: 'post' | 'category' | 'tag' | 'list' | 'page'
  title: string
  text: string
  url: string
  image?: string | null
  locale: string
}
```

默认取值建议：

- `title`：文章标题或页面 SEO 标题。
- `text`：优先用文章摘要，其次回退到站点描述。
- `url`：当前语言下的公开 canonical URL。
- `image`：文章封面或站点 Logo，仅用于原生分享或预览增强，不作为分享 URL 的必需参数。

## 5. 链接口径设计

### 5.1 Canonical 优先原则

分享链接不应直接拼接 `window.location.href` 的完整原值，而应优先复用项目已经存在的 canonical 逻辑：

- 页面级 SEO 已在 [composables/use-page-seo.ts](../../composables/use-page-seo.ts) 中维护 `canonicalUrl`。
- 文章详情页已有 locale 感知的路径与公开 URL 信息，可作为分享入口的事实来源。

建议新增一个共享 helper，例如：

```ts
function buildShareCanonicalUrl(options: {
  siteUrl: string
  localePath: (path: string) => string
  pageKind: 'post' | 'category' | 'tag' | 'list' | 'page'
  slug?: string | null
  id?: string | null
  path?: string | null
})
```

其规则如下：

1. 对文章详情页，优先使用 slug，缺失时回退 id。
2. 通过 `localePath(...)` 生成当前语言版本路径，而不是手写语言前缀。
3. 只保留公开可访问路径，不带 hash。
4. 对无意义的 query（如调试参数、预览参数）进行剥离。

### 5.2 参数策略

第一阶段建议默认分享**干净 canonical URL**，暂不强制追加 UTM，原因如下：

- 复制出来的链接更短、更可信。
- 不会污染用户二次转发的链接。
- 与当前 SEO canonical 保持完全一致，降低分析复杂度。

如果后续确实需要分享归因，建议使用两种更稳妥的方式之一：

1. 可选开关式 UTM：`utm_source=share&utm_medium=<channel>&utm_campaign=post-share`
2. 独立的短链 / 跳转归因层，而不是直接把 tracking 参数暴露给最终用户

### 5.3 locale 规则

分享系统必须显式遵守当前项目的语言路由策略：

- 分享当前正在阅读的语言版本，而不是强制回退到默认语言。
- 生成 URL 时始终走 `localePath`，避免语言前缀口径漂移。
- 若当前页面是翻译聚合中的某个 locale 版本，则分享该 locale 的文章路径。

## 6. 平台分层策略

### 6.1 第一类：可直接拼接分享链接的平台

这些平台可以通过稳定 URL 模板直接打开分享页，适合作为“真正的一键分享”：

| 平台 | 建议形式 | 备注 |
| :--- | :--- | :--- |
| X / Twitter | 直接拼接分享 URL | 适合标题 + URL 组合 |
| Facebook | 直接拼接分享 URL | 以 URL 为主 |
| LinkedIn | 直接拼接分享 URL | 更适合文章链接传播 |
| Telegram | 直接拼接分享 URL | 支持文本 + URL |
| WhatsApp | 直接拼接分享 URL | 移动端体验较好 |
| Email | `mailto:` | 适合作为通用兜底 |

建议 builder 层统一做 `encodeURIComponent`，避免标题和摘要中的多语言字符破坏分享 URL。

### 6.2 第二类：应优先降级为复制分享的平台

以下平台更适合提供图标入口，但默认采取“复制链接 / 复制标题摘要 / 客户端手动粘贴”的方式：

| 平台 | 推荐策略 | 原因 |
| :--- | :--- | :--- |
| 微信 | 复制链接 + 二维码 | 网页端直达分享能力受限 |
| 微博 | 复制链接 + 复制文案 | 已确认开放平台的第三方分享接口依赖 OAuth 登录、access_token、真实 IP 与安全域名绑定，不适合当前公开文章页的一键匿名分享链路 |
| 小红书 | 复制标题摘要 + 链接 | 官方网页拼链不稳定，客户端导向更常见 |
| 掘金 | 复制链接 | 更像内容平台投稿 / 转发场景，不适合依赖网页分享模板 |
| bilibili | 复制链接 | 站外网页分享入口稳定性弱 |
| 知乎 | 复制链接 | 平台更偏回答 / 专栏创作，网页直分享并不稳定 |

这类平台仍然需要保留图标，因为用户在认知上会把它们视为“目标发布渠道”；但交互上不应伪装成一定能自动发布的能力。

### 6.3 已确认不纳入本阶段的直发能力

经调研，微博开放平台的第三方分享接口本质上属于**登录授权后的商业 / 开放平台能力**，而不是适合普通访客公开页面直接调用的网页分享模板。因此：

- 本阶段**不接入**微博登录态直发。
- 文章页中的微博入口统一按“复制式分享”处理。
- 后续若要接入开放平台能力，应作为独立的账号绑定 / OAuth 能力评估，而不是混入当前文章页分享入口。

结论：**分享系统应区分“直达分享”和“复制式分享”，而不是把所有平台都强行设计成同一种能力。**

## 7. 交互设计建议

### 7.1 文章页入口位置

建议在 [pages/posts/[id].vue](../../../pages/posts/[id].vue) 的文章正文后、版权声明与赞助卡片之间，增加统一的分享组件，例如：

- [components/article-share.vue](../../../components/article-share.vue)

放置原因：

- 用户已经读完正文，分享意愿更强。
- 与版权声明、赞助卡片相邻，信息架构自然。
- 不会打断文章标题区与首屏阅读体验。

### 7.2 组件形态

桌面端建议：

- 默认展示“分享”主按钮 + 复制链接快捷按钮。
- 点击后弹出 Popover / Dialog，展示平台图标网格。
- 将平台分为“直接分享”和“复制到平台”两个分组。

移动端建议：

- 优先调用 `navigator.share(...)`。
- 若浏览器不支持，再回退到站内分享面板。

### 7.3 最低按钮集合

第一阶段推荐最小集合：

- 原生分享
- 复制链接
- 复制标题 + 摘要 + 链接
- X / Twitter
- Facebook
- LinkedIn
- Telegram
- WhatsApp
- Email
- 微信 / 微博 / 小红书 / 掘金 / bilibili / 知乎（按复制式分享展示）

## 8. 工程落点建议

### 8.1 建议新增的前端文件

| 文件 | 作用 |
| :--- | :--- |
| `components/article-share.vue` | 文章页分享组件 |
| `utils/shared/share.ts` | 平台注册表、分享 URL builder、payload 归一化 |
| `utils/shared/share.test.ts` | 分享拼链与 canonical 口径测试 |

### 8.2 与现有系统的复用点

- 图标映射优先复用当前商业化平台的共享定义，避免再维护第二套平台表。
- URL 生成优先复用 [composables/use-page-seo.ts](../../composables/use-page-seo.ts) 与 [utils/shared/seo.ts](../../utils/shared/seo.ts) 的 canonical / absolute URL 逻辑。
- i18n 文案统一收敛到 `components.post.share.*` 命名空间，避免硬编码平台说明。

## 9. 当前阶段的调研清单

在真正进入实现前，需要明确以下调研结论：

1. 已确认：微博开放平台的第三方分享接口需要 OAuth 登录、access_token 与安全域名绑定，本阶段不做站内直发，统一降级为复制式分享。
2. 是否需要在分享链接中追加 UTM；若需要，使用显式参数还是站内跳转短链。
3. 复制式分享是否需要同时提供“复制标题摘要”和“仅复制链接”两种动作。
4. 微信场景是否需要补二维码弹层，还是只保留复制链接。
5. taxonomy / list 页是否在本阶段一并开放，还是等文章页组件稳定后再扩面。

## 10. 分阶段落地建议

### 10.1 本阶段建议交付

- 完成文章详情页统一分享入口。
- 完成分享 payload、链接 builder 与平台分层。
- 完成 Web Share API + 复制链接 + 主要国际平台直拼分享。
- 为国内内容平台提供复制式分享与图标入口。

### 10.2 下一阶段可继续增强

- 分享事件埋点与归因统计。
- taxonomy / 归档 / about / 友链页扩面。
- 二维码分享增强与海报化预览。
- 短链系统或分享活动追踪。
