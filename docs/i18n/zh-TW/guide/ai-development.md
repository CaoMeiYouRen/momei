# AI 協同開發指南

::: warning 翻譯說明
本頁對應 [中文原文](../../../guide/ai-development.md)。若內容有差異，請以中文原文為準。
:::

本指南面向希望在墨梅專案中有效利用 AI 代理的開發者。我們把現代開發視為一種「人定義意圖、AI 負責執行重複性工作」的協作模式，而不是完全放任模型自行產出未經驗證的結果。

## 1. 核心理念：人機雙軌

在墨梅中，常見任務大致分成兩條路徑：

1. **傳統手動路徑**：適合簡單文字調整、特殊環境排查或 AI 無法安全覆蓋的工作。
2. **AI 增強路徑（推薦）**：適合功能開發、Bug 修復、大規模重構、測試補齊與文件同步。

關鍵並不在於「讓 AI 代替開發者」，而是讓 AI 幫你承擔搜尋、樣板生成、規範對齊與回歸驗證等高頻重複工作。

## 2. 智能體體系

專案已為常見任務預先規劃多種角色，詳見 [AGENTS.md](../../../../AGENTS.md)。常見分工如下：

| 任務類型 | 建議智能體 | 常見用途 |
| :--- | :--- | :--- |
| 新增功能 | `@full-stack-master` | 規劃、修改程式碼、補文件、跑驗證 |
| 修復 Bug | `@full-stack-master` / 對應開發角色 | 定位問題、最小修復、補回歸測試 |
| 測試增強 | `@test-engineer` | 撰寫 Vitest / E2E 測試 |
| 專案問答 | `@qa-assistant` | 只讀分析文檔與現有實作 |
| 代碼審查 | `@code-auditor` | 聚焦缺陷、風險與測試缺口 |

## 3. 標準工作流：PDTFC+

所有 AI 代理都應遵循 PDTFC+ 循環：

1. **Plan**：理解需求、讀取待辦與相關文檔，必要時反問澄清。
2. **Do**：進行實際修改，保持改動範圍聚焦。
3. **Test**：執行 lint、typecheck、定向測試或文檔站構建。
4. **Fix**：針對驗證中暴露的問題進行修正。
5. **Commit / Enhance**：在完整交付視角下補齊文件、測試、待辦與說明。

對人類開發者而言，最重要的是在 **Plan** 階段把控方向，在 **Test / Fix** 階段驗證結果，而不是只看最終生成的程式碼片段。

## 4. 對開發者的建議

- **清楚定義目標**：如果需求本身模糊，先讓 AI 幫你拆需求，而不是直接要求「實作」。
- **信任但核實**：AI 可以處理大量機械工作，但架構、權限邊界與發布風險仍要由你拍板。
- **主動提供上下文**：若你知道相關檔案、設計文檔或規範，直接告訴 AI，能顯著提升準確率。
- **關注閉環**：改完功能之後，還要追蹤測試、文檔、待辦與 UI 驗證是否同步完成。

## 5. 針對不同 AI 工具的適配

雖然專案以 GitHub Copilot 為主，但也兼容其他常見工具：

- **Claude Code**：適合大範圍閱讀文檔後再執行任務。
- **Cursor / Codex 類工具**：建議將專案指令檔納入 Rules 或工作區規範。
- **Windsurf 等流程型代理**：可以搭配 PDTFC+ 流程做更高自動化的交付。
