# 墨梅博客 长期规划与积压项 (Backlog)

本文档用于记录暂不进入当前迭代、但具备中长期价值的功能与治理事项。当前阶段执行面请参阅 [项目计划](./roadmap.md)、[待办事项](./todo.md) 与 [待办归档](./todo-archive.md)。

> **维护规则**:
> 1. 新功能需求、非阻塞优化与长期治理事项优先写入本文件，而不是直接写入 `todo.md`。
> 2. 一旦某项 backlog 被正式上收为当前/下一阶段任务，必须从本文件移除，避免重复统计。
> 3. 当前仓库的 backlog 以中文为唯一事实源；翻译文档只保留摘要或跳转说明。

## 当前 backlog
1. **README 多语言镜像存放位置重评估**
- **目录方案评估**:
    - 评估是否将 `README.<locale>.md` 迁移到 `docs/` 下的专门目录（如 `docs/readme/` 或 `docs/i18n/<locale>/readme/`），但必须先明确 GitHub 仓库入口、外部引用与文档站导航是否仍需保留根目录镜像或兼容跳板。
- **规范与链接同步**:
    - 若目录调整成立，需要同步修正文档规范、VitePress 导航、README 互跳链接、翻译治理说明与外部链接映射，避免出现“仓库入口一套、文档站一套”的双重事实源。
- **迁移门槛**:
    - 在未证明不会削弱 GitHub 首屏可读性、不会破坏现有多语言 README 入口之前，该事项维持 backlog 评估，不直接纳入当前阶段执行。

2. **AI 初始化 / 问答助手评估与工程复用治理**
	- 验收: 评估是否提供面向新部署者的 AI 初始化助手或配置问答助手，先收敛可回答范围、可写配置范围、ENV 锁定字段、越权保护与失败回退策略，再决定是否立项实现。
	- 验收: 若进入实现阶段，应优先复用现有安装向导、Settings 元信息、`qa-assistant` 与文档知识源，而不是额外维护第二套配置事实源。

3. **外部 Skills 引入与内部 Skills 可见性分层治理**
    - **内部 Skills 可见性收敛**:
        - 为项目内部维护的 `.github/skills/**/SKILL.md` 建立统一 frontmatter 约定，默认补齐 `metadata.internal: true`，避免内部技能与外部来源、平台内置技能混入同一发现面。
        - 若存在必须对外暴露的内部 skill，必须建立例外清单、触发条件与保留理由，不能靠“暂时不加 metadata”形成隐式规则。
    - **外部 Skills 文档与同步地址**:
        - 在 `docs/` 中新增一份外部 skills 接入说明，明确“项目内部维护资产 / 外部同步资产 / 平台内置技能”的边界、安装方式、同步地址、更新频率、失效处理与转内部化重写门槛。
        - 文档中至少沉淀首批候选来源与同步地址，如 `antfu/skills`、`vueuse/skills`、`vuejs-ai/skills`、`vercel-labs/agent-skills`，避免后续只记 skill 名称却丢失上游事实源。
    - **首批候选清单按当前技术栈择优**:
        - 首批优先评估与现有依赖、框架和工作流高度相关的外部 skills：`antfu`、`nuxt`、`vue`、`vite`、`vitest`、`pnpm`、`vitepress`、`vueuse-functions`、`vue-best-practices`、`vue-router-best-practices`、`vue-testing-best-practices`。
        - `pinia`、`unocss`、`slidev`、`tsdown`、`turborepo` 与通用设计类 skills 默认不进入首批必选范围；仅当仓库依赖、目录结构或使用场景明确落地后，再单独评估是否上收。
    - **治理校验与落地门槛**:
        - 若进入实现阶段，应补齐最小自动化检查，至少覆盖 frontmatter 结构、`metadata.internal` 一致性、目录名与 `name` 对齐、同步地址可追溯性，以及外部 skill 清单与文档说明的漂移检测。
        - 该事项属于非阻塞治理增强，不直接插入当前阶段；仅在后续容量与 ROI 评估满足准入条件时再正式上收。

4. **PostgreSQL 数据库流量消耗专项分析与治理**
- **观测口径先行**:
    - 在进入实现前，先明确“流量消耗大”究竟指向数据库出网流量、查询次数、连接数、慢查询、账单异常还是高频日志写入，避免在证据不足时直接扩写为数据库全面优化工程。
- **专项分析范围**:
    - 聚焦 TypeORM 连接与日志、设置读取高频查库、定时任务扫描与高频写入路径，输出最小证据链、热点路径与是否构成当前阶段阻塞的判定。
- **上收条件**:
    - 仅当观测结果证明其已阻塞当前交付、造成明显成本异常或影响线上稳定性时，再作为修复型或性能治理任务上收至正式阶段。

5. **重复代码治理与纯函数复用基线建设**
- **延期原因**:
    - 为遵循第十八阶段 5 - 6 项容量约束，在新增 WechatSync 修复与后台翻译工作流收口后，当前阶段不再继续并行推进重复代码治理。
- **治理目标保留**:
    - 输出重复纯函数、类型守卫、共享 payload 结构的分组清单，明确“可提取 / 延后 / 保持局部实现”三类结论。
    - 至少合并一组高频纯函数或共享类型，补齐最小单元测试或等价验证，并执行定向 `lint` / `typecheck` / 受影响模块回归。
- **后续准入条件**:
    - 待当前阶段的浏览器验证、性能预算、i18n 收口与 WechatSync 修复完成后，再评估是否作为单独治理任务上收。

6. **RSS 资源挂载与外部动态聚合展示**
- **RSS / RSSHub 接入层**:
    - 为 RSS 与 RSSHub 源建立统一拉取、清洗、缓存与失败降级层，支持将其他平台的动态或文章映射为站内可消费的数据结构，而不是在页面中直接拼接外部返回结果。
- **页面挂载模型**:
    - 支持将外部内容以卡片流、时间线或侧栏模块挂载到首页、作者页或独立聚合页，并明确排序、去重、来源标识、语言过滤与跳转策略。
- **稳定性与治理边界**:
    - 明确抓取频率、缓存时效、可选关闭开关、来源归属说明与解析失败回退方案，避免外部 RSS 源抖动或 RSSHub 不可用时拖垮主站页面稳定性。

7. **`caomei-auth` 第三方登录支持评估与接入预研**
- **协议与能力基线**:
    - 评估 `caomei-auth` 是否提供可被 `better-auth` / `genericOAuth` 稳定消费的 OAuth 2.1 / OIDC 能力，至少包括授权地址、令牌地址、用户信息地址或等效 discovery 信息，以及 `state`、PKCE、`nonce`、错误码和 scope 约束。
    - 明确登录、刷新、登出、令牌撤销、JWK / 签名校验与多环境回调地址的支持边界，避免接入后只能“能登录一次”，却无法稳定维护会话生命周期。
- **账户映射与资料契约**:
    - 明确 `caomei-auth` 返回的稳定用户标识、`email` / `email_verified`、昵称、头像、locale 等字段契约，确保与现有 `better-auth` 用户模型、账号绑定与多语言界面兼容。
    - 评估同邮箱合并、已存在账号绑定、第三方账号解绑与异常回退策略，避免引入重复账号或破坏现有认证链路。
- **配置、安全与反向需求输出**:
    - 梳理接入所需的 Client ID / Secret、回调 URL、权限范围、品牌文案、图标、合规提示与 ENV 锁定策略，确认哪些配置必须继续维持部署期环境变量管理。
    - 基于接入过程反推 `caomei-auth` 需要补齐的能力，例如 discovery 文档、标准 claims、刷新令牌、撤销接口、JWK 暴露、租户隔离或审计字段，再决定是否正式排期。

8. **多语言社交与赞助平台扩展**
- **平台扩容**:
    - 在现有社交 / 赞助系统上继续补齐默认平台，优先覆盖掘金、小红书、抖音等高频中文平台，和 StackOverflow、Instagram、LinkedIn、Twitch 等英文平台，并同步提供图标、颜色与展示文案约定。
- **语言一致性**:
    - 平台名称、标签与编辑选项需覆盖当前项目已支持的语言集合，避免仅有中文 / 英文而在 `zh-TW`、`ko-KR` 下退回硬编码平台名。
- **兼容边界**:
    - 扩容不得破坏作者级 / 全局级回退、locale 过滤、自定义平台与二维码 / URL 双模式展示链路。

9. **桌面端应用 (Desktop Application)**
- **Tauri 跨平台应用**:
    - 实现桌面客户端骨架，支持单站点/多站点管理。
    - 支持离线 Markdown 写作与间断性云端同步功能。
    - 提供原生菜单和系统集成，提升桌面体验。

10. **极客技术增强 (Geek Tech Extras)**
- **可执行代码块支持**:
    - 实现 Markdown 代码块在特定环境下的运行与结果输出。
    - 支持常见编程语言的代码执行（如 JavaScript、Python、Shell）。

11. **主题生态系统 (Theme Ecosystem)**
- **主题社区与发布平台**:
    - 允许创作者发布、分享并由他人安装自定义主题。
    - 建立安全审核机制防范 XSS 攻击。
- **画廊与市场**:
    - 支持保存多套自定义配置，建立主题预览、发布与管理中心。
    - 提供主题评分、评论和下载统计。
- **安全性与性能**:
    - 实现针对发布主题的自动化 XSS 扫描。
    - 使用 CSS Layer 分层策略提升主题性能。

12. **商业化与会员体系 (Monetization & Membership)**
- **支付体系集成**:
    - 集成 Stripe/支付宝支付接口。
    - 支持打赏功能或 Pro 会员准入策略。

13. **播客与多媒体扩展 (Podcast & Multimedia)**
- **全站沉浸式播放**:
    - 全站悬浮播放器，支持跨页面断点续播与内容同步。

14. **AI 视频生成与增强 (AI Video Generation & Enhancement)**
- **多模态内容产出**:
    - 探索集成视频生成模型（如 Seedance 2.0、Sora 2.0 等），支持基于文章内容或脚本生成动态视频素材。
    - 实现“文章转视频”工作流，为技术博文自动生成短视频概览或演示。

15. **文章详情相关文章推荐与内容发现增强 (Related Content Discovery Enhancements)**
- **优先级重评估**:
    - 作为第十三阶段中的可选增强项，复审后 ROI 约为 1.25，当前不满足进入阶段核心任务的准入基线。
    - 仅在配置国际化、迁移守卫与验证基线完成后，再评估是否上收至后续阶段。
- **推荐策略**:
    - 在文章详情页按分类、标签或 translation cluster 相似度提供相关文章推荐，并避免同一翻译簇重复占位。
- **展示与验证**:
    - 推荐模块应以不打断正文阅读的轻量形态落地，并补齐多语言过滤、去重与推荐规则回归测试。

16. **回归日志滚动归档后的检索与对比体验治理**
- **归档导航与索引**:
    - 在 regression-log 采用滚动归档后，评估是否需要提供按阶段、按主题或按时间的索引入口，避免历史回归记录分散后难以检索。
- **比较基线体验**:
    - 明确主日志与归档日志之间的分工，避免“当前基线在主日志、对比基线在归档中但没有跳转入口”的使用摩擦。
- **工具与文档联动**:
    - 若后续归档记录继续增多，再评估是否需要配套脚本或索引文档来辅助定位历史回归证据。
