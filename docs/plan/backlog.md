# 墨梅博客 长期规划与积压项 (Backlog)

本文档用于记录暂不进入当前迭代、但具备中长期价值的功能与治理事项。当前阶段执行面请参阅 [项目计划](./roadmap.md)、[待办事项](./todo.md) 与 [待办归档](./todo-archive.md)。

> **维护规则**:
> 1. 新功能需求、非阻塞优化与长期治理事项优先写入本文件，而不是直接写入 `todo.md`。
> 2. 一旦某项 backlog 被正式上收为当前/下一阶段任务，必须从本文件移除，避免重复统计。
> 3. 当前仓库的 backlog 以中文为唯一事实源；翻译文档只保留摘要或跳转说明。

## 当前 backlog
1. **README 多语言镜像存放位置重评估**
- **目录方案评估**:
    - 评估是否将 `README.<locale>.md` 迁移到 `docs/` 下的专门目录（如 `docs/readme/` 或 `docs/i18n/<locale>/readme/`），但必须先明确 GitHub 仓库入口、外部引用与文档站导航是否仍需保留根目录镜像或兼容跳板。
- **规范与链接同步**:
    - 若目录调整成立，需要同步修正文档规范、VitePress 导航、README 互跳链接、翻译治理说明与外部链接映射，避免出现“仓库入口一套、文档站一套”的双重事实源。
- **迁移门槛**:
    - 在未证明不会削弱 GitHub 首屏可读性、不会破坏现有多语言 README 入口之前，该事项维持 backlog 评估，不直接纳入当前阶段执行。

2. **AI 初始化 / 问答助手评估与工程复用治理**
	- 验收: 评估是否提供面向新部署者的 AI 初始化助手或配置问答助手，先收敛可回答范围、可写配置范围、ENV 锁定字段、越权保护与失败回退策略，再决定是否立项实现。
	- 验收: 若进入实现阶段，应优先复用现有安装向导、Settings 元信息、`qa-assistant` 与文档知识源，而不是额外维护第二套配置事实源。

3. **RSS 资源挂载与外部动态聚合展示**
- **RSS / RSSHub 接入层**:
    - 为 RSS 与 RSSHub 源建立统一拉取、清洗、缓存与失败降级层，支持将其他平台的动态或文章映射为站内可消费的数据结构，而不是在页面中直接拼接外部返回结果。
- **页面挂载模型**:
    - 支持将外部内容以卡片流、时间线或侧栏模块挂载到首页、作者页或独立聚合页，并明确排序、去重、来源标识、语言过滤与跳转策略。
- **稳定性与治理边界**:
    - 明确抓取频率、缓存时效、可选关闭开关、来源归属说明与解析失败回退方案，避免外部 RSS 源抖动或 RSSHub 不可用时拖垮主站页面稳定性。

4. **`caomei-auth` 第三方登录支持评估与接入预研**
- **协议与能力基线**:
    - 评估 `caomei-auth` 是否提供可被 `better-auth` / `genericOAuth` 稳定消费的 OAuth 2.1 / OIDC 能力，至少包括授权地址、令牌地址、用户信息地址或等效 discovery 信息，以及 `state`、PKCE、`nonce`、错误码和 scope 约束。
    - 明确登录、刷新、登出、令牌撤销、JWK / 签名校验与多环境回调地址的支持边界，避免接入后只能“能登录一次”，却无法稳定维护会话生命周期。
- **账户映射与资料契约**:
    - 明确 `caomei-auth` 返回的稳定用户标识、`email` / `email_verified`、昵称、头像、locale 等字段契约，确保与现有 `better-auth` 用户模型、账号绑定与多语言界面兼容。
    - 评估同邮箱合并、已存在账号绑定、第三方账号解绑与异常回退策略，避免引入重复账号或破坏现有认证链路。
- **配置、安全与反向需求输出**:
    - 梳理接入所需的 Client ID / Secret、回调 URL、权限范围、品牌文案、图标、合规提示与 ENV 锁定策略，确认哪些配置必须继续维持部署期环境变量管理。
    - 基于接入过程反推 `caomei-auth` 需要补齐的能力，例如 discovery 文档、标准 claims、刷新令牌、撤销接口、JWK 暴露、租户隔离或审计字段，再决定是否正式排期。

5. **多语言社交与赞助平台扩展**
- **平台扩容**:
    - 在现有社交 / 赞助系统上继续补齐默认平台，优先覆盖掘金、小红书、抖音等高频中文平台，和 StackOverflow、Instagram、LinkedIn、Twitch 等英文平台，并同步提供图标、颜色与展示文案约定。
- **语言一致性**:
    - 平台名称、标签与编辑选项需覆盖当前项目已支持的语言集合，避免仅有中文 / 英文而在 `zh-TW`、`ko-KR` 下退回硬编码平台名。
- **兼容边界**:
    - 扩容不得破坏作者级 / 全局级回退、locale 过滤、自定义平台与二维码 / URL 双模式展示链路。

- **推荐策略**:
    - 在文章详情页按分类、标签或 translation cluster 相似度提供相关文章推荐，并避免同一翻译簇重复占位。
- **展示与验证**:
    - 推荐模块应以不打断正文阅读的轻量形态落地，并补齐多语言过滤、去重与推荐规则回归测试。

13. **AI 媒体生成在 Vercel 场景下的超时巡检与状态补偿**
- **问题性质与分流结论**:
    - 该事项属于已交付能力（封面图生成、TTS / 播客音频生成、任务查询与自动回填）在 Vercel 部署面上的明确回归候选，而不是全新功能；但当前第二十阶段尚未归档，先按 backlog 记录，避免未经准入直接膨胀当前 todo。
    - 若后续确认该问题稳定复现于线上或默认 Vercel 部署主路径，且会导致媒体生成任务长期卡死、状态不同步或用户误判任务失败，可按修复型 Hotfix 上收为当前阶段插队事项。
- **巡检与超时补偿**:
    - 为图像生成与播客音频生成任务补齐统一的超时阈值、长轮询失联检测、状态回写补偿与最终失败落点，避免任务在长轮询后仍停留在 `processing` 或等价中间态。
    - 定时任务应增加“超时任务扫描”或等价机制，主动处理长时间无心跳 / 无结果回写的任务，而不是只依赖前端轮询自然超时。
- **Vercel 部署边界**:
    - 明确区分“云函数超时导致生成未完成”“供应商任务已完成但站内状态未同步”“轮询链路丢失结果回调”三类失败来源，避免继续把所有失败都归因于单一长轮询问题。
    - 验收时需覆盖 Vercel 定时任务触发、任务状态回补、失败态可见性与重复扫描幂等性，避免补偿逻辑二次覆盖已成功结果。

14. **默认封面生成文案压缩与大字号策略优化**
- **体验目标**:
    - 优化默认封面生成策略，使封面主文案更短、更聚焦，并在默认版式下呈现更大的字号，减少当前“标题过长、字号偏小、主视觉不够聚焦”的问题。
    - 该事项属于 AI 视觉资产链路的体验增强，不构成当前阶段阻塞，默认维持 backlog 候选。
- **策略收敛**:
    - 为默认封面文案建立“优先短标题 / 关键词摘要 / 显式封面短句”的选择顺序，而不是直接把完整文章标题原样塞入封面。
    - 在既有五维提示词模型基础上，补齐默认文本长度、字号级别、换行规则与超长文案截断策略，确保不同语言下的版式都可预测。
- **验证要求**:
    - 至少覆盖中英文及其他长词语言场景下的默认封面回退效果，避免仅对中文短标题优化却让英文或多语言封面更易溢出。
    - 需同时验证“自动回填封面”和“仅生成候选图待人工确认”两条链路的默认视觉表现一致性。

15. **文章批量翻译到其他语言的编排能力评估**
- **能力定位**:
    - 当前系统已具备单篇 `translate-post`、CLI / MCP 任务编排与预览确认流；本事项评估的是将其上抬为“多篇文章批量翻译到多个目标语言”的运营级能力，而不是重新实现第二套翻译引擎。
    - 该事项属于后续能力扩展，默认进入 backlog，不直接并入当前质量治理阶段。
- **最小能力边界**:
    - 批量操作需支持文章范围选择、目标语言矩阵、预览确认、任务进度、失败明细与人工重试，不得把批量翻译设计成不可回滚的黑盒一次性提交。
    - 必须优先复用现有 `translationId`、翻译范围、标签 / 分类映射、封面 / 音频默认不同步策略与 AI 任务状态模型，而不是在批量入口复制一套旁路逻辑。
- **准入前问题**:
    - 需要先评估配额 / 成本治理、任务并发上限、批量失败重入、已存在目标语言草稿时的覆盖策略，以及批量运行对 Vercel / Serverless 超时预算的影响。
    - 若没有明确的并发节流、进度汇总与取消策略，该能力不应直接进入开发。

16. **标签翻译前置化与正文翻译解耦优化**
- **问题定义**:
    - 当前标签翻译虽然已纳入翻译工作流进度展示，但用户反馈其完成时机仍偏后，体验上像是在等待正文翻译收尾后才最后处理标签，影响“元数据先可用、正文后续渐进回填”的效率感知。
    - 该事项属于既有翻译编排体验优化，不是 taxonomy 基础能力缺失。
- **优化方向**:
    - 评估将标签翻译前置为与标题 / 摘要同级的早期步骤，或在正文长文本分段执行前先完成标签翻译与目标语言标签绑定，减少用户等待全文结束后才能检查标签结果的延迟。
    - 保持标签翻译与既有 `translationId -> slug -> id` 绑定口径一致，不得为了“更快出结果”重新退化为按名称碰撞创建标签。
- **验收与风险边界**:
    - 验收应覆盖标签先完成、正文仍在流式 / 分段回填、用户中途取消、标签翻译失败但正文继续执行等状态分叉，确保进度条、失败态与重试入口不互相污染。
    - 若后续实现，需要与批量翻译、单篇翻译、CLI / MCP 自动化三个入口保持同一编排语义，避免出现“后台编辑器一套顺序、外部自动化另一套顺序”的双重事实源。
