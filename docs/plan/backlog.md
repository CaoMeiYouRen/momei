# 墨梅博客 长期规划与积压项 (Backlog)

本文档用于维护尚未进入正式阶段执行面的统一候选池，并按“长期主线任务”与“短期 / 一次性候选任务”双轨区分。当前阶段执行面请参阅 [项目计划](./roadmap.md)、[待办事项](./todo.md) 与 [待办归档](./todo-archive.md)。

> **维护规则**:
> 1. 新功能需求、非阻塞优化与长期治理事项优先写入本文件，而不是直接写入 `todo.md`。
> 2. backlog 必须区分“长期主线任务”和“短期 / 一次性候选任务”：长期主线可跨阶段保留，短期 / 一次性候选在正式上收后必须去重。
> 3. 长期主线被某阶段抽取后，不删除主线卡片，只补记最近一次上收阶段、当前状态与下一次可切片方向。
> 4. 当前仓库的 backlog 以中文为唯一事实源；翻译文档只保留摘要或跳转说明。

## 现状摘要与方向判断

> 本区基于 `roadmap.md`、`todo-archive.md`、`CHANGELOG.md` 与当前源码 / 文档事实源整理，用于 backlog 背景判断与后续切片参考；第二十八阶段已正式开启执行，当前中文 `roadmap.md` 与 `todo.md` 为阶段事实源。

- **已实现能力摘要**:
    - 产品主干已经基本闭环：内容管理、AI 创作 / 翻译 / 配图 / 语音、多语言内容治理、第三方分发、通知 / 营销、主题与设置中心、上传与对象存储、迁移 CLI / MCP、审计与 Review Gate、周期性回归与发布门禁均已落地。
    - 工程治理基础也已具备：覆盖率、ESLint 收紧、重复代码、依赖安全、性能预算、文档事实源、外部 skills 准入等都有明确入口与证据链。
- **当前主要缺口**:
    - 部署与初始化体验仍偏重，能力很多，但对新部署者而言配置成本、排障心智负担和“最小可运行路径”仍有优化空间。
    - Cloudflare 仍停留在 R2 / Scheduled Events 等外围能力层，应用主体受 TypeORM 与 Node 运行时依赖限制，尚未形成可维护的运行时适配路径。
    - 编辑器链路仍依赖 `mavon-editor` 的异步包装接入，说明当前“能用”但不代表“后续可持续演进”；正文渲染、翻译编辑与预览一致性仍有继续打磨空间。
    - 国际化链路虽然已经具备模块化与动态加载能力，但“locale 模块归属、运行时加载命中、共享组件文案边界”仍缺少周期性治理闭环；这类问题容易表现为翻译字段未生效、raw key 直出或跨页面复用后命名空间漂移。
    - PostgreSQL 数据库流量与高频读路径的结果集体量仍缺少长窗口治理闭环；当前已完成首轮热点收敛，但尚未形成持续可复核的热点画像、缓存策略与长期趋势对比口径。
    - 构建速度、包体预算与高频页面依赖收敛已经建立守线，但仍是持续性成本，不宜视为已彻底完成。
    - 长期治理主线仍应继续切片推进，特别是测试有效性、ESLint / 类型债、重复代码与注释治理，不能因为上一阶段已做一轮就退出 backlog 主线。
- **方向判断**:

1. **`caomei-auth` 第三方登录支持评估与接入预研**

- **配置、安全与反向需求输出**:

9. **GEO / SEO / AI crawler 可见性与文章可引用性补强（候选）**
- **候选背景（2026-04-15 补记）**:
    - 公开端点抽样显示：`robots.txt` 与 `sitemap.xml` 可访问，但历史上 `feed.xml` / `feed.atom` / `feed.json` 曾出现 `500` 风险，需继续保持回归守线。
    - 线上尚未提供 `llms.txt` 或同等 AI 入口说明时，AI 搜索系统对站点结构、重点页面与引用入口的理解仍偏弱。
    - 当前结构化数据以 `WebSite` / `CollectionPage` / `BlogPosting` 为主；文章层对“摘要、关键要点、FAQ、作者实体、更新时间”等更利于 AI 引用的信号仍有增强空间。
- **候选目标**:
    - 补齐 AI 搜索入口、爬虫访问口径与实体信号，提升 ChatGPT / Claude / Perplexity / Gemini 等系统对站点与文章的可发现性、可理解性与可引用性。
    - 让文章内容在不牺牲可读性的前提下，更适合被 AI 搜索系统抽取摘要、答案片段与结论引用，而不是只追求传统 SEO 排名指标。
    - 在不扩大当前阶段范围的前提下，形成可复用的 GEO 验证基线（端点可用性 + 元信息 + schema + AI crawler 策略）。
- **建议切片（由高到低）**:
    - 切片 A（P0）：`feed` 三协议路由稳定性与公开分发回归守线（`/feed.xml`、`/feed.atom`、`/feed.json`）。
    - 切片 B（P1）：新增 `llms.txt`、AI crawler 访问说明与站点级 AI 入口导航，并与 `robots` / sitemap 保持一致。
    - 切片 C（P1）：结构化数据增强（`Organization.sameAs`、`WebSite.SearchAction`、`BreadcrumbList`、按页面条件补 `FAQPage`）。
    - 切片 D（P1）：文章级可引用性增强，例如稳定摘要、关键结论区块、常见问题、作者/更新时间信号与更清晰的标题层级。
    - 切片 E（P2）：统一 canonical 与 `og:url` 斜杠策略，继续减少重复 URL 与信号分散。
- **准入与上收门槛**:
    - 若公开 feed / sitemap / crawler 相关链路再次出现持续性异常，并影响索引抓取或订阅可用性，可按 `planning.md` 的 B 类（回归修复）评估是否插队进入当前阶段。
    - 其余 GEO / AI 搜索优化默认继续维持 backlog 候选，待当前阶段收口后统一评估，不直接扩写成当期大任务。
- **最低验收口径（上收后执行）**:
    - 公开端点：`robots` / `sitemap` / `feed` / `llms` 的状态码与内容校验脚本可复用。
    - 页面元信息：`canonical`、`hreflang`、OG、Twitter、JSON-LD 抽样验证通过，并至少覆盖首页、文章页、分类页三类入口。
    - 内容可引用性：至少抽样验证 `3 - 5` 篇文章在摘要、FAQ / 要点区块或结构化数据增强后，未破坏现有渲染、国际化与阅读体验。
    - 回归留痕：将验证命令、结果摘要与剩余风险写入 `docs/reports/regression/current.md` 并在 Review Gate 引用。
