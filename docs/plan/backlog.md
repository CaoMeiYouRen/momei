# 墨梅博客 长期规划与积压项 (Backlog)

本文档用于维护尚未进入正式阶段执行面的统一候选池，并按“长期主线任务”与“短期 / 一次性候选任务”双轨区分。当前阶段执行面请参阅 [项目计划](./roadmap.md)、[待办事项](./todo.md) 与 [待办归档](./todo-archive.md)。

> **维护规则**:
> 1. 新功能需求、非阻塞优化与长期治理事项优先写入本文件，而不是直接写入 `todo.md`。
> 2. backlog 必须区分“长期主线任务”和“短期 / 一次性候选任务”：长期主线可跨阶段保留，短期 / 一次性候选在正式上收后必须去重。
> 3. 长期主线被某阶段抽取后，不删除主线卡片，只补记最近一次上收阶段、当前状态与下一次可切片方向。
> 4. 当前仓库的 backlog 以中文为唯一事实源；翻译文档只保留摘要或跳转说明。

5. **存量代码注释治理与注释漂移收敛**

6. **Postgres 查询与数据库出网流量治理**
- **目标**:
    - 继续压缩 PostgreSQL 到应用层的结果集体量与高频重复读取，优先降低数据库出网流量与查询压力，而不是先追求 CPU 指标最优。
    - 建立“热点查询识别 -> 结果集裁剪 / 读取收敛 / 短 TTL 缓存 -> `pg_stat_statements` 或等价证据复核”的闭环，避免只凭连接数或主观体感优化。
- **状态**:
    - 进行中。
- **当前状态**:
    - 历史专项已完成公开设置链路与定时任务扫描的首轮治理，并在 PostgreSQL 样本中验证过 `setting` 读取与定时发布扫描的结果集体量已下降。
    - 当前剩余热点更可能集中在公开列表型接口、归档/分类/标签查询、设置读取复用边界，以及缺少服务端短 TTL 缓存或最小字段集约束的高频读路径；这些问题会直接放大数据库返回流量，并可能伴随不必要的 CPU 唤醒，但 CPU 仍属于次级观察指标。
- **最近一次上收阶段**:
    - 第二十六阶段（已审计归档）。
- **下一次可切片方向**:
    - 候选组 A：为 `posts / archive / categories / tags` 等公开高频读接口补齐热点画像、最小字段集检查与必要的短 TTL 服务端缓存，优先减少宽结果集与重复查库。
    - 候选组 B：为数据库层补充更可执行的观测口径，例如 `pg_stat_statements` 长窗口采样、热点 SQL 分组、结果集大小估算或查询指纹统计，先回答“到底哪些请求最耗流量”。
    - 候选组 C：在不破坏一致性的前提下，审计设置读取、locale 读取与公开页面装配链路中是否仍有可合并的重复查库；必要时复用现有 Redis / LRU 基础设施，但不默认扩写成全站缓存工程。


## 现状摘要与方向判断

> 本区基于 `roadmap.md`、`todo-archive.md`、`CHANGELOG.md` 与当前源码 / 文档事实源整理，只用于下一阶段候选分析，不构成正式 roadmap / todo。

- **已实现能力摘要**:
    - 产品主干已经基本闭环：内容管理、AI 创作 / 翻译 / 配图 / 语音、多语言内容治理、第三方分发、通知 / 营销、主题与设置中心、上传与对象存储、迁移 CLI / MCP、审计与 Review Gate、周期性回归与发布门禁均已落地。
    - 工程治理基础也已具备：覆盖率、ESLint 收紧、重复代码、依赖安全、性能预算、文档事实源、外部 skills 准入等都有明确入口与证据链。
- **当前主要缺口**:
    - 部署与初始化体验仍偏重，能力很多，但对新部署者而言配置成本、排障心智负担和“最小可运行路径”仍有优化空间。
    - Cloudflare 仍停留在 R2 / Scheduled Events 等外围能力层，应用主体受 TypeORM 与 Node 运行时依赖限制，尚未形成可维护的运行时适配路径。
    - 编辑器链路仍依赖 `mavon-editor` 的异步包装接入，说明当前“能用”但不代表“后续可持续演进”；正文渲染、翻译编辑与预览一致性仍有继续打磨空间。
    - PostgreSQL 数据库流量与高频读路径的结果集体量仍缺少长窗口治理闭环；当前已完成首轮热点收敛，但尚未形成持续可复核的热点画像、缓存策略与长期趋势对比口径。
    - 构建速度、包体预算与高频页面依赖收敛已经建立守线，但仍是持续性成本，不宜视为已彻底完成。
    - 长期治理主线仍应继续切片推进，特别是测试有效性、ESLint / 类型债、重复代码与注释治理，不能因为上一阶段已做一轮就退出 backlog 主线。
- **方向判断**:
    - 第二十五阶段默认应以“打磨现有能力 + 平台边界研究 + 长期治理切片”为主，而不是继续扩写新的大功能面。
    - 新能力只建议保留两类：一类是直接降低部署 / 使用门槛的启用型能力；另一类是能回答关键平台方向问题的受控预研。
    - 与当前阶段最契合的收口方向，不是“再做一个新系统”，而是把部署体验、编辑器体验、构建与包体、Cloudflare 边界、数据库流量治理，以及代码可维护性继续向前推一轮。

### 延后新增能力保留池（当前不建议优先上收）

1. **`caomei-auth` 第三方登录支持评估与接入预研**
- **协议与能力基线**:
    - 评估 `caomei-auth` 是否提供可被 `better-auth` / `genericOAuth` 稳定消费的 OAuth 2.1 / OIDC 能力，至少包括授权地址、令牌地址、用户信息地址或等效 discovery 信息，以及 `state`、PKCE、`nonce`、错误码和 scope 约束。
    - 明确登录、刷新、登出、令牌撤销、JWK / 签名校验与多环境回调地址的支持边界，避免接入后只能“能登录一次”，却无法稳定维护会话生命周期。
- **账户映射与资料契约**:
    - 明确 `caomei-auth` 返回的稳定用户标识、`email` / `email_verified`、昵称、头像、locale 等字段契约，确保与现有 `better-auth` 用户模型、账号绑定与多语言界面兼容。
    - 评估同邮箱合并、已存在账号绑定、第三方账号解绑与异常回退策略，避免引入重复账号或破坏现有认证链路。
- **配置、安全与反向需求输出**:
    - 梳理接入所需的 Client ID / Secret、回调 URL、权限范围、品牌文案、图标、合规提示与 ENV 锁定策略，确认哪些配置必须继续维持部署期环境变量管理。
    - 基于接入过程反推 `caomei-auth` 需要补齐的能力，例如 discovery 文档、标准 claims、刷新令牌、撤销接口、JWK 暴露、租户隔离或审计字段，再决定是否正式排期。
