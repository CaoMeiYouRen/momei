1. **配置项多语言国际化与回退治理 (P0)**:
    - **多语言配置建模**: 为站点标题、关键词、公告、SEO 文案、法律文本简介等管理员可编辑配置建立多语言存储结构，并兼容现有单字符串配置。
    - **回退链统一**: 明确 `zh-TW -> zh-CN -> en-US` 与 `当前 locale -> en-US` 的读取顺序，并统一公开设置、SEO、邮件与通知链路的消费口径。
    - **迁移与回归**: 补齐旧配置兼容迁移、缺词回退与跨语言读取测试，避免配置国际化引入新的展示漂移。
2. **AI 翻译扩展到可生成多语言的配置与法律文本 (P1)**:
    - **可翻译配置扩面**: 将 AI 翻译能力扩展到公告、站点介绍、SEO 文案等配置输入入口，降低管理员逐语种重复录入成本。
    - **法律文本审校闭环**: 为用户协议、隐私政策等合规文本建立“AI 草案 -> 人工审校 -> 激活生效”的流程，不将 AI 输出直接视为可生效的法律版本。
    - **与配置模型联动**: 让 AI 生成结果直接兼容多语言配置模型与 fallback 规则，避免公告、法律页与设置中心形成三套多语口径。
3. **迁移导入 Slug 守卫与路径别名校验 (P1)**:
    - **导入前校验**: 在 CLI / API 导入文章时，对 `slug`、`permalink`、`abbrlink` 执行格式、保留字、冲突与语言维度唯一性校验。
    - **冲突处理策略**: 默认尊重源站合法别名，仅在非法、冲突或语义不完整时进入显式修复、回退或人工确认分支。
    - **批量导入审计**: 为跳过、修复、回退与冲突处理生成结构化报告，确保迁移流程可审计、可复跑。
4. **Cloudflare 部署支持声明纠偏 (P1)**:
    - **平台能力声明收口**: 修正文档、快速开始、功能介绍与设计说明中“可直接部署到 Cloudflare Pages / Workers”的表述，明确当前受 TypeORM 运行时限制，项目暂不支持在 Cloudflare 运行时完整部署。
    - **能力边界区分**: 区分“Cloudflare R2 / Scheduled Events 等外围能力可接入”与“应用主体可在 Cloudflare 运行时部署”两类能力，避免用户误把局部支持理解为整站可部署。
    - **部署矩阵同步**: 同步更新部署矩阵、平台比较、排障文档与对外入口文案；若未来替换 ORM 或形成可行适配层，再作为独立规划项重新评估。
5. **后台配置页动作与反馈入口统一 (P1)**:
    - **保存入口统一**: 将系统设置及其他后台配置页的保存动作统一为右下角悬浮样式，减少长页表单的保存查找成本。
    - **反馈入口显性化**: 将反馈入口收敛为更明确的问号图标与一致文案，并在公开侧与后台侧统一显隐逻辑。
    - **可用性回归**: 补齐滚动场景、移动端 / 窄屏场景与固定按钮遮挡测试，避免统一交互反而引入误触。

    - **验证与证据**: 输出本轮规则上收结论、残余债务清单与下一轮候选规则建议，便于阶段收口与后续准入复用。

### 第三十一阶段：认证预研与治理执行面正式上收 (Auth Evaluation & Governance Execution Promotion) (规划中)

**时间表**: 2026-04-22 ~ 2026-05-13（预计三周）
**目标**: 在第三十阶段完成远程仓库同步与治理基线细化后，继续沿“`1` 个新功能预研 + `4` 个治理切片 + `1` 个战略评估”的受控组合推进当前阶段执行面：以 `caomei-auth` 第三方登录支持评估与接入预研作为唯一新增功能候选，同时正式上收路线图 / Todo 深度归档治理、国际化运行时加载与文案复用治理、ESLint / 类型债治理、测试覆盖率与有效性治理，以及商业化转型可行性重评六条主线。

**准入结论**: 六条主线均已具备执行范围、非目标、验收标准与最小验证矩阵，总数控制在 `6` 项内，处于规划规范允许上限。新增功能仍仅保留 `caomei-auth` 预研这一条，且明确要求先完成协议能力、字段契约与账号绑定边界评估，再决定是否进入真实接入实现；其余 5 条中，“路线图 / Todo 深度归档治理”与“商业化转型可行性重评”属于短期 / 一次性候选正式上收，“国际化运行时加载与文案复用治理”“ESLint / 类型债治理”“测试覆盖率与有效性治理”继续保持 backlog 长期主线属性，并在本轮以受控切片方式进入当前阶段。

**执行约束**: 本阶段所有主线必须继续遵守“先范围、后实现；先设计、后扩面”的口径。`caomei-auth` 预研在未完成协议能力、字段映射与账号绑定风险判断前，不得直接切入登录按钮或回调实现；文档深度归档不得改写既有阶段完成事实；覆盖率与 ESLint 治理不得借机扩写为全仓清债工程；商业化重评不得提前进入支付、会员或价格页实现。

**ROI 评估**: `caomei-auth` 第三方登录支持评估与接入预研 `1.33`；路线图 / Todo 深度归档治理 `1.40`；国际化运行时加载与文案复用治理 `1.60`；ESLint / 类型债治理（`composables` 子桶）`1.50`；测试覆盖率与有效性治理 `1.83`；商业化转型可行性重评 `1.20`。其中测试覆盖率与国际化运行时治理为 P0 主线；文档深度归档与商业化重评虽然不属于最高 ROI，但均已满足“可验收、边界清晰、不会与当前主线重复”的正式上收条件。

1. **主线：`caomei-auth` 第三方登录支持评估与接入预研 (P1)**:
    - **执行范围**: 基于 `better-auth` 当前 `genericOAuth({ config: [] })` 的空配置入口，核对 `caomei-auth` 的 Discovery、授权、令牌、用户信息、JWKS、动态客户端注册与刷新令牌能力，明确字段映射、同邮箱合并、账号绑定、回调地址与部署期 ENV 锁定边界，并给出“允许进入实现 / 需上游补齐 / 暂缓接入”的三选一结论。
    - **非目标**: 不在本阶段直接落地真实登录按钮、回调处理或账号绑定实现，不重写现有 GitHub / Google OAuth 接入，也不在未确认 PKCE / revocation / 稳定 claims 前提前承诺真实接入。
    - **验收标准**: 已明确本地接入锚点、上游协议能力、稳定字段映射、`PKCE` / `offline_access` / `locale` claim / revocation 的当前状态，并形成专项预研文档，足以支撑后续是否进入实现阶段的决策。
    - **验证与证据**: 至少完成本地 `genericOAuth` 接入面核对、`caomei-auth` OAuth / OIDC 文档核对与阻塞项清单沉淀，证据统一回链到专项预研文档。

2. **主线：路线图 / Todo 深度归档治理 (P1)**:
    - **执行范围**: 对 `roadmap.md` 与 `todo-archive.md` 执行首轮深度归档，优先剥离更早阶段正文，只在主文档保留近线窗口、摘要与索引，并同步补齐 `docs/plan/archive/index.md` 的区间分片规则、兼容入口与回链说明。
    - **非目标**: 不改写既有阶段结论，不把所有 `docs/plan/` 文档都重做一次，也不把本轮扩写成多语翻译同步工程。
    - **验收标准**: `roadmap.md` 与 `todo-archive.md` 至少完成一轮可追溯的正文分片与索引回链；若未能完全回到健康窗口，必须明确记录剩余阻塞原因与下一步拆分点。
    - **验证与证据**: 量化归档前后主文档行数、索引与兼容链接有效性，并通过 Markdown 文档质量门。

3. **主线：国际化运行时加载与文案复用治理 (P0)**:
    - **执行范围**: 把治理重点收敛到“运行时加载命中 + 共享命名空间准入 + 缺词 blocker 口径”，扩大 `i18n:verify:runtime` 覆盖面，至少命中一条公开页装配链路与一组跨公开页 / 后台共享组件文案。
    - **非目标**: 不发起全仓 i18n 大迁移，不把 `unused` 字段清理升级为本轮 blocker，也不把所有相同中文文本都强制提取成共享 key。
    - **验收标准**: `i18n:audit:missing` 继续保持 `0` blocker；共享 key 上收准入标准、raw key / 缺词问题定级口径与下一批高风险历史热点均已明确落盘。
    - **验证与证据**: 至少补跑 `i18n:audit:missing`、`i18n:verify:runtime` 与必要的 parity / 运行时场景验证，并把结果回写到活动回归窗口。

4. **主线：ESLint / 类型债治理（`composables` 子桶） (P1)**:
    - **执行范围**: 冻结下一轮治理目标为 `@typescript-eslint/no-non-null-assertion` 在 `composables/` 子桶中的窄切片，先产出命中清单、回滚边界与替代写法策略，再执行有限范围清理。
    - **非目标**: 不并行开启第二条全仓 ESLint 规则治理，不把 `any`、`unsafe-*`、`max-lines` 等其他长期主线打包进同一项，也不为了清零规则牺牲可读性。
    - **验收标准**: 已冻结本轮 `composables/` 子桶命中清单，目标子桶中的非空断言已替换为显式守卫、默认值、类型收窄或提前返回，且规则收紧未外溢到非目标目录。
    - **验证与证据**: 至少完成目标子桶 ESLint 校验与窄范围类型检查，并把本轮切片与回滚边界沉淀到专项治理文档。

5. **主线：测试覆盖率与有效性治理 (P0)**:
    - **执行范围**: 以高风险运行时链路为优先，围绕认证配置边界、locale 归一化 / i18n 运行时加载，以及共享组件文案命中补齐高价值测试，并强调失败断言的红绿有效性。
    - **非目标**: 不把本轮写成全仓 coverage 冲 `80%` 的铺量工程，不接受只有 snapshot 的低价值补测，也不以“测试成本高”为由跳过高风险链路的失败用例。
    - **验收标准**: 全仓 coverage 基线不低于当前 `76%+` 水位；至少补齐一组“命名空间漂移 / raw key 暴露 / 认证配置退化时会失败”的行为断言，并说明新增覆盖与未覆盖边界。
    - **验证与证据**: 至少补跑 coverage、i18n 运行时回归与目标测试文件，并把新增失败用例覆盖的真实风险回写到活动回归窗口。

6. **主线：商业化转型可行性重评 (P1)**:

    - **验收标准**: 已按统一评分维度完成明确打分；已明确一个最值得继续验证的付费增强能力，或明确说明当前尚不存在该主卖点；已指出统一承接入口的优先落点，并把下一阶段是否允许继续上收商业化实现写成清晰条件。
    - **验证与证据**: 以专项框架文档、当前站点入口核对结果与规划同步状态作为最小证据链，输出可被后续规划直接复用的决策结论。



## 3. 技术架构与质量保证

### 技术栈

-   **前端/框架**: Nuxt 4 (Vue 3 + TypeScript)。
-   **样式**: SCSS + PrimeVue。
-   **数据库**: SQLite (本地/演示模式) / MySQL / PostgreSQL (生产推荐)。
-   **认证**: Better-Auth (GitHub/Google/Credentials)。
-   **部署**: Vercel / Docker (支持 VPS 手动部署存储)。

### 质量保证策略

1.  **测试**:
    -   **单元测试**: 使用 Vitest 测试核心逻辑 (slug 生成，数据转换)。
    -   **E2E**: 使用 Cypress/Playwright 测试关键路径 (登录 -> 发布)。
    -   **覆盖率目标**: 核心功能覆盖率分阶段提升到 80% 以上，并以每阶段约 5% 的增幅持续推进、记录基线与未覆盖边界。
    -   在近期计划中为归档页与移动导航补充测试并纳入 CI。
2.  **性能**:
    -   **SSG/SSR**: 尽可能预渲染静态内容。
    -   **图片优化**: 使用 `<NuxtImg>`。
    -   **监控**: CI/CD 中的 Lighthouse 审计。

-   Microsoft Clarity 集成 - 行为分析和会话回放工具集成
