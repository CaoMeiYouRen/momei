### 第二十二阶段：质量有效性与创作编排治理深化 (Quality Effectiveness & Authoring Orchestration Deepening, Archived)

**时间表**: 约 1 - 1.5 个月
**目标**: 在第二十一阶段完成验证入口与创作工作流收口之后，继续把收益最高的“测试有效性、周期性回归执行节奏、批量翻译编排能力、ESLint 规则守线、时间与日期事实源一致性，以及后台高频入口效率”正式上收为当前阶段，避免后续治理再次回到“规范存在但执行不连续、能力存在但编排不可扩展”的状态。

**审计结论**: 第二十二阶段六条主线已在实现代码、测试 / 验证、回归记录、设计文档与规划文档中完成闭环。`todo.md` 已清理、`todo-archive.md` 已收录阶段归档块、[回归活动窗口](../reports/regression/current.md) 已完成滚动归档并回到窗口内，因此本阶段满足归档条件。下一阶段目前仅保留候选分析，不在本轮直接上收为正式 Phase 23。

**ROI 评估**: 测试有效性增强治理 2.00；周期性回归任务实盘化 2.25；文章批量翻译到其他语言的编排能力评估 1.29；ESLint 规则分阶段收紧治理 1.17；时间配置解析与日期事实源收敛 1.75；后台文章管理入口前置到顶栏 2.50。其中特别适合作为本阶段主线起点的是测试有效性、周期性回归实盘化、时间 / 日期事实源收敛与后台高频入口优化；批量翻译编排评估与 ESLint 规则收紧则以受控副线方式纳入。

1. **主线：测试有效性增强治理 (P0)**:
    - **红绿测试优先**: 围绕一组高回归风险模块或链路引入“先失败、后修复、再守线”的节奏，减少只测成功路径的覆盖偏差。
    - **边缘 / 失败路径补强**: 优先覆盖异常输入、失败分支、回退逻辑与边界条件，而不是把测试补强退化为机械堆 coverage。
    - **证据可复盘**: 每轮补强都要在回归记录中保留失败用例、转绿结果、未覆盖边界与下一轮优先范围。
2. **主线：周期性回归任务实盘化 (P0)**:
    - **从规范到节奏**: 将既有周期性回归规范进一步收敛为周级、发版前与阶段收口前的最小执行组合与责任边界。
    - **回归产物统一**: 继续统一沉淀到 [回归活动窗口](../reports/regression/current.md)，并明确何时滚动归档、何时升级为 blocker，而不是新增第二套记录面。
    - **高漂移项优先**: 优先覆盖测试 / coverage、依赖安全、文档同步、脚本入口与性能基线等高漂移治理面。
3. **主线：文章批量翻译到其他语言的编排能力评估 (P1)**:
    - **能力边界先行**: 先明确文章范围选择、目标语言矩阵、预览确认、失败重入、取消与人工重试，不直接把批量翻译设计成黑盒式批处理。
    - **复用现有事实源**: 批量能力必须复用现有 `translationId`、翻译范围、标签 / 分类映射、封面 / 音频默认策略与任务状态模型。
    - **准入判断明确**: 即便本阶段不完整实现，也必须形成成本、并发、超时预算与覆盖策略的准入结论。
4. **主线：ESLint 规则分阶段收紧治理 (P1)**:
    - **基线先行**: 先建立现有规则豁免、warning 债务与高频误报点基线，明确首批要收紧的规则族和目录范围。
    - **分批可回滚**: 每一批规则提升都要附带预期收益、回滚方式和最小验证矩阵，避免一次性强收紧导致返工失控。
    - **范围受控**: 测试、脚本、迁移和历史遗留目录继续保持独立豁免边界。
5. **主线：时间配置解析与日期事实源收敛 (P1)**:
    - **时长配置统一**: 对涉及超时、轮询、缓存 TTL 与等待窗口的毫秒型配置优先收敛到 `ms` 包解析，减少手写换算与字符串 / 数值混用。
    - **日期逻辑优先 `dayjs`**: 业务逻辑中的日期格式化、比较、排序、区间判断继续以 `dayjs` 为首选，模板保持 `useI18nDate`，`new Date()` 只保留在运行时边界。
    - **治理必须落代码**: 本主线至少补齐一组“时间配置解析 + 日期逻辑替换”的定向验证，不允许只停留在规范层。
6. **主线：后台文章管理入口前置到顶栏 (P1)**:
    - **高频入口可达性提升**: 将文章管理入口提升到顶栏或等价高频导航位，降低后台创作流的进入成本。
    - **响应式与权限边界**: 必须同步考虑不同角色、不同视口与当前顶栏信息密度，不得通过简单堆入口破坏现有导航平衡。
    - **最小回归补齐**: 至少验证桌面 / 窄屏下的入口显隐、跳转与角色边界。

### 第二十三阶段：全球触达扩展与运行时稳态治理 (Global Reach Expansion & Runtime Stability Governance) (已审计归档)

**时间表**: 约 1 - 1.5 个月
**目标**: 在第二十二阶段完成质量与编排收口之后，继续把用户可感知收益和运行期稳态价值都较高的五类事项正式收敛为下一阶段候选主线，围绕“多语言触达能力扩面、Vercel 场景下 AI 媒体任务补偿、依赖安全巡检自动化、实时 ASR 会话稳定性提升，以及文章详情页内容发现增强”形成新的阶段组合，避免治理面与体验面长期分离推进。

**审计结论**: 第二十三阶段五条主线已在实现代码、定向测试、统一调度入口、规划文档与多语路线图摘要中完成闭环。`todo.md` 已清理当前阶段正文，`todo-archive.md` 已收录阶段归档块；后续正式上收的第二十四阶段见下文当前规划，不影响本阶段归档结论。

**ROI 评估**: 多语言社交与赞助平台扩展 1.60；AI 媒体生成在 Vercel 场景下的超时巡检与状态补偿 1.75；依赖包风险每日巡检自动化 1.60；实时语音识别鉴权有效期优化 1.50；文章详情相关文章推荐与内容发现增强 1.50。五项均已按本阶段准入边界完成收口。

1. **主线：多语言社交与赞助平台扩展 (P1)**:
    - **平台扩容**: 在现有社交 / 赞助系统上补齐中英文高频平台，并同步提供图标、颜色与展示文案约定。
    - **语言一致性**: 平台名称、标签与编辑选项需覆盖当前已支持语言集合，避免在 `zh-TW`、`ko-KR` 等语种下退回硬编码平台名。
    - **兼容边界**: 已保持作者级 / 全局级回退、locale 过滤，以及二维码 / URL 双模式展示链路稳定。
2. **主线：AI 媒体生成在 Vercel 场景下的超时巡检与状态补偿 (P0)**:
    - **任务补偿闭环**: 为图像生成与播客音频生成任务补齐超时阈值、失联检测、状态回写补偿与最终失败落点。
    - **调度巡检**: 增加定时扫描或等价机制，主动处理长时间无心跳 / 无结果回写的任务，避免前端轮询自然超时后长期卡在中间态。
    - **部署边界**: 已明确区分云函数超时、供应商已完成但站内未同步、轮询链路丢失结果三类失败来源，并补齐幂等验证。
3. **主线：依赖包风险每日巡检自动化 (P0)**:
    - **调度复用**: 复用现有 `security:audit-deps` 与依赖风险检查脚本，增加每天执行一次的自动调度入口。
    - **结果留痕**: 至少产出结构化结果摘要，能够区分“无高危风险”“发现可修复风险”“审计执行失败”三类结论，并保留近期可追溯记录。
    - **最小告警**: 当发现 `high` 及以上级别风险或执行失败时，当前已提供 GitHub issue 与 failed workflow + artifact 的双层告警 / 降级路径。
4. **主线：实时语音识别鉴权有效期优化 (P1)**:
    - **会话稳定性提升**: 将实时语音识别鉴权有效期优化到 10 分钟级别，降低长会话中频繁失效导致的中断与重连成本。
    - **续期与时钟偏差治理**: 已同步评估签名策略、续期方式、前后端时钟偏差与过期后的恢复路径。
    - **回归验证**: 已补齐长会话、续期失败、过期恢复与不同环境下的鉴权一致性验证。
5. **主线：文章详情相关文章推荐与内容发现增强 (P1)**:
    - **推荐策略落地**: 在文章详情页按分类、标签或 translation cluster 相似度提供相关文章推荐，并避免同一翻译簇重复占位。
    - **展示克制**: 推荐模块已以不打断正文阅读的轻量形态落地，不削弱正文主路径可读性。
    - **多语言与回归**: 已补齐多语言过滤、去重与推荐规则的最小回归测试，避免推荐结果跨语言漂移。

4. **主线：阶段级回归任务执行与收口证据链复盘 (P0)**:

**ROI 评估**: `caomei-auth` 第三方登录支持评估与接入预研 `1.33`；路线图 / Todo 深度归档治理 `1.40`；国际化运行时加载与文案复用治理 `1.60`；ESLint / 类型债治理（`composables` 子桶）`1.50`；测试覆盖率与有效性治理 `1.83`；商业化转型可行性重评 `1.20`。其中测试覆盖率与国际化运行时治理为 P0 主线；文档深度归档与商业化重评虽然不属于最高 ROI，但均已满足“可验收、边界清晰、不会与当前主线重复”的正式上收条件。

1. **主线：`caomei-auth` 第三方登录支持评估与接入预研 (P1)**:
    - **执行范围**: 基于 `better-auth` 当前 `genericOAuth({ config: [] })` 的空配置入口，核对 `caomei-auth` 的 Discovery、授权、令牌、用户信息、JWKS、动态客户端注册与刷新令牌能力，明确字段映射、同邮箱合并、账号绑定、回调地址与部署期 ENV 锁定边界，并给出“允许进入实现 / 需上游补齐 / 暂缓接入”的三选一结论。
    - **非目标**: 不在本阶段直接落地真实登录按钮、回调处理或账号绑定实现，不重写现有 GitHub / Google OAuth 接入，也不在未确认 PKCE / revocation / 稳定 claims 前提前承诺真实接入。

    - **验证与证据**: 至少完成本地 `genericOAuth` 接入面核对、`caomei-auth` OAuth / OIDC 文档核对与阻塞项清单沉淀，证据统一回链到专项预研文档。

2. **主线：路线图 / Todo 深度归档治理 (P1)**:
    - **执行范围**: 对 `roadmap.md` 与 `todo-archive.md` 执行首轮深度归档，优先剥离更早阶段正文，只在主文档保留近线窗口、摘要与索引，并同步补齐 `docs/plan/archive/index.md` 的区间分片规则、兼容入口与回链说明。
    - **非目标**: 不改写既有阶段结论，不把所有 `docs/plan/` 文档都重做一次，也不把本轮扩写成多语翻译同步工程。
    - **验收标准**: `roadmap.md` 与 `todo-archive.md` 至少完成一轮可追溯的正文分片与索引回链；若未能完全回到健康窗口，必须明确记录剩余阻塞原因与下一步拆分点。
    - **验证与证据**: 量化归档前后主文档行数、索引与兼容链接有效性，并通过 Markdown 文档质量门。

    - **非目标**: 不发起全仓 i18n 大迁移，不把 `unused` 字段清理升级为本轮 blocker，也不把所有相同中文文本都强制提取成共享 key。

    - **验证与证据**: 至少补跑 `i18n:audit:missing`、`i18n:verify:runtime` 与必要的 parity / 运行时场景验证，并把结果回写到活动回归窗口。

    - **非目标**: 不并行开启第二条全仓 ESLint 规则治理，不把 `any`、`unsafe-*`、`max-lines` 等其他长期主线打包进同一项，也不为了清零规则牺牲可读性。
    - **验收标准**: 已冻结本轮 `composables/` 子桶命中清单，目标子桶中的非空断言已替换为显式守卫、默认值、类型收窄或提前返回，且规则收紧未外溢到非目标目录。
    - **验证与证据**: 至少完成目标子桶 ESLint 校验与窄范围类型检查，并把本轮切片与回滚边界沉淀到专项治理文档。

5. **主线：测试覆盖率与有效性治理 (P0)**:
    - **执行范围**: 以高风险运行时链路为优先，围绕认证配置边界、locale 归一化 / i18n 运行时加载，以及共享组件文案命中补齐高价值测试，并强调失败断言的红绿有效性。
    - **非目标**: 不把本轮写成全仓 coverage 冲 `80%` 的铺量工程，不接受只有 snapshot 的低价值补测，也不以“测试成本高”为由跳过高风险链路的失败用例。
    - **验收标准**: 全仓 coverage 基线不低于当前 `76%+` 水位；至少补齐一组“命名空间漂移 / raw key 暴露 / 认证配置退化时会失败”的行为断言，并说明新增覆盖与未覆盖边界。
    - **验证与证据**: 至少补跑 coverage、i18n 运行时回归与目标测试文件，并把新增失败用例覆盖的真实风险回写到活动回归窗口。

6. **主线：商业化转型可行性重评 (P1)**:

    - **非目标**: 不直接实现支付、价格页、会员中心或营销后台增强，不把评估再次扩写成广告联盟、SaaS 托管或咨询交付等多条并行研究，也不在没有主卖点结论时提前开启商业化实现阶段。
    - **验收标准**: 已按统一评分维度完成明确打分；已明确一个最值得继续验证的付费增强能力，或明确说明当前尚不存在该主卖点；已指出统一承接入口的优先落点，并把下一阶段是否允许继续上收商业化实现写成清晰条件。
    - **验证与证据**: 以专项框架文档、当前站点入口核对结果与规划同步状态作为最小证据链，输出可被后续规划直接复用的决策结论。

### 第三十二阶段：多语言内容资产化承接入口与高风险治理推进 (Multilingual Content Assetization Intake & High-Risk Governance Advancement) (执行中，4/5 主线 + 1 派生切片已闭合)

**时间表**: 2026-05-01 ~ 2026-05-21（预计三周）
**目标**: 在第三十一阶段完成商业化转型重评与多条治理主线收口后，把“多语言内容资产化增强包”的统一承接入口正式上收为下一阶段唯一新增能力，同时继续推进测试覆盖率与有效性治理、重复代码与纯函数复用、ESLint / 类型债治理，以及 Postgres 查询 / CPU / 连接生命周期平衡治理四条高风险优化主线，保持“1 个新功能 + 4 个优化”的受控组合。

**准入结论**: 五条正式主线均已具备明确执行范围、非目标、验收标准与最小验证矩阵，满足规划规范对下一阶段正式上收的要求。新增功能只保留“统一承接入口”一项，并明确先完成独立说明 / 申请页与真实 CTA，不进入支付、会员中心或营销后台实现；测试覆盖率把 `80%+` 降为冲刺目标而非关闭线；重复代码、ESLint 与 Postgres 三条优化继续保持窄切片，避免重新膨胀为全仓治理。`存量代码注释治理与注释漂移收敛` 仅保留为备用项，继续留在 backlog，只有当前五项容量允许时才可额外补入单组切片。

**执行约束**: 本阶段仍遵守“先范围、后实现；先设计、后扩面”的口径。统一承接入口因涉及新页面入口、跨入口 CTA 与商业化承接边界，进入实现前必须先补齐 `docs/design/governance/` 下的专项设计文档；覆盖率治理不得退化为单纯追求 `80%` 数值的铺量工程；重复代码与 ESLint 治理必须保持单切片策略；Postgres 治理只能选择一组请求入口或一组公开热点读链路，不扩写为全站性能重构。

**ROI 评估**: 多语言内容资产化增强包的统一承接入口 `1.60`；测试覆盖率与有效性治理 `1.67`；重复代码与纯函数复用 `1.60`；ESLint / 类型债治理 `1.50`；Postgres 查询、CPU 与连接生命周期平衡治理 `1.57`。其中统一承接入口与测试覆盖率治理为本轮优先上收项；重复代码、ESLint 与 Postgres 三条优化均属于 backlog 长期主线的受控切片延续。

1. **主线：多语言内容资产化增强包的统一承接入口 (P1)**:
    - **执行范围**: 以独立说明 / 申请页作为唯一事实源，收敛“多语言内容资产化增强包”的主卖点、免费核心与付费增强边界，并从 GitHub README、Demo Banner、About 页或等价公开入口接入至少一条真实 CTA，形成“入口 -> 承接页 -> 申请 / 试用 / 候补名单”的最小闭环。
    - **非目标**: 不直接实现支付、价格页、会员中心、营销自动化后台，不把本轮扩写为完整销售站点改版，也不把现有免费能力简单加锁伪装成付费增强包。
    - **验收标准**: 已补齐专项设计文档，明确首版只做独立说明 / 申请页与真实 CTA；已形成单一主卖点文案，并明确免费核心与付费增强边界；至少有一条公开入口完成导流接入，且页面包含可验证的申请 / 试用 / 候补名单动作。
    - **验证与证据**: 至少完成受影响页面与 CTA 跳转的定向测试或浏览器验证，并将专项设计文档、页面入口与转化动作说明回链到阶段回归记录。

2. **主线：测试覆盖率与有效性治理 (P0)**:
    - **执行范围**: 在当前 `76.03% / 76.08%` 基线上继续提升覆盖率，并优先锁定统一承接入口、公开页 runtime、认证配置退化、共享组件 raw key 暴露，以及公开热点读链路等高风险行为断言。
    - **非目标**: 不把本轮写成全仓 coverage 冲 `80%+` 的铺量工程；`80%+` 仅作为冲刺目标，不作为阶段关闭线；不接受只有 snapshot、缺少失败断言或与高风险链路无关的低价值补测。
    - **验收标准**: 全仓 coverage 基线继续提升且不低于当前水位；至少补齐一组与新承接入口或高风险运行时链路直接相关的失败断言，并明确新增覆盖命中的真实风险、未覆盖边界与 `80%+` 冲刺仍缺的关键模块。
    - **验证与证据**: 至少补跑定向 Vitest、`pnpm i18n:verify:runtime`、`pnpm test:coverage` 与 `pnpm exec nuxt typecheck`，并把 coverage 增量与未覆盖边界回写到活动回归窗口。

3. **主线：重复代码与纯函数复用收敛 (P1)**:
    - **执行范围**: 只处理公共页模板片段与列表型查询 helper 两组高收益重复区，优先收敛 CTA 卡片 / 公共说明块、查询参数归一化、列表读模型组装与分页 helper 的重复实现。

    - **验证与证据**: 至少完成受影响公共页与查询 helper 的定向测试 / 类型验证，并在阶段回归记录中补充重复代码基线与剩余热点清单。

4. **主线：ESLint / 类型债治理 (P1)**:
    - **执行范围**: 本轮只允许继续上收单规则窄切片，优先在单文件或双文件范围内推进 `@typescript-eslint/no-explicit-any` 或 `@typescript-eslint/no-non-null-assertion` 的下一组高 ROI 命中点，并同步处理受影响文件的类型债。
    - **非目标**: 不开启第二条并行规则治理，不扩写到 `no-unsafe-*`、全仓 `any` 清零或大规模目录级样式迁移，也不借机清理与本轮目标无关的 warning。
    - **验收标准**: 在进入实现前已明确候选规则、命中清单、影响文件、预期收益、回滚方式与最小验证矩阵；目标切片清理完成后，未外溢到非目标目录，且目录级或定向 ESLint / typecheck 可以稳定通过。
    - **验证与证据**: 至少完成目标文件或目标目录的 ESLint 校验、窄范围类型检查与必要单测，并把残余债务与下一轮候选切片记录到回归窗口或专项治理文档。

5. **主线：Postgres 查询、CPU 与连接生命周期平衡治理 (P0)**:
    - **执行范围**: 本轮只从“一组请求入口”或“一组公开热点读链路”中二选一推进，优先候选为 `0b-db-ready` / 安装态检查 / 匿名鉴权等请求级入口，或 `posts / archive / categories / tags / settings / friend-links` 中的一组公开热点读链路。
    - **非目标**: 不把本轮扩写为全站性能重构，不同时并行推进请求入口与热点读链路两大方向，也不把 CPU / 连接问题泛化为所有数据库访问都要重写。
    - **验收标准**: 已明确当前切片属于哪一组入口或哪一组热点读链路，并给出最小字段集、短 TTL、请求去重或数据库唤醒边界中的至少一组收敛方案；阶段结论需能说明查询次数、结果集体量或连接活跃窗口存在可追溯下降趋势。
    - **验证与证据**: 至少补齐受影响请求入口或热点读链路的定向测试，并通过 `pg_stat_statements`、运行期样本或等价 live sample 记录本轮量化对比结果。
