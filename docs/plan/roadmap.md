### 第二十五阶段：部署体验与可持续演进收敛 (Deployment Experience & Sustainable Evolution Consolidation) (已审计归档)

**时间表**: 约 1 - 1.5 个月
**目标**: 在第二十四阶段完成质量守线与回归执行深化之后，基于 backlog 的方向分析，把“功能已接近完备后应优先打磨什么”正式收敛为下一阶段执行面，围绕部署与初始化体验、编辑器与正文渲染第二轮收口、构建速度与包体性能深化、Cloudflare 平台边界研究，以及 AI 启用式问答 / 配置助手评估五条主线，优先提升新部署者体验、创作稳定性与工程可持续演进能力，而不是继续扩写新的大功能面。

**审计结论**: 第二十五阶段已围绕部署与初始化体验、编辑器与正文渲染第二轮收口、构建速度与包体性能深化、Cloudflare 平台边界研究，以及 AI 启用式问答 / 配置助手评估五条主线完成收口。当前阶段的实现代码、指南 / 设计文档、多语言翻译页与专项 Review Gate 证据均已落盘，正式放行依据为 [2026-04-11-phase-25-archive.md](../../artifacts/review-gate/2026-04-11-phase-25-archive.md) 的 `Pass` 结论。

**ROI 评估**: 部署体验与初始化收敛 2.00；编辑器与正文渲染体验第二轮收口 1.80；构建速度与包体性能深化 1.60；AI 初始化 / 配置问答助手评估 1.33；Cloudflare 运行时兼容研究与止损结论 1.14。五项均满足进入下一阶段候选的准入条件，其中部署体验与编辑器 / 渲染收口为 P0 主线。

1. **主线：部署体验与初始化收敛 (P0)**:
    - **最小可运行路径**: 继续收敛“首次部署到可成功启动”的最小链路，减少环境变量理解成本、启动前置心智负担与排障入口分散问题。
    - **前置体检与可观测性**: 优先评估部署前体检、配置缺失提示、环境变量分层说明与初始化阶段的错误反馈收敛，不再让新部署者主要依赖手动查文档排障。
    - **事实源统一**: 相关优化必须复用现有安装向导、设置元信息、README、部署指南与变量文档，不得引入第二套配置事实源。
2. **主线：编辑器与正文渲染体验第二轮收口 (P0)**:
    - **兼容性与稳定性**: 在继续沿用 `mavon-editor` 异步包装接入的前提下，收敛当前编辑器链路的兼容性、交互稳定性与核心创作路径体验。
    - **预览与渲染一致性**: 继续优化正文渲染、翻译编辑、Markdown 预览与最终展示之间的一致性，降低“编辑时看到的内容”与“实际渲染结果”之间的偏差。
    - **范围受控**: 本阶段不直接启动编辑器整体替换工程，优先完成既有链路的第二轮体验打磨与问题收口。
3. **主线：构建速度与包体性能深化 (P1)**:
    - **热点优先**: 基于现有 bundle budget 与性能预算体系，锁定 1 - 2 组高收益热点继续收敛构建耗时、首屏依赖与大块 vendor 负担。
    - **守线转深化**: 不把已有预算转绿视为“问题已解决”，而是继续推进构建速度、包体体积与高频页面依赖的结构性优化。
    - **证据留痕**: 继续通过 `test:perf:budget`、strict 预算或等价证据记录本轮优化收益与未覆盖边界。
4. **主线：Cloudflare 运行时兼容研究与止损结论 (P1)**:
    - **研究优先**: 明确回答当前项目是否值得继续投入 Cloudflare Pages / Workers 运行时适配，而不是直接承诺整站可部署支持。
    - **阻塞清单**: 输出 TypeORM、Node 运行时依赖、数据库与存储契约等关键阻塞点，以及最小样机路径与替代技术栈代价。
    - **结论导向**: 本阶段允许得出“继续推进 / 暂停投入 / 仅保留外围能力接入”任一结论，但必须有可追溯的证据链支持。
5. **主线：AI 初始化 / 配置问答助手评估 (P1)**:
    - **启用式助手边界**: 评估是否为新部署者提供 AI 初始化或配置问答助手，优先解决“看不懂配置项、分不清必填项、不会排障”的启用门槛问题。
    - **安全与权限约束**: 先收敛可回答范围、可写配置范围、ENV 锁定字段、越权保护与失败回退策略，不直接承诺实现全功能站内 AI 助手。
    - **工程复用**: 若进入实现阶段，必须优先复用现有安装向导、设置元信息、`qa-assistant` 与文档知识源，而不是额外维护第二套知识或配置体系。

### 第二十六阶段：质量治理与数据库流量收敛 (Quality Governance & Database Traffic Consolidation) (已审计归档)

**时间表**: 约 1 - 1.5 个月
**目标**: 在第二十五阶段完成部署体验与可持续演进收口之后，把 backlog 中已经明确成形、且与当前“打磨既有能力而非扩写新系统”方向最契合的五条治理切片正式上收为下一阶段执行面，围绕“测试覆盖率继续推进到约 72%、ESLint 规则继续分批收紧、存量代码注释治理首轮落地、重复代码与纯函数复用继续收敛，以及 PostgreSQL 查询与数据库出网流量治理”形成新的阶段组合，优先降低维护成本、数据库压力与后续演进风险。

**准入结论**: 五条主线均来自 [backlog.md](./backlog.md) 的长期主线或已评估候选，符合当前阶段“核心任务控制在 5 项以内”的容量约束。其中 PostgreSQL 专项的主目标明确为“优先降低数据库出网流量与查询压力”，CPU 使用时机仅作为次级观测指标，不把本阶段扩写为全栈性能重构。

**审计结论**: 第二十六阶段围绕测试覆盖率推进、ESLint / 类型债第二轮收紧、存量代码注释治理、重复代码与纯函数复用收敛，以及 PostgreSQL 查询与数据库出网流量治理五条主线，已在实现代码、定向测试、回归记录、运行期监控与规划文档中完成闭环，满足归档条件。全仓覆盖率已达到阶段目标（约 `72%`），且后台监控显示 PostgreSQL 查询与连接压力已显著缓解。

**阶段收口**: 第二十七阶段的候选内容已在代码、测试、回归记录与规划文档中确认闭环，现已进入归档收束与翻译同步。

**ROI 评估**: 测试覆盖率与红绿测试有效性推进 2.00；ESLint / 类型债第二轮收紧 1.50；存量代码注释治理首轮落地 1.33；重复代码与纯函数复用继续收敛 1.40；PostgreSQL 查询与数据库出网流量治理 1.80。五项均满足当前阶段主线准入条件，其中测试治理与 PostgreSQL 流量治理为 P0 主线。

1. **主线：测试覆盖率与红绿测试有效性推进 (P0)**:
    - **覆盖率目标**: 在当前 `68.85%` 基线上继续提升约 `4%`，阶段目标先达到约 `72%`，并明确记录本轮新增覆盖率、模块分布与未覆盖边界。
    - **高风险路径优先**: 优先补齐 `server/services/ai/text.ts`、公开查询热点 API 与数据库治理相关服务层的失败路径、边界断言与回退逻辑，而不是平均铺开式补量。
    - **结果留痕**: 每轮补强都要保留失败用例、转绿结果、仍未覆盖范围与下一轮优先顺序，继续服务后续阶段切片。
2. **主线：ESLint / 类型债第二轮收紧 (P1)**:
    - **分批推进**: 至少再正式收紧 `1 - 2` 条高 ROI 规则，优先选择命中范围有限、回滚边界清晰、对生产代码收益明确的规则族。
    - **验证与回退**: 每批规则提升都要补齐命中清单、回滚方式与最小验证矩阵，避免一次性强收紧演变为失控返工。
    - **边界隔离**: 测试、脚本、迁移和历史遗留目录继续保持受控豁免，不借此扩写成全仓 `any` 清零工程。
3. **主线：存量代码注释治理首轮落地 (P1)**:
    - **高复杂度链路优先**: 从设置读取 / 来源判定、locale 归一化、鉴权上下文挂载、上传存储解析、文章访问控制、AI 文本 / 配额治理，以及公开查询裁剪逻辑中选择 `1 - 2` 组模块优先落地。
    - **注释价值约束**: 注释应解释约束、契约、边界、副作用与回退原因，不做逐行复述代码式补注释。
    - **漂移检查闭环**: 每轮切片都要记录已补注释类型、未覆盖边界与注释漂移检查结论，并结合受影响模块的最小验证结果一起留痕。
4. **主线：重复代码与纯函数复用继续收敛 (P1)**:
    - **热点优先**: 继续围绕前台 legal pages、categories / tags 公共页、列表型查询 helper、查询参数处理与列表读模型组装等热点重复区域做收敛。
    - **最小切片**: 本阶段至少完成 `1` 组公共页模板或查询 helper / 纯函数抽象，不把长期复用治理扩写成跨目录大重构。
    - **证据对比**: 输出本轮重复代码基线、抽象收益与仍需观察的模块，并复核共享抽象是否引入过度泛化风险。
5. **主线：PostgreSQL 查询与数据库出网流量治理 (P0)**:
    - **热点读路径优先**: 聚焦 `posts / archive / categories / tags` 等公开高频读接口、设置读取复用边界与页面装配链路，优先压缩宽结果集与重复查库带来的数据库出网流量。
    - **观测口径先行**: 建立 `pg_stat_statements` 或等价长窗口样本、热点 SQL 分组、查询指纹或结果集大小估算，先回答“哪些请求最耗流量”，再决定最小字段集、短 TTL 缓存与查询裁剪策略。
    - **目标受控**: 本阶段主目标是降低数据库出网流量与查询压力；CPU 使用时机仅作为次级观察信号，不把当前治理面扩写为全链路性能重构。

**审计结论**: 本阶段五条主线已在代码、测试、回归记录、性能基线与规划文档中完成闭环，满足归档条件。RSS 路由错误修复、WechatSync 微博 / 小红书回归、分享入口与图标系统、性能基线与 E2E 首轮矩阵均已收口。

    - **执行范围**: 聚焦关键页面性能瓶颈并建立统一采样口径，优先完成第一阶段目标。
    - **验收标准**: 核心页面性能评分稳定达到 >= 50，沉淀措施与副作用记录。

### 第二十八阶段：内容运营洞察与运行时治理推进 (Content Insights & Runtime Governance Advancement) (已审计归档)

**时间表**: 2026-04-18 ~ 2026-05-09（预计三周）
**目标**: 在第二十七阶段完成渠道稳定性、性能基线与 E2E 首轮矩阵收口后，正式开启“1 个新功能 + 4 个优化”的下一阶段执行面：以后台内容统计与热点分析看板作为唯一新增能力，同时把测试覆盖率提升到 `76%+`、把 Postgres 主线升级为 CPU / 连接生命周期 / 查询体量平衡治理，并继续推进国际化运行时加载与编辑器 Markdown / 外观一致性两条体验型优化。

**准入结论**: 五条主线已完成候选分析，且总数控制在 5 项内，满足当前阶段容量约束。新增功能仅保留“后台内容统计与热点分析看板”一条，避免与长期治理主线叠加后继续膨胀执行面。

**审计结论**: 第二十八阶段五条主线已在实现代码、定向测试、活动回归窗口与规划文档中完成闭环，满足归档条件。后台内容洞察首页已经落地，Postgres 平衡治理已完成数据库预热边界、匿名公开链路会话解析与公开热点读缓存收口；结合最新全仓 coverage 收口结果与后台运行期观测，coverage `> 76%` 与 Postgres 两条 P0 主线均可关闭。

**ROI 评估**: 后台内容统计与热点分析看板 1.60；Postgres 查询、CPU 与连接生命周期平衡治理 1.75；测试覆盖率与有效性治理（`> 76%`）1.83；国际化运行时加载与文案复用治理 1.60；编辑器 Markdown 与外观一致性增强 1.40。五项均满足当前阶段主线准入条件，其中 Postgres 平衡治理与覆盖率提升为 P0 主线。

1. **主线：后台内容统计与热点分析看板 (P1)**:
    - **执行范围**: 在后台管理侧提供内容运营概览卡片、趋势图与热门文章 / 标签 / 分类排行，并补齐时间范围、语言与公开状态筛选。
    - **验收标准**: 第一轮必须先明确时间聚合、翻译簇去重、时区、空数据与草稿过滤口径，避免演变为通用 BI 系统。
