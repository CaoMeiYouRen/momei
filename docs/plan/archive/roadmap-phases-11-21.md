### 第十一阶段：迁移治理与自动化交付深化 (Migration Governance & Automation Delivery Deepening) (已审计归档)

**时间表**: 约 1.5 - 2 个月
**目标**: 围绕迁移可控性、版本可追溯性、协议治理、第三方分发解耦与 CLI / MCP 自动化扩展，补齐系统从“站内能力完整”走向“跨工具协作、可迁移、可追溯、可合规治理”的交付闭环。

**ROI 评估**: 线性版本追踪与回滚 1.67；迁移链接治理与云端资源重写 1.80；第三方分发解耦与投递控制 1.60；CLI / MCP 自动化能力扩展 1.67；协议版本治理与合规展示 1.80。五项均满足下一阶段准入条件。

**审计结论**: 阶段核心能力已在代码、CLI / MCP 工具面与管理端完成闭环；原待办中唯一仍显示未完成的“第三方分发解耦与投递控制”已由统一分发状态模型、手动同步入口、时间线审计与定向测试完成落地。本阶段不再保留专属遗留项，跨阶段积压统一回收到 [backlog.md](./backlog.md)。

1. **线性版本追踪与内容回滚 (Linear Version Tracking & Rollback) (P0)**:
    - **Git Commit 化追踪**: 已为文章变更建立线性版本模型，沉淀可读的变更历史、提交摘要、作者上下文与恢复来源。
    - **差异查看与比较体验**: 已提供版本历史列表、相邻版本与指定版本的差异查看能力，并统一前后端 diff 口径。
    - **简化回滚模型**: 已支持版本恢复、回滚副作用边界与幂等保护，且不引入分支管理，保持单主线更新流程。
2. **迁移链接治理与云端资源重写 (Migration Link Governance & Cloud Asset Rewrite) (P0)**:
    - **资源链接重写**: 已为历史文章中的绝对资源 URL 提供 dry-run、域名过滤、批量重写、审计报告与后台查看入口。
    - **旧站链接迁移**: 已在迁移 CLI 与治理服务中建立“旧 URL -> 新 URL”的映射能力，覆盖文章、分类、标签、归档与独立页面。
    - **迁移校验与承接**: 已补齐 redirect seeds / alias 承接、静态校验、站内校验、失败重试与可选在线检测策略。
3. **第三方分发解耦与投递控制 (Content Distribution Decoupling) (P1)**:
    - **手动同步与重试**: 已将文章发布与 Memos / WechatSync 分发解耦，支持编辑页分发面板中的手动同步、重试、人工终止与幂等状态追踪。
    - **状态边界清晰化**: 已明确“首次同步 / 更新原内容 / 重推新内容 / 失败重试”的状态边界，并要求用户在更新与重推之间显式决策。
    - **投递审计补完**: 已统一第三方分发链路的日志、时间线、失败分类与结果回写，避免发布副作用与外部分发策略耦合过深。
4. **CLI / MCP 自动化能力扩展 (CLI / MCP Automation Surface Expansion) (P1)**:
    - **全文翻译工作流外部化**: 已为 CLI 与 MCP 落地全文翻译能力，支持源语言显式指定、预览确认、目标文章回填与翻译簇绑定。
    - **媒体生成与文章回填**: 已支持通过 CLI / MCP 调用封面图生成、TTS / 播客音频生成，并在任务完成后自动回填文章字段。
    - **分类 / 标签自动生成与翻译**: 已支持基于正文或摘要生成分类、标签与多语言建议，并与站内 taxonomy / translationId 口径保持一致。
    - **其他自动化工作流**: 已补齐任务查询、发布编排与关键长任务回填测试闭环，为后续批处理能力扩展预留统一契约。
5. **协议版本治理与合规展示 (Agreement Governance & Compliance Delivery) (P1)**:
    - **生效语义闭环**: 已重构协议“主语言 / 权威版本 / 当前生效版本 / 参考译本”的语义边界，统一后台激活、设置记录与公开接口的读取口径。
    - **多语言协议治理**: 已区分具有法律效力的权威版本与仅供参考的翻译版本，并补齐生效日期、最近更新日期、版本说明、最近版本列表与历史查看能力。
    - **后台约束与公开展示补完**: 已收敛已生效、已被用户同意、来自环境变量等场景下的编辑限制，并同步优化后台管理页与公开协议页的状态展示与操作引导。


### 第十二阶段：内容编排与创作输入体验收敛 (Content Orchestration & Authoring Input Consolidation) (已审计归档)

**时间表**: 约 1 - 1.5 个月
**目标**: 围绕首页内容编排、语音输入复用、编辑器稳定性与站点品牌收敛，补齐“运营可控性 + 创作输入效率 + 展示语义一致性”的产品闭环，避免继续在多个页面复制相近能力或混用设置语义。

**ROI 评估**: 首页内容编排与置顶 1.75；通用文本域语音输入 1.67；编辑器兼容性收敛与渐进替换预案 1.60；页脚品牌与版权配置解耦 1.60。四项核心任务均满足本阶段准入条件；新增的“多语言创作与第三方分发链路热修复”属于已交付能力回归，按插队规则以 Hotfix 形式并入当前阶段。

**审计结论**: 第十二阶段核心能力已在代码、测试与配置链路中闭环：首页已形成“最新文章 + 热门文章”双区块与置顶排序体系，通用文本域语音输入已沉淀为可复用 composable / 组件，页脚版权已完成站点配置解耦，翻译字段、Memos permalink、WechatSync 投递与默认语言选项同步等 Hotfix 也已落地。编辑器路线明确维持现有 `mavon-editor` 异步包装接入，并将整体替换继续留在“功能矩阵验证后再推进”的闸门之下。本阶段不再保留专属遗留项。

1. **首页内容编排与文章置顶机制 (Homepage Orchestration & Post Pinning) (P0)**:
    - **双区块首页已落地**: 首页已建立“最新文章 + 热门文章”双区块，并以公开查询参数完成跨区块去重与热门回填。
    - **置顶语义已统一**: 文章模型、后台管理入口、公开查询与文章卡片展示已统一消费 `isPinned` 语义。
    - **排序回归已补齐**: 首页页级测试与 posts API 排序测试已覆盖置顶优先、热门排序与回退场景。
2. **通用文本域语音输入能力 (Reusable Textarea Voice Input) (P1)**:
    - **能力抽象已完成**: 已沉淀 `useVoiceInput`、`AppVoiceInputTrigger`、`AppVoiceInputOverlay` 三层复用结构，统一 Web Speech 与云端 ASR 模式切换。
    - **多入口复用已建立**: 文章编辑器与其他多行文本输入场景已不再复制页面级语音状态机。
    - **核心测试已覆盖**: composable 定向测试已覆盖模式切换、转录重置与基础录音流程。
3. **编辑器兼容性收敛与渐进替换预案 (Editor Compatibility Hardening & Incremental Replacement Plan) (P1)**:
    - **兼容性结论已收敛**: 当前继续使用 `mavon-editor` 的异步客户端包装层，避免 Nuxt 4 SSR 与样式注入问题扩大化。
    - **关键功能基线已保留**: 图片上传回填、双栏预览、全屏与现有交互能力维持稳定，不在本阶段直接整体替换编辑器。
    - **替换闸门已明确**: 后续若推进替换，必须先建立无功能回退的适配层与灰度验证基线。
4. **页脚品牌与版权配置解耦 (Footer Branding & Copyright Configuration) (P1)**:
    - **独立设置键已打通**: `site_copyright_owner` / `site_copyright_start_year` 已覆盖安装向导、后台设置、公开接口与前台页脚渲染。
    - **兼容迁移已补齐**: 旧键名与旧环境变量别名仍可读取，避免站点升级时页脚版权信息丢失。
    - **职责边界已验证**: 页脚版权展示与文章默认版权协议已通过公开设置测试完成分离。
5. **多语言创作与第三方分发链路热修复 (Multilingual Authoring & Distribution Hotfix) (Hotfix)**:
    - **翻译字段已补漏**: 后台 AI 翻译类型已补齐 `translate_name_batch` 多语言词条。
    - **Memos permalink 已修复**: 远端链接回写已统一为真实公开 permalink，并处理实例 URL 中 `/api/v1` 的归一化。
    - **WechatSync 状态链路已恢复**: 结果回写、失败分类、人工终止与默认语言选项同步均已补齐实现与测试。

### 第十三阶段：多语言创作资产与渠道分发深化 (Multilingual Creative Asset & Channel Distribution Deepening) (已审计归档)

**时间表**: 约 1 - 1.5 个月
**目标**: 基于第十二阶段已收口的创作输入与分发基础，继续补齐“多语言正文 -> 多语言资产 -> 多渠道分发 -> 文章详情阅读链路”的一致性，减少译文资产复用失真、渠道格式漂移、外部分发链路碎片化以及文章详情页阅读承接断层。

**ROI 评估**: 多语言封面与音频资产一致性 1.80；渠道分发内容模板与标签适配 1.67；Newsletter 外部分发通道扩展 1.60；长文本翻译渐进式回填体验 1.75；文章详情阅读链路与 SEO 品牌强化 1.67。五项核心任务均满足下一阶段准入条件。

**审计结论**: 第十三阶段核心交付与两项 Hotfix 已在代码、测试与文档中完成闭环，满足归档条件；原待办中唯一未交付的“相关文章推荐”已按可选增强重新评估并移入 [backlog.md](./backlog.md)，不再作为本阶段阻塞项。

1. **多语言封面与音频资产一致性 (P0)**:
    - **多语言封面生成**: 翻译文章生成封面时支持以目标语言标题、摘要或提示词为输入，不再默认复用源语言封面。
    - **多语言 TTS / Podcast 资产绑定**: 支持按 locale 生成独立音频资产，并明确与 `translationId`、文章语言及回填字段的绑定关系。
    - **回退与测试策略**: 补齐封面 / 音频缺失时的展示回退与回填测试，避免不同语言版本共享错误媒体。
2. **渠道分发内容模板与标签适配 (P0)**:
    - **渠道导出模板分层**: 区分正文、摘要、封面、标签与版权尾注在 Memos、WechatSync、B 站专栏等渠道的导出格式，不再沿用单一 Markdown / HTML 模板。
    - **标签清洗规则统一**: 为 `#标签` 导出建立清洗、去重、长度限制与非法字符处理规则，避免平台侧拒收或展示错乱。
    - **分发预览基线**: 为主要渠道建立最小可用的导出预览或结构化调试信息，降低人工试错成本。
3. **Newsletter 外部分发通道扩展 (P1)**:
    - **listmonk 集成补齐**: 补齐文章或 Campaign 到 listmonk 的分发集成，明确受众映射、幂等更新与失败回写口径。
    - **配置链路打通**: 同步补齐 settings / ENV / 文档中的 listmonk 配置说明，避免接入链路只停留在代码层。
    - **审计与回归**: 补齐外部分发结果追踪与最小回归验证，避免新增渠道破坏既有营销推送口径。
4. **长文本翻译渐进式回填体验 (P1)**:
    - **流式 / 分段双模式回填**: 文章编辑页长文本翻译优先复用现有 `/api/ai/translate.stream` 能力，并按文本长度在“字段内实时流式回填”和“分段逐块回填”之间自动切换。
    - **字段级进度与中断恢复**: 对标题、摘要、正文分别展示阶段进度、已回填内容、取消 / 重试入口，避免长文翻译失败时整段结果丢失。
    - **契约与测试补齐**: 补齐 SSE / chunk 解析、块合并、失败恢复与大文本阈值切换测试，确保编辑器不会因流式写回导致内容错位或性能抖动。
5. **文章详情阅读链路与 SEO 品牌强化 (P1)**:
    - **上一篇 / 下一篇导航**: 在文章详情页增加“上一篇 / 下一篇”导航，默认仅基于“同语言、已发布、可公开访问”文章集合，并按 `publishedAt` 计算邻接关系。
    - **标题页 SEO 与品牌模板统一**: 统一文章详情页默认 SEO 模板，默认项目品牌使用“墨梅博客”，并强化“AI 驱动 / AI 博客”语义，同时保留站点后台 `siteTitle` 自定义能力。

> 按迭代内新增事项闸门，本阶段原则上仅补入修复型 Hotfix，且不计入上方 5 项核心任务；若发现阻塞当前交付的明确回归或部署基线缺陷，可在产品评估后继续以修复型 Hotfix 插入，但不得借机扩张为新功能。

6. **创作资产导入导出保真 (Hotfix)**:
    - **音频元数据保留**: 导入 / 导出文章时保留 `metadata.audio`、`metadata.tts` 的核心字段，避免已生成的 TTS / Podcast 资产在迁移、备份或重导入后丢失绑定语义。
    - **双向兼容**: 兼容既有 Markdown Front-matter、CLI 导入与批量 ZIP 导出，不再将音频元数据继续视为“应丢弃的特有字段”。
    - **回归验证**: 补齐导入 / 导出定向测试，覆盖单语音频、多语言译文音频与仅远程 URL 音频场景。

7. **第三方分发启用开关识别与数据库初始化基线同步 (Hotfix)**:
    - **布尔型开关识别修复**: 修复 `LISTMONK_ENABLED`、`MEMOS_ENABLED` 等第三方分发启用开关在 ENV 或设置层为字符串 `"true"` 时仍被误判为未启用的问题，统一布尔型配置解析口径并补齐回归测试。
    - **多数据库初始化脚本同步**: 更新 `database/sqlite/init.sql`、`database/mysql/init.sql`、`database/postgres/init.sql`，确保与当前实体、索引和默认值保持一致，避免新实例初始化后与运行时 Schema 漂移。
    - **数据库设计文档收口**: 同步更新 `docs/design/database.md`，明确当前事实源、核心实体、初始化脚本适用边界与文档维护约束，避免后续继续出现“代码已变、文档未跟”的部署基线偏差。

### 第十四阶段：多语言配置治理与部署基线纠偏 (Multilingual Configuration Governance & Deployment Baseline Alignment) (已审计归档)

**时间表**: 约 1 - 1.5 个月
**目标**: 围绕管理员可编辑配置的多语言化、合规文本翻译审校、导入路径守卫、Cloudflare 部署声明纠偏与后台配置交互一致性，补齐“内容已多语、配置仍单语、迁移导入缺少守卫、平台能力声明易误读”的剩余短板，让系统从“多语言创作完成”走向“多语言站点治理可控、部署边界清晰、迁移导入更稳健”。

**ROI 评估**: 配置项多语言国际化与回退治理 2.00；AI 翻译扩展到配置与法律文本 1.60；迁移导入 Slug 守卫与路径别名校验 2.00；Cloudflare 部署支持声明纠偏 1.75；后台配置页动作与反馈入口统一 1.75。五项核心任务均满足下一阶段准入条件。

**审计结论**: 第十四阶段五项核心任务已在代码、测试与文档中完成闭环，满足归档条件；多语言配置事实源、AI 法律文本审校链路、导入路径守卫、Cloudflare 部署边界说明与后台配置页交互统一均已收口。

1. **配置项多语言国际化与回退治理 (P0)**:
    - **多语言配置建模**: 为站点标题、关键词、公告、SEO 文案、法律文本简介等管理员可编辑配置建立多语言存储结构，并兼容现有单字符串配置。
    - **回退链统一**: 明确 `zh-TW -> zh-CN -> en-US` 与 `当前 locale -> en-US` 的读取顺序，并统一公开设置、SEO、邮件与通知链路的消费口径。
    - **迁移与回归**: 补齐旧配置兼容迁移、缺词回退与跨语言读取测试，避免配置国际化引入新的展示漂移。
2. **AI 翻译扩展到可生成多语言的配置与法律文本 (P1)**:
    - **可翻译配置扩面**: 将 AI 翻译能力扩展到公告、站点介绍、SEO 文案等配置输入入口，降低管理员逐语种重复录入成本。
    - **法律文本审校闭环**: 为用户协议、隐私政策等合规文本建立“AI 草案 -> 人工审校 -> 激活生效”的流程，不将 AI 输出直接视为可生效的法律版本。
    - **与配置模型联动**: 让 AI 生成结果直接兼容多语言配置模型与 fallback 规则，避免公告、法律页与设置中心形成三套多语口径。
3. **迁移导入 Slug 守卫与路径别名校验 (P1)**:
    - **导入前校验**: 在 CLI / API 导入文章时，对 `slug`、`permalink`、`abbrlink` 执行格式、保留字、冲突与语言维度唯一性校验。
    - **冲突处理策略**: 默认尊重源站合法别名，仅在非法、冲突或语义不完整时进入显式修复、回退或人工确认分支。

    - **接入层统一**: 为 RSS / RSSHub 源建立统一拉取、清洗、缓存与失败降级层，避免页面直接依赖外部返回结构。
    - **挂载模型清晰**: 明确首页、作者页或独立聚合页的挂载位置、排序、去重、来源标识、语言过滤与跳转策略。
    - **稳定性优先**: 补齐抓取频率、缓存时效、可选关闭开关与回退策略，确保外部源异常不会拖垮主站体验。
5. **主线：标签翻译前置化与正文翻译解耦优化 (P1)**:
    - **标签更早反馈**: 将标签翻译前置到正文长流程完成前更早暴露，减少用户必须等待全文收口后才能检查元数据的延迟感。
    - **状态分叉收敛**: 覆盖“标签完成、正文处理中、用户取消、标签失败但正文继续”等状态分叉，保持现有 `translationId` 与 taxonomy 绑定口径不漂移。
    - **入口语义统一**: 对齐后台编辑器、CLI 与 MCP 三类入口的翻译编排语义，避免出现双重事实源。
6. **主线：默认封面生成文案压缩与大字号策略优化 (P1)**:
    - **文案优先级重排**: 建立“封面短句 > 关键词摘要 > 短标题 > 截断原始标题”的稳定选择顺序，避免默认封面继续承载过长原始标题。
    - **版式规则统一**: 统一字号级别、换行与截断策略，确保中英文与长词语言都具备可预测的默认视觉表现。
    - **双链路一致性**: 对齐自动回填封面与候选图待确认链路的默认版式，避免两条工作流继续出现默认视觉漂移。
