### 第二十二阶段：质量有效性与创作编排治理深化 (Quality Effectiveness & Authoring Orchestration Deepening, Archived)

**时间表**: 约 1 - 1.5 个月
**目标**: 在第二十一阶段完成验证入口与创作工作流收口之后，继续把收益最高的“测试有效性、周期性回归执行节奏、批量翻译编排能力、ESLint 规则守线、时间与日期事实源一致性，以及后台高频入口效率”正式上收为当前阶段，避免后续治理再次回到“规范存在但执行不连续、能力存在但编排不可扩展”的状态。

**审计结论**: 第二十二阶段六条主线已在实现代码、测试 / 验证、回归记录、设计文档与规划文档中完成闭环。`todo.md` 已清理、`todo-archive.md` 已收录阶段归档块、[回归活动窗口](../reports/regression/current.md) 已完成滚动归档并回到窗口内，因此本阶段满足归档条件。下一阶段目前仅保留候选分析，不在本轮直接上收为正式 Phase 23。

**ROI 评估**: 测试有效性增强治理 2.00；周期性回归任务实盘化 2.25；文章批量翻译到其他语言的编排能力评估 1.29；ESLint 规则分阶段收紧治理 1.17；时间配置解析与日期事实源收敛 1.75；后台文章管理入口前置到顶栏 2.50。其中特别适合作为本阶段主线起点的是测试有效性、周期性回归实盘化、时间 / 日期事实源收敛与后台高频入口优化；批量翻译编排评估与 ESLint 规则收紧则以受控副线方式纳入。

1. **主线：测试有效性增强治理 (P0)**:
    - **红绿测试优先**: 围绕一组高回归风险模块或链路引入“先失败、后修复、再守线”的节奏，减少只测成功路径的覆盖偏差。
    - **边缘 / 失败路径补强**: 优先覆盖异常输入、失败分支、回退逻辑与边界条件，而不是把测试补强退化为机械堆 coverage。
    - **证据可复盘**: 每轮补强都要在回归记录中保留失败用例、转绿结果、未覆盖边界与下一轮优先范围。
2. **主线：周期性回归任务实盘化 (P0)**:
    - **从规范到节奏**: 将既有周期性回归规范进一步收敛为周级、发版前与阶段收口前的最小执行组合与责任边界。
    - **回归产物统一**: 继续统一沉淀到 [回归活动窗口](../reports/regression/current.md)，并明确何时滚动归档、何时升级为 blocker，而不是新增第二套记录面。
    - **高漂移项优先**: 优先覆盖测试 / coverage、依赖安全、文档同步、脚本入口与性能基线等高漂移治理面。
3. **主线：文章批量翻译到其他语言的编排能力评估 (P1)**:
    - **能力边界先行**: 先明确文章范围选择、目标语言矩阵、预览确认、失败重入、取消与人工重试，不直接把批量翻译设计成黑盒式批处理。
    - **复用现有事实源**: 批量能力必须复用现有 `translationId`、翻译范围、标签 / 分类映射、封面 / 音频默认策略与任务状态模型。
    - **准入判断明确**: 即便本阶段不完整实现，也必须形成成本、并发、超时预算与覆盖策略的准入结论。
4. **主线：ESLint 规则分阶段收紧治理 (P1)**:
    - **基线先行**: 先建立现有规则豁免、warning 债务与高频误报点基线，明确首批要收紧的规则族和目录范围。
    - **分批可回滚**: 每一批规则提升都要附带预期收益、回滚方式和最小验证矩阵，避免一次性强收紧导致返工失控。
    - **范围受控**: 测试、脚本、迁移和历史遗留目录继续保持独立豁免边界。
5. **主线：时间配置解析与日期事实源收敛 (P1)**:
    - **时长配置统一**: 对涉及超时、轮询、缓存 TTL 与等待窗口的毫秒型配置优先收敛到 `ms` 包解析，减少手写换算与字符串 / 数值混用。
    - **日期逻辑优先 `dayjs`**: 业务逻辑中的日期格式化、比较、排序、区间判断继续以 `dayjs` 为首选，模板保持 `useI18nDate`，`new Date()` 只保留在运行时边界。
    - **治理必须落代码**: 本主线至少补齐一组“时间配置解析 + 日期逻辑替换”的定向验证，不允许只停留在规范层。
6. **主线：后台文章管理入口前置到顶栏 (P1)**:
    - **高频入口可达性提升**: 将文章管理入口提升到顶栏或等价高频导航位，降低后台创作流的进入成本。
    - **响应式与权限边界**: 必须同步考虑不同角色、不同视口与当前顶栏信息密度，不得通过简单堆入口破坏现有导航平衡。
    - **最小回归补齐**: 至少验证桌面 / 窄屏下的入口显隐、跳转与角色边界。

### 第二十三阶段：全球触达扩展与运行时稳态治理 (Global Reach Expansion & Runtime Stability Governance) (已审计归档)

**时间表**: 约 1 - 1.5 个月
**目标**: 在第二十二阶段完成质量与编排收口之后，继续把用户可感知收益和运行期稳态价值都较高的五类事项正式收敛为下一阶段候选主线，围绕“多语言触达能力扩面、Vercel 场景下 AI 媒体任务补偿、依赖安全巡检自动化、实时 ASR 会话稳定性提升，以及文章详情页内容发现增强”形成新的阶段组合，避免治理面与体验面长期分离推进。

**审计结论**: 第二十三阶段五条主线已在实现代码、定向测试、统一调度入口、规划文档与多语路线图摘要中完成闭环。`todo.md` 已清理当前阶段正文，`todo-archive.md` 已收录阶段归档块；后续正式上收的第二十四阶段见下文当前规划，不影响本阶段归档结论。

**ROI 评估**: 多语言社交与赞助平台扩展 1.60；AI 媒体生成在 Vercel 场景下的超时巡检与状态补偿 1.75；依赖包风险每日巡检自动化 1.60；实时语音识别鉴权有效期优化 1.50；文章详情相关文章推荐与内容发现增强 1.50。五项均已按本阶段准入边界完成收口。

1. **主线：多语言社交与赞助平台扩展 (P1)**:
    - **平台扩容**: 在现有社交 / 赞助系统上补齐中英文高频平台，并同步提供图标、颜色与展示文案约定。
    - **语言一致性**: 平台名称、标签与编辑选项需覆盖当前已支持语言集合，避免在 `zh-TW`、`ko-KR` 等语种下退回硬编码平台名。
    - **兼容边界**: 已保持作者级 / 全局级回退、locale 过滤，以及二维码 / URL 双模式展示链路稳定。
2. **主线：AI 媒体生成在 Vercel 场景下的超时巡检与状态补偿 (P0)**:
    - **任务补偿闭环**: 为图像生成与播客音频生成任务补齐超时阈值、失联检测、状态回写补偿与最终失败落点。
    - **调度巡检**: 增加定时扫描或等价机制，主动处理长时间无心跳 / 无结果回写的任务，避免前端轮询自然超时后长期卡在中间态。
    - **部署边界**: 已明确区分云函数超时、供应商已完成但站内未同步、轮询链路丢失结果三类失败来源，并补齐幂等验证。
3. **主线：依赖包风险每日巡检自动化 (P0)**:
    - **调度复用**: 复用现有 `security:audit-deps` 与依赖风险检查脚本，增加每天执行一次的自动调度入口。
    - **结果留痕**: 至少产出结构化结果摘要，能够区分“无高危风险”“发现可修复风险”“审计执行失败”三类结论，并保留近期可追溯记录。
    - **最小告警**: 当发现 `high` 及以上级别风险或执行失败时，当前已提供 GitHub issue 与 failed workflow + artifact 的双层告警 / 降级路径。
4. **主线：实时语音识别鉴权有效期优化 (P1)**:
    - **会话稳定性提升**: 将实时语音识别鉴权有效期优化到 10 分钟级别，降低长会话中频繁失效导致的中断与重连成本。
    - **续期与时钟偏差治理**: 已同步评估签名策略、续期方式、前后端时钟偏差与过期后的恢复路径。
    - **回归验证**: 已补齐长会话、续期失败、过期恢复与不同环境下的鉴权一致性验证。
5. **主线：文章详情相关文章推荐与内容发现增强 (P1)**:
    - **推荐策略落地**: 在文章详情页按分类、标签或 translation cluster 相似度提供相关文章推荐，并避免同一翻译簇重复占位。
    - **展示克制**: 推荐模块已以不打断正文阅读的轻量形态落地，不削弱正文主路径可读性。
    - **多语言与回归**: 已补齐多语言过滤、去重与推荐规则的最小回归测试，避免推荐结果跨语言漂移。

### 第二十四阶段：质量守线与回归执行深化 (Quality Guardrails & Regression Execution Deepening) (已审计归档)

**审计结论**: 第二十四阶段围绕测试覆盖率与红绿测试有效性深化、ESLint / 类型债分批收紧、重复代码与纯函数复用收敛，以及阶段级回归任务执行与收口证据链复盘四条主线，已在代码、测试、规划文档与收口 artifact 中完成闭环。正式放行依据为 [2026-04-08-phase-close-regression.md](../../artifacts/review-gate/2026-04-08-phase-close-regression.md) 的 `Pass` 结论；当前 `docs:check:source-of-truth` 中英文翻译时效告警仍按 pre-release 非阻断 warning 口径留给后续治理，不影响本阶段归档。

**时间表**: 约 1 - 1.5 个月
**目标**: 在第二十三阶段完成触达扩面与运行时稳态收口之后，把 backlog 中已经持续存在、且适合跨阶段切片推进的四条长期治理主线正式上收为当前阶段，围绕“测试覆盖率与红绿有效性、ESLint / 类型债分批收紧、重复代码与纯函数复用收敛，以及一次真实的阶段级回归任务”形成新的执行组合，避免这些长期任务在阶段切换后再次失去明确落点。

**准入结论**: 本阶段四条主线均来自 backlog 长期主线任务区，已按双轨治理规则完成切片执行、证据沉淀与归档；下一阶段是否正式上收，转由 backlog 候选池与容量评估决定。

**ROI 评估**: 测试覆盖率与红绿测试有效性治理 2.00；ESLint / 类型债分批收紧 1.50；重复代码与纯函数复用收敛 1.40；阶段级回归任务执行与证据链复盘 2.00。四项均满足本阶段主线准入条件，其中测试有效性与阶段级回归任务执行为 P0 主线。

1. **主线：测试覆盖率与红绿测试有效性深化 (P0)**:
    - **覆盖率目标**: 在当前基线上继续提升至少 5%，阶段目标先达到约 68%，并明确记录本轮新增覆盖率、模块分布与未覆盖边界。
    - **红绿测试优先**: 围绕高回归风险模块补齐失败路径、边界断言与回退逻辑验证，避免把测试补强退化为只堆成功路径的 coverage 工程。
    - **结果留痕**: 每一轮补强都要保留失败用例、转绿结果、仍未覆盖范围与下一轮优先顺序，作为后续阶段继续切片的基线。
2. **主线：ESLint / 类型债分批收紧 (P1)**:
    - **分批推进**: 仅选择 1 - 2 组收益最高、回滚边界最清晰的规则族继续上收，不做一次性全仓强收紧。
    - **基线与回退**: 对 warning、目录豁免和高频误报先建立基线，再为每批规则提升补齐收益说明、回滚方式与最小验证矩阵。
    - **边界隔离**: 测试、脚本、迁移和历史遗留目录继续保持受控豁免，防止规则收紧演变为失控返工。
3. **主线：重复代码与纯函数复用收敛 (P1)**:
    - **热点优先**: 先聚焦重复度最高且近期频繁变更的模块，按“抽象收益大于引入复杂度”原则收敛共享 helper、纯函数和重复分支。
    - **最小切片**: 本阶段只做一轮小范围治理，不把长期复用收敛扩写成跨目录的大重构。
    - **证据对比**: 输出本轮重复代码基线、抽象后收益与仍需观察的模块，避免后续阶段再次从零开始评估。
4. **主线：阶段级回归任务执行与收口证据链复盘 (P0)**:
    - **真实执行**: 基于既有固定入口完成一次带明确 timeout budget 的阶段级回归任务，而不是只保留规范层面的命令组合。
    - **结果统一**: 至少覆盖 coverage、lint / typecheck、重复代码、文档一致性与 Review Gate 结论，并统一沉淀到 [回归活动窗口](../reports/regression/current.md)。
    - **blocker 判定**: 若回归任务发现 blocker，必须显式回灌当前阶段待办；若未发现 blocker，也要记录未覆盖边界与下一次补跑条件。
