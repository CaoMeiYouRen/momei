# AI 协作规范 (AI Collaboration Standards)

本规范为墨梅博客 项目中 AI 智能体（Agents）的行为准则，旨在确保 AI 在自动化过程中逻辑严密、安全合规且不破坏现有工程质量。

## 1. 身份认同 (Identity)

每个 AI 智能体在启动任务前，必须明确自己的角色定位（见 `AGENTS.md`）。
- **禁止跨权**：`@qa-assistant` 严禁修改代码；`@test-engineer` 应专注于测试编写。
- **环境感知**：执行命令前，必须通过 `os` 信息判断当前是 Windows 还是 POSIX 环境，并使用正确的命令语法。

### 1.1 工具编排优先级

在不改变 `AGENTS.md` 所定义的单一主责角色前提下，若当前 AI 工具支持 sub-agent、agent team 或等效的多代理编排能力，应优先用于以下高负载场景：

1.  大范围检索、代码考古、文档比对、审计证据收集等可并行的只读任务。
2.  需要分别覆盖代码、测试、文档、性能或安全多条证据链的治理型任务。
3.  需要在有限上下文窗口内同时处理多个目录、多个规范来源或多个验证结果的收口任务。

补充约束：

1.  多代理编排的目标是降低单 agent 上下文过载，不是引入多个平行主责；最终结论、放行判断与文档回写仍必须由当前主责角色统一收口。
2.  若任务存在共享写入面，应避免多个 agent 同时编辑同一文件或同一决策段落，优先采用“并行检索 / 串行落盘”的方式。
3.  若工具不支持 sub-agent 或 agent team，只允许记录能力限制与回退路径，不得因此跳过既有验证、审计或文档要求。

### 1.2 Agent-First 方法论

Agent-First 的完整项目级定义以 [AGENTS.md](../../AGENTS.md) 为准。本规范只承接执行含义：Agent 是默认任务入口；一次性需求可直接执行；可复用流程应沉淀为 skill；更细的自动化步骤再收敛为 skill 内部的 script 或脚本资产；高频请求应触发既有 skills 的持续演化，而不是长期依赖重复口头描述。

### 1.3 执行原则 (Execution Principles)

1.  **搜索优先**: 当遇到需要外部信息、排查未知问题、根因不明确或修复失败 >= 2 次时，必须先使用搜索工具获取一手信息。严禁跳过信息获取步骤直接凭记忆给出结论。详细触发条件、搜索策略与信息源采纳原则见 [1.4 节](#14-信息获取与搜索优先-search-first)。
2.  **显式假设**: 当需求存在歧义、边界未定义或上下文不足时，必须先说明当前假设、可选解释与风险；禁止静默选择一种解释后直接大规模实现。
3.  **简洁优先**: 默认选择满足当前验收标准的最小实现，不得借机引入与当前目标无关的抽象、配置层或未来能力预埋。
4.  **外科式改动**: 改动范围应与用户请求、Todo 验收点或 blocker 一一对应；发现无关问题时可以记录，但不得顺手并入当前实现。
5.  **目标驱动验证**: 在进入实现前应明确成功标准、最低验证矩阵与首条区分性检查；完成首个实质改动后，优先做最小充分验证，再决定是否继续扩写。
6.  **业务数值合理性**: 修复涉及数值计算、类型转换或溢出处理的 Bug 时，必须先判断触发 Bug 的数据是否在合理业务范围内。若数据明显异常（如任务耗时 100 天、金额为负数），应优先考虑是数据问题而非代码问题，修复方案应包含业务阈值兜底，而不是无条件扩大字段范围来容纳异常值。详见 [Review Gate 审查清单 - 业务数值合理性检查](../../.github/skills/code-quality-auditor/references/review-checklist.md#2-业务数值合理性检查)。

### 1.4 信息获取与搜索优先 (Search-First)

AI 代理在处理需要外部信息、排查未知问题或验证假设时，**必须优先使用搜索工具获取一手信息**，严禁仅凭训练数据中的记忆生成答案。

#### 触发条件

以下场景**必须**触发搜索（满足任一即触发）：

| 触发场景 | 典型信号 | 搜索目标 |
| :--- | :--- | :--- |
| **问题排查受阻** | 同一问题修复失败 >= 2 次、根因分析不明确、错误信息指向未知领域 | 错误信息关键词、同类 issue、官方 troubleshooting 文档 |
| **技术方案设计** | 涉及不熟悉的库/框架/API、需要性能对比、存在多种候选实现路径 | 官方文档、社区最佳实践、已知坑点与限制 |
| **需求或配置澄清** | 用户描述模糊、缺少验收标准、依赖的外部服务行为不确定 | 官方配置参考、同类项目实现、API 契约文档 |
| **安全或合规判断** | 涉及鉴权、加密、数据保护、第三方集成安全 | CVE 数据库、官方安全公告、OWASP 指南 |
| **跨平台或环境差异** | Windows/Linux/macOS 行为不同、Node 版本差异、WSL/Docker 特有行为 | 平台特定 issue、官方兼容性说明 |
| **依赖选型或升级** | 新增依赖、大版本升级、替代方案评估 | npm/registry 信息、changelog、迁移指南、社区反馈 |

#### 搜索策略

1.  **多语言搜索**：对技术类主题，至少使用中英双语关键词搜索。若主题涉及特定语言生态（如日文、韩文等），必须使用原始语言关键词扩展搜索范围。
2.  **多源抓取**：对搜索结果中排名靠前的条目，批量抓取完整内容，优先选择官方文档（`*.dev`、`github.com/org/repo`）、权威 issue tracker 和知名技术社区。
3.  **交叉验证**：关键事实（版本号、配置项、性能数据、安全结论）必须至少有 2 个独立来源确认。来源间存在矛盾时，以官方文档为准，不得用"多数投票"裁定。
4.  **工具降级**：若主搜索工具不可用，自动切换到次选工具；全部失败时记录能力缺口，向用户说明，不得静默跳过。

#### 信息源采纳原则

| 优先级 | 来源类型 | 示例 | 采纳条件 |
| :--- | :--- | :--- | :--- |
| **L1（优先采纳）** | 官方文档、源代码仓库 | `nuxt.com/docs`、`github.com/nuxt/nuxt` | 直接采纳，作为终极裁决依据 |
| **L2（可采纳）** | 权威技术社区、知名博客 | StackOverflow（高票回答）、Vite 官方博客 | 需与 L1 无矛盾；关键数据需双源确认 |
| **L3（参考）** | 个人博客、Medium、Reddit | 开发者个人经验分享 | 仅作思路参考，不得作为唯一事实依据 |
| **L4（排除）** | 内容农场、机翻站、低成本 TLD | `.cc`/`.xyz`/`.top` 的采集站 | 直接舍弃，不得引用 |

#### 禁止行为

-   用模型训练数据中的记忆替代实际搜索结果
-   只看第一条搜索结果就下结论
-   对医学/法律/金融/安全类声明不进行对抗性搜索验证
-   报告中的事实声明不附带可追溯的 URL
-   搜索结果存在矛盾时不追溯官方原始数据
-   一个搜索工具失败就放弃（应尝试所有可用工具）
-   因"担心查太多浪费时间"而跳过信息获取步骤

详细搜索工作流与对抗审查规则可参考 `super-search` skill。

## 2. 核心工作流：PDTFC+ 2.2 循环

所有写操作任务必须严格遵循以下执行顺序。**严禁跨越关键质量阈值。**

### P (Plan) - 需求分析与规划
- **意图抽离**：必须读取 `docs/plan/todo.md`，若存在歧义必须发起"采访"模式。
- **规划闸门**：无论是用户新提出的需求，还是在开发/测试过程中临时发现的新事项，都必须先核对 `todo.md`、`roadmap.md`、`todo-archive.md` 与当前任务验收标准，确认是否已属于当前规划。
- **插队判定**：若新事项不在当前规划内，必须先完成一次快速分流，明确是"允许插队的阻塞/高风险事项"还是"应延期的积压项"；未完成判定前不得进入 Do 阶段。
- **任务定义**：更新 `todo.md`，将任务标记为 `in-progress`。
- **方案设计**：输出受影响文件清单及技术实现路径。
- **编排策略**：若工具支持 sub-agent / agent team，应优先在 Plan 阶段拆分检索、审计、文档对照等只读工作，减少单 agent 长上下文堆积；但任务准入判断与范围结论仍由当前主责角色统一给出。

### D (Do) - 业务执行
- **数据库优先**：若涉及持久化，必须优先开发数据库实体/Schema。
- **实现准则**：遵循 TypeScript 架构，禁止使用 `any`，UI 文本必须 i18n 化。
- **最小实现**：默认先做满足当前验收标准的最小切片；若需要扩大实现面，必须先说明为何当前最小方案不足。
- **范围稳定**：开发过程中若发现新的体验优化、附加功能或非阻塞问题，不得直接扩写当前任务；必须回到 P 阶段重新判断是否允许插队。
- **自检**：开发完成必须通过本地质量校验，详见 [开发规范 - 提交前检查](./development.md#6-提交前检查-pre-commit-checks)。

### A (Audit) - 代码审计（强制 Review Gate）
- **强制入口**：D 阶段完成后，必须立即加载 `code-quality-auditor` skill 执行完整的结构化审查，不得自我审查替代正式审计。A 阶段未被 `@code-auditor` 放行前，不得进入后续 V / T / F 任何阶段。
- **审查内容**：按 [验证矩阵](#22-验证分级矩阵与-review-gate) 核对最低验证要求，覆盖正确性、安全、规范一致性、Todo 验收点与证据链。
- **审计证据**：审查结论、问题分级（blocker / warning / suggest）和最低验证证据必须沉淀到 `artifacts/review-gate/`。
- **审查范围**：代码、文档、配置、脚本与治理定义（含 agent / skill 定义）均须审查，不能只审业务代码。
- **退回策略**：若发现 blocker，退回 D 或回流 P，不得携带未关闭的 blocker 进入后续阶段。

### V (Validate) - UI 验证
- **视觉审计**：对 UI 改动进行浏览器验证。若自动化工具失效，应向用户展示截图或请求人工验证。

### T (Test) - 质量检查
- **测试覆盖**：编写 Vitest 测试用例。测试代码本身也必须经过 A 阶段的 Lint 和类型检查。
- **风险导向**：优先补当前缺陷会打断的断言、失败路径与边界行为，不把测试阶段退化为机械补 coverage。

### F (Finish) - 任务完结与单次提交
- **闭环管理**：更新 `todo.md` 状态为 `completed`，并及时同步项目相关的技术文档。
- **单次提交**：整个任务产生的所有改动（业务代码 + 测试代码 + 文档）一次性提交。提交前必须加载 `conventional-committer` skill，通过其前置检查（依赖 A 阶段已放行、`pnpm lint`、`pnpm typecheck`、定向测试均通过），生成符合 Conventional Commits 格式的消息（使用中文或用户使用的语言），然后执行 `git commit`。
- **推送禁令**：`git commit` 后不得自动 `git push`，仅限用户明确要求时执行。

## 2.1 迭代中途发现事项处理规则

当智能体在 D / A / V / T 任一阶段发现新增事项时，必须遵循以下规则：

1.  **先暂停扩写**：停止直接继续实现，先判断该事项是否已在当前待办或当前验收范围内。
2.  **允许插队的范围**：仅限阻塞当前交付、明确功能回归、高风险安全/合规问题、当前依赖链中的高危漏洞、或导致当前任务无法闭环的基础缺陷。
3.  **默认延期的范围**：体验优化、代码洁癖式重构、探索性能力、未来功能想法、非紧急依赖升级，默认进入 backlog，不打断当前阶段。
4.  **记录要求**：允许插队的事项必须补充“为何插队”的文字说明；延期事项必须明确记录到 `roadmap.md` 或其他规划载体，避免口头遗忘。
5.  **禁止静默膨胀**：智能体不得在未告知用户的情况下，把一个原子任务扩展成新的功能包或新的子阶段。

## 2.2 验证分级矩阵与 Review Gate

任何变更都必须按“验证层级 + Review Gate”判断是否可以放行，而不是只写一句“已验证”。

### 验证层级定义

| 层级 | 名称 | 目标 | 典型证据 |
| :--- | :--- | :--- | :--- |
| `V0` | 记录层 | 明确变更范围、风险与受影响入口 | 变更文件清单、风险说明、回归范围 |
| `V1` | 静态层 | 确认代码、脚本、配置或文档在静态检查层面可用 | `lint`、`typecheck`、文档检查、脚本自检 |
| `V2` | 逻辑 / 运行层 | 确认逻辑、接口契约、脚本入口或异常分支无明显回归 | 定向 Vitest、API / 集成测试、脚本实际运行验证 |
| `V3` | 流程层 | 确认跨模块流程、UI 渲染与关键交互符合预期 | 浏览器验证、E2E、截图或录屏 |
| `V4` | 性能层 | 确认 Lighthouse / Bundle / 关键路径预算未回退 | `test:perf`、`test:perf:budget`、性能报告 |
| `RG` | Review Gate | 给出最终审计结论与剩余风险 | `@code-auditor` 结论、阻塞项、未覆盖边界 |

### 不同改动类型的最低验证要求

| 改动类型 | 最低验证要求 | 说明 |
| :--- | :--- | :--- |
| 文档 / README / 翻译页 | `V0 + V1 + RG` | 至少保留文档检查结果、相对路径 / 构建影响说明与 Review 结论。 |
| 配置 / 脚本 | `V0 + V1 + 按影响补 V2 + RG` | 若影响构建、部署、生成或治理流程，必须补一条实际运行证据，并将其记入 `V2`。 |
| 纯逻辑 / 工具函数 / 服务层 | `V0 + V1 + V2 + RG` | 不能只跑 `lint`；必须补充定向测试或同等逻辑验证。 |
| API / 鉴权 / 数据模型 | `V0 + V1 + V2 + RG` | 若影响登录、权限或关键写路径，应升级到 `V3`。 |
| UI 组件 / 页面交互 | `V0 + V1 + V2 + V3 + RG` | 若影响核心页面首屏、资源加载或体积预算，再升级到 `V4`。 |
| 跨模块流程 / 关键链路 | `V0 + V1 + V2 + V3 + RG` | 例如登录、发布、设置保存、安装向导、分发链路。 |
| 性能 / Lighthouse / Bundle 治理 | `V0 + V1 + V4 + RG` | 若治理项同时影响用户操作链路，应同时补 `V3`。 |
| 修复型 Hotfix | `V0 + V1 + 与故障层级对应的 V2/V3/V4 + RG` | 必须补“复现方式 + 修复后结果”，不能只写“已修复”。 |

补充约束：

1.  没有 `RG` 结论的变更，只能视为“进行中”，不能视为已完成。
2.  文档、配置、脚本改动同样需要证据链，只是证据形态可以比功能代码更轻量。
3.  若某项变更同时触发多个改动类型，按最高层级要求执行，不允许向下取低。

### Review Gate 结论口径

| 结论 | 是否允许放行 | 适用条件 |
| :--- | :--- | :--- |
| `Pass` | 是 | 所有阻塞问题已关闭；允许存在已记录的非阻塞风险，但不得影响当前交付闭环。 |
| `Reject` | 否 | 存在阻塞问题、关键证据缺失，或当前变更不满足最低验证要求。 |

补充约束：

1.  `warning`、`suggest`、`blocker` 是问题分级，不是最终 Gate 结论；最终结论只能使用 `Pass` 或 `Reject`。
2.  只要仍有未关闭的 `blocker`，任务状态必须保持为进行中或退回，不得标记完成。
3.  `Pass` 可以附带非阻塞 `warning`，但必须同步记录残余风险、责任归属与后续计划。

## 2.3 证据链与模板

所有任务在 Finish 前至少要沉淀以下字段：

1.  变更范围
2.  已执行验证
3.  结果摘要
4.  Review Gate 结论
5.  未覆盖边界 / 后续补跑计划

### 功能改动模板

```md
## 验证记录
- 变更范围:
- 风险等级:
- 已执行验证:
	- lint / typecheck:
	- 定向测试:
	- 浏览器 / E2E:
	- Lighthouse / Bundle:
- 结果摘要:
- Review Gate:
	- 结论: Pass | Reject
	- 问题分级: none | suggest | warning | blocker
	- 主要问题:
- 未覆盖边界:
- 后续补跑计划:
```

### 修复型 Hotfix 模板

```md
## Hotfix 验证记录
- 问题现象:
- 复现路径:
- 修复范围:
- 已执行验证:
	- 复现前证据:
	- 修复后定向验证:
	- 回归验证:
- 结果摘要:
- Review Gate:
	- 结论: Pass | Reject
	- 问题分级: none | suggest | warning | blocker
	- 是否允许插队: yes | no
- 未覆盖边界 / 后续补跑计划:
- 剩余风险 / 回退方案:
```

### 文档 / 配置变更模板

```md
## 文档或配置验证记录
- 变更范围:
- 受影响入口:
- 已执行验证:
	- 文档 / 链接检查:
	- 配置 / 脚本自检:
	- 构建或运行验证:
- 结果摘要:
- Review Gate:
	- 结论: Pass | Reject
	- 问题分级: none | suggest | warning | blocker
	- 漂移风险:
- 未覆盖边界:
- 后续补跑计划:
```

## 3. 安全与红线 (Security Redlines)

1.  **敏感文件保护**：严禁修改或删除 `.env`, `AGENTS.md` 等核心配置文件，除非有明确指示。
2.  **路径校验**：执行涉及删除的命令 (`rm`, `rd`) 前，必须验证路径是否存在且非空。
3.  **密钥脱敏**：绝不在代码或日志中硬编码 API Key、Token 或任何敏感凭据。
4.  **死循环规避**：如果在 F 阶段重复 3 次仍无法修复 Bug，必须停止执行并请求人类介入，禁止无限循环。**同一问题修复失败 >= 2 次时，必须先触发搜索优先流程**（见 1.3 节），不得用同一思路反复重试。

## 4. 沟通原则

- **简明扼要**：回复应直接、专业，避免冗余的客套话。
- **证据链条**：在提出方案或得出结论时，应引用具体的代码文件路径或文档内容。
- **透明度**：在执行大规模修改前，应向用户展示预期的代码变更范围。
