# 墨梅博客 项目规划与评估规范 (Planning & Evaluation Standards)

## 1. 概述 (Overview)

本文档旨在为 “墨梅博客” 的项目演进提供一套科学、可执行的规范与方法论。通过明确的约束条件和评估矩阵，确保开发资源集中于核心价值，防止项目因 “范围蔓延 (Scope Creep)” 而失去焦点。

## 2. 规划约束 (Planning Constraints)

在进行任何阶段性规划（Phase Planning）或重大功能引入时，必须遵循以下硬性约束：

-   **阶段性聚焦 (Phase Focus)**: 
    -   单个开发迭代 (Phase) 中的核心任务大项应控制在 **5-6 项** 以内。
    -   当新需求加入导致数量超标时，必须遵循 “进一出一” 原则，确保团队精力不被分散。
-   **功能解耦 (Decoupling)**: 
    -   新功能的引入不得破坏现有核心逻辑（如认证、文章显示、SEO）。
    -   如果涉及核心架构变动，必须先进行技术预研并更新 `docs/design/` 文档。
-   **测试先行 (Test-First Planning)**: 
    -   在规划任务时，必须同步制定测试方案（单元测试或端到端测试）。
    -   任何不包含测试补全计划的功能规划均不具备准入资格。
-   **优先级驱动 (Priority Driven)**: 
    -   规划时必须进行 ROI（投资回报率）分析，优先安排对用户体验有直接提升或具有核心竞争力的功能。
-   **阶段切换闸门 (Phase Transition Gate)**:
    -   即便当前阶段的待办条目已经全部完成，只要尚未完成 `todo.md` 清理、回归记录归档、阶段总结或 `todo-archive.md` 归档整理，就视为**仍处于当前阶段收口中**。
    -   在当前阶段尚未归档前，禁止直接在 `roadmap.md` 中开启下一阶段的正式规划；任何新来的需求默认先写入 `backlog.md`，待当前阶段归档完成后再统一评估是否上收。
-   **规划与文档 Review 必须先行 (Planning & Documentation Review First)**:
    -   规划文档、待办文档、归档文档、规范文档与其他 Markdown 文档的改动，同样属于需要审查的正式变更，不得因为“只是写文档”或“只是改规划”而跳过 review。
    -   任何改动在进入提交前，都必须至少经过一轮 review；若 review 未形成明确结论或问题列表仍未闭环，不得进入 commit 阶段。
-   **迭代内新增事项闸门 (In-Iteration Intake Gate)**:
    -   在开发、测试、审计或联调过程中发现的新问题、新想法、新优化点，默认**不自动并入**当前阶段待办。
    -   智能体或开发者必须先核对该事项是否已包含在当前 `todo.md`、当前任务验收标准或 `roadmap.md` 的本阶段范围内；只有已属于当前范围的事项，才允许直接继续执行。
    -   若该事项不在当前规划内，必须先执行一次快速需求评估，判定其是“允许插队的高优先级事项”还是“应延期的积压项”；未完成判定前，严禁直接进入开发。

-   **需求采访与真实意图抽离 (Requirement Interview & Intent Extraction)**:
    -   针对模糊的需求，智能体**必须**通过“采访”方式反问用户，而非盲目猜测。
    -   采访原则：先结构后细节，一步步递进；一个问题一个问题地问；目标是抽离出最核心、最真实的业务需求。
    -   在核心需求未对齐前，严禁进入开发 (Do) 阶段。

## 3. 评估方法论：Momei 评估矩阵 (Evaluation Methodology)

为了客观评估一个提议的功能或变动是否应进入 “近期规划”，引入以下评分系统 (Score)：

### 3.1 评分维度

| 维度 | 描述 | 分值 (1-5) |
| :--- | :--- | :--- |
| **价值 (Value)** | 对最终用户的核心体验提升或业务价值。 | 1 (低) - 5 (极高) |
| **契合度 (Alignment)** | 与 [项目计划](../plan/roadmap.md) 中长期目标的契合程度。 | 1 (离散) - 5 (核心) |
| **复杂度 (Difficulty)** | 技术实现难度、开发工时及第三方库引入限制。 | 1 (极易) - 5 (极大) |
| **风险 (Risk)** | 带来的潜在 Bug、性能退化或长期地狱式维护成本。 | 1 (安全) - 5 (高危) |

### 3.2 计算公式

$$Score = \frac{Value + Alignment}{Difficulty + Risk}$$

### 3.3 准入基准 (Decision Rules)

-   **Score \> 1.5**: 优先进入当前或下一阶段 Phase。
-   **1.0 \< Score \<= 1.5**: 进入 `Backlog` (待办项)，视剩余资源情况分配。
-   **Score <= 1.0**: 除非是安全性修复 (Hotfix)，否则应予以取消或长期搁置。

### 3.4 迭代中途新增事项的快速分流 (Mid-Iteration Triage)

当在当前迭代执行过程中发现额外事项时，必须按以下顺序处理：

1.  **范围核对**: 先检查该事项是否已经存在于 `docs/plan/todo.md`、当前任务的验收标准、`docs/plan/roadmap.md` 当前阶段或 `docs/plan/todo-archive.md` 的历史条目中。
2.  **分类决策**:
    -   **A 类：当前范围内事项**。如果该事项本质上只是补足当前任务的既有验收标准、修正当前实现遗漏或收口当前任务必需的配套工作，则允许继续执行，但应在任务记录中补充说明，不得借机扩大需求边界。
    -   **B 类：允许插队事项**。仅当该事项满足以下至少一项时，才允许插入当前迭代：阻塞当前待办交付、造成明确功能回归、属于高风险安全/合规问题、属于当前依赖链中的高危漏洞或当前任务无法绕开的基础缺陷。此类事项必须补充原因说明，并作为修复型或派生型任务纳入当前阶段。
    -   **C 类：延期事项**。新的体验优化、非阻塞重构、探索性想法、未来功能延伸、非紧急依赖升级等，统一延后处理，不得直接打断当前阶段实施。
3.  **容量控制**: 若 B 类事项进入当前阶段后导致核心任务数量超出 5-6 项，必须遵循“进一出一”原则，明确挪出一项低优先级内容至 `backlog.md`。
4.  **记录落点**:
    -   A 类事项：保留在当前任务上下文中处理，必要时补充验收说明。
    -   B 类事项：写入 `todo.md` 当前阶段，并标注插队原因或阻塞背景。
    -   C 类事项：写入 `docs/plan/backlog.md`，不得伪装成当前阶段任务。

## 4. 路线图与待办管理流程 (Management Workflow)

### 4.1 路线图 (Roadmap) 更新
-   `docs/plan/roadmap.md` 每月进行一次大版本审查。
-   已完成的任务应在其旁边标注 `(已完成)`。
-   新的 Phase 必须在上一 Phase 核心指标达成后方可开启。
-   **阶段切换前置条件**: 上一阶段只有在 `todo.md` 已完成清理、必要的回归记录已归档、对应内容已进入 `todo-archive.md` 后，才允许在 `roadmap.md` 中启动下一阶段的正式规划；在此之前，新需求一律先进入 `backlog.md`。
-   **backlog 双轨维护原则**: `docs/plan/backlog.md` 必须至少区分“长期主线任务”和“短期 / 一次性候选任务”两类条目。长期主线用于跨多个阶段持续推进的治理或能力建设，短期 / 一次性候选用于单次评估、单阶段落地或完成后即可关闭的事项。
-   **阶段升格去重原则**:
    -   长期主线任务被某个阶段抽取为当期切片时，不得整体从 `docs/plan/backlog.md` 消失；应保留其主线卡片，并补记“最近一次上收阶段”“当前状态”与“下一次可继续切片方向”。
    -   短期 / 一次性候选被正式纳入新的 Phase、当前阶段待办或修复型 Hotfix 时，必须立即从 `docs/plan/backlog.md` 中移除，或显式改写为“已上收至某阶段”的历史说明。
    -   严禁同一事项的完整实施正文同时出现在“当前 / 新阶段规划”和 backlog 两处；backlog 只保留长期主线摘要卡片或短期候选入口，避免重复统计与优先级混乱。

### 4.2 待办事项 (Todo) 维护
-   `docs/plan/todo.md` 是实时操作手册，仅包含**当前阶段的具体实施任务**。
-   `docs/reports/regression/current.md` 用于集中维护周期性回归、阶段基线与补跑记录；`todo.md` 仅保留当前阶段上下文、摘要与链接，不再堆叠长篇回归正文。
 -   `docs/reports/regression/current.md` 作为活动回归窗口，默认只保留最近 1 - 2 个阶段或最近 6 - 8 条完整回归记录；更早的历史记录应按滚动归档规则迁移到 `docs/reports/regression/archive/` 或其后续分拆文件。
-   任务状态必须清晰标记：`[ ]` (待办), `[x]` (已完成), `[-]` (已取消)。
-   任务描述应包含具体的 "验收标准 (Acceptance Criteria)"。
-   **查重原则**: 在规划或添加任何新功能前，**必须**同步查阅 `docs/plan/todo.md`、`docs/plan/roadmap.md` 以及 `docs/plan/todo-archive.md`。严禁重复设计或规划已被归档、已在当前待办中或已在路线图中的任务。
-   **中途发现事项处理原则**: 在执行当前任务时新增发现的问题，不得直接把所有事项都塞入 `todo.md`。只有通过“3.4 迭代中途新增事项的快速分流”判定为 A 类或 B 类的内容，才允许进入当前阶段的执行面。
-   **功能需求管理原则**:
    -   **新功能/新需求**: 必须添加到 `docs/plan/backlog.md`。
    -   **阶段未归档时的新增需求**: 若当前阶段尚未完成归档整理，即便现有待办已全部打勾，新来的需求仍必须先写入 `docs/plan/backlog.md`，不得直接当作下一阶段正式任务写入 `roadmap.md`。
    -   **当前阶段任务**: 只有在经过评估并决定纳入当前阶段执行后，才可以从 backlog 提取到 roadmap / todo 的当前阶段部分。
    -   **长期主线维护**: 覆盖率治理、测试有效性、ESLint / 类型债收紧、重复代码 / 纯函数复用收敛、周期性回归、性能 / 安全基线等可跨阶段反复切片的事项，应维护在 backlog 的“长期主线任务”区域，并记录最近一次上收阶段、当前基线与下一切片候选，而不是随着某一阶段提取后整体删除。
    -   **阶段提取后去重**: 短期 / 一次性候选一旦被提取并写入当前阶段或下一阶段的正式规划，必须同步从 `docs/plan/backlog.md` 删除；长期主线则保留主卡片，只移除与当期阶段完全重复的实施正文。
    -   **紧急修复**: Bug 修复和安全性修复可以添加到当前阶段的 todo 部分，并明确标注为修复类型。
    -   **优化需求**: 必须经过评估后，根据优先级决定是放入 `backlog.md` 还是当前阶段的 todo。
-   **Review 前置原则**: 规划变更、归档整理、回归记录更新与规范文档更新，在提交前同样必须经过至少一轮 review；未经过 review 的规划改动不得视为已正式生效。
-   **周期性回归任务**: 代码优化与复用收敛、ESLint warning / 类型债治理、`database/*/init.sql` 与实体 / 设计文档同步、README / 部署 / 翻译文档同步、i18n 翻译大文件拆分后的初始化字段完整性、测试补齐与覆盖率治理、性能基线审计、依赖安全审计（Dependabot / `pnpm audit --registry=https://registry.npmjs.org/`）、`scripts/**` 中未使用长期脚本 / 失效入口引用 / 临时脚本残留 / 无效脚本清理，以及 `max-lines` 超限文件与 `/* eslint-disable max-lines */` 等临时豁免收敛等易漂移事项，应以独立治理任务或阶段形式规划。此类任务允许执行全量测试与 coverage，但必须显式声明 timeout budget，并记录结果与未覆盖边界。
-   **回归任务隔离原则**: 上述周期性事项默认属于独立治理任务，不自动膨胀进普通功能需求；只有在阻塞当前交付、造成明确功能回归或构成高风险问题时，才允许插队进入当前阶段。
 -   **回归记录落点**: 周期性回归、阶段基线、补跑记录与 Review Gate 证据链正文默认写入 `docs/reports/regression/current.md`；`todo.md` 和 `roadmap.md` 只保留任务状态、阶段摘要与文档入口，避免规划文档反复复制同一份长记录。
 -   **回归日志滚动归档**: 当 `docs/reports/regression/current.md` 超过 300 - 400 行、或累计超过 6 - 8 条完整记录且已影响当前阶段阅读效率时，应将较早且不再服务最近基线比较的记录整体迁移到 `docs/reports/regression/archive/`；主窗口必须继续保留足够支撑最近阶段对比与发版判断的近线记录。

#### 依赖安全回归补充规则

依赖安全审计属于周期性回归任务中的强制检查项，执行时应遵循以下口径：

1.  **数据来源优先级**: 优先读取仓库 Dependabot / Security 页面；若当前工具链无法稳定访问，再回退到 `pnpm audit --registry=https://registry.npmjs.org/` 或其他可复现的官方审计来源。
2.  **可修复项优先处理**: 若告警存在明确可升级版本、可替换版本或可通过锁文件覆盖修复的依赖版本，应优先安排修复，并在同一任务中补齐 `lint`、`typecheck`、定向测试或等价运行验证，确认升级后仍可正常运行。
3.  **不可修复项记录范围**: 若当前不存在可用修复版本、上游尚未发布补丁，或修复需要超出当前阶段范围的架构替换，则文档中只记录 `high` 及以上级别问题；`moderate` / `low` 可在当次回归结论中忽略，不要求写入治理记录。
4.  **延期或计划修复判定**: 对于已记录的不可修复 `high+` 风险，必须额外写明影响范围、临时缓解措施、可暂时搁置的理由，或纳入后续计划修复的触发条件与预期窗口，禁止只保留“存在风险”而不做处置判断。
5.  **插队边界**: 当前依赖链中的 `high+` 漏洞若阻塞发版、影响线上入口、触发合规要求，允许作为 B 类事项插入当前阶段；其余低优先级依赖升级不得借机扩写为大规模升级工程。

#### 周期性回归任务模板

每个回归任务至少应包含以下字段：

| 字段 | 说明 |
| :--- | :--- |
| 回归范围 | 明确模块、目录、命令或质量债范围，避免“顺手全仓库治理”。 |
| 触发条件 | 例如发版前、阶段收口、目录迁移后、实体字段调整后、CI 连续告警后。 |
| 执行频率 | 例如每周、每次发版前、每月一次、专项阶段结束前。 |
| timeout budget | 标明定向测试、全量测试、coverage、verify 的预算或升级条件。 |
| 输出格式 | 至少包含已执行验证、结果摘要、Review Gate 结论、问题分级、未覆盖边界与后续补跑计划；再按验证矩阵要求补充测试结果、浏览器验证、性能结果，以及依赖安全回归时的漏洞来源、修复状态与延期/计划修复判断。 |

建议按以下口径规划常见回归项：

| 回归事项 | 默认频率 | 常见触发条件 |
| :--- | :--- | :--- |
| ESLint warning / 类型债 / `max-lines` 豁免清理 | 每周一次 | 告警持续积压、单文件继续膨胀 |
| `database/*/init.sql` 与实体 / 设计文档同步 | 每次发版前 | 实体字段、迁移逻辑或初始化数据变更 |
| README / 部署 / 翻译文档同步 | 每次发版前 | 新增脚本、部署入口、文档目录迁移 |
| i18n 初始化字段完整性审计 | 每次 locale 大文件拆分后 | locale key 拆分、注册表新增语言、设置项新增多语言字段 |
| 长期脚本入口与残留清理 | 每两周一次 | 新增长期脚本、临时脚本落盘、入口引用漂移 |
| 覆盖率治理与性能基线审计 | 每周一次或阶段收口前 | 核心模块覆盖率下滑、Lighthouse / Bundle 告警 |
| 依赖安全审计 | 每次发版前或每周一次 | Dependabot 告警、新增依赖、锁文件集中变更、安全通报出现 |

#### 固定调度入口

为避免同一类周期性回归长期停留在“规范有了但每次靠人工重新拼命令”，当前阶段统一使用以下三条固定入口：

| 节奏 | 正式入口 | 最小固定组合 | 责任边界 | blocker 规则 |
| :--- | :--- | :--- | :--- | :--- |
| 周级治理 | `pnpm regression:weekly` | `test:coverage` + `security:audit-deps` + `docs:check:source-of-truth` + `docs:check:i18n` + `duplicate-code:check` | `@full-stack-master` 或当前值班开发者执行；`@code-auditor` 复核 blocker；`@documentation-specialist` 回写 `docs/reports/regression/current.md` | 任一 required 命令失败即 blocker；活动日志窗口超限先记 warning |
| 发版前 | `pnpm regression:pre-release` | `release:check:full` + `docs:check:i18n` + `test:perf:budget:strict` + `duplicate-code:check` | `@full-stack-master` 执行；`@code-auditor` 决定放行；结果摘要继续沉淀到 `docs/reports/regression/current.md` | 任一 required 命令失败即 blocker |
| 阶段收口前 | `pnpm regression:phase-close` | `test:coverage` + `release:check:full` + `docs:check:i18n` + `test:perf:budget:strict` + `duplicate-code:check:strict` + `review-gate:generate:check` | `@full-stack-master` 执行并补齐 Review Gate；`@code-auditor` 给出 Pass / Reject；`@todo-manager` / `@documentation-specialist` 仅在通过后推进归档 | 任一 required 命令失败即 blocker；若 `docs/reports/regression/current.md` 超过 `400` 行或 `8` 条记录且尚未滚动归档，也直接视为 blocker |

补充约束：

1. 三条固定入口生成的 artifact 仅作为引用证据，正式结果摘要仍统一沉淀到 `docs/reports/regression/current.md`。
2. `duplicate-code:check` 在周级 / 发版前入口中默认只作为 warning 基线；进入 `phase-close` 时必须升级到 strict 口径。
3. `docs/reports/regression/current.md` 触发滚动归档的条件继续保持唯一事实源：超过 `300 - 400` 行、或超过 `6 - 8` 条完整记录且已影响当前阶段阅读效率；`phase-close` 入口将其中的上限口径编码为正式 blocker。

- 执行频率:

    - 已执行验证:
    - 结果摘要:

    回归记录维护要求：

    1.  同一次回归的正文只保留在一个位置，默认写入 `docs/reports/regression/current.md`。
    2.  若某次回归服务于当前阶段收口，可在 `todo.md` 中保留一句摘要与链接，但不得再次完整复制正文。
    3.  若某次回归直接影响阶段准入或发版决策，应在 `roadmap.md` 或对应阶段文档中补充结论摘要，而不是重复粘贴整份记录。

### 4.3 **不可逆原则：绝不回溯修改已完成阶段**
-   **禁止修改原则**: 一旦某个阶段的任务被标记为 `[x]` (已完成)，严禁回溯修改该阶段的条目，包括但不限于：
    -   修改验收标准
    -   删除已完成的任务
    -   更改任务优先级
    -   重新规划或重新定义已交付的功能
-   **条目放置规则**:
    -   **新功能需求**: 必须添加到 `docs/plan/backlog.md`。只有在经过评估并决定纳入当前阶段执行后，才可以从 backlog 移动到 roadmap / todo。
    -   **Bug 修复**: 仅允许添加到当前阶段的"待办"部分，并明确标注为修复类型
    -   **优化需求**: 必须经过评估后，根据优先级决定是放入 roadmap 还是当前阶段的 todo。
-   **变更策略**:
    -   如果发现已完成的功能有不完善之处，应：
        1.  在当前阶段创建 "功能增强" 或 "优化" 任务
        2.  评估后决定是立即执行还是放入长期积压
        3.  基于现有的功能进行扩展，而非删除或重写

### 4.4 归档管理 (Archive Management)
-   **归档时机**: 当一个阶段 (Phase) 的所有核心任务标记为已完成后，应将其详细内容归档至 `docs/plan/todo-archive.md`。
-   **归档形式**: 归档应保留原始待办事项的层级结构、复选框状态及验收标准，确保历史可追溯。
-   **保持简洁**: 归档后，`docs/plan/todo.md` 应仅保留正在进行的阶段的具体实施任务；长期主线与短期候选统一在 `docs/plan/backlog.md` 中管理，`roadmap.md` 只保留正式阶段规划与归档结论。

### 4.5 阶段归档与下一阶段规划标准流程 (Phase Archive Workflow)

任何“当前阶段已做完，准备归档并评估下一阶段”的动作，都必须遵循以下统一流程。该流程是 `todo.md`、`todo-archive.md`、`roadmap.md` 以及相关翻译摘要的唯一归档口径；智能体技能与其他文档只能引用，不得再各自发明第二套步骤。

#### Step 1. 归档准入检查

1. 先确认当前阶段的核心条目已经全部完成，且 `todo.md` 中不再保留未关闭的阶段阻塞项。
2. 若仍有“进行中事项”“待继续观察”“待补跑”之类描述，必须先判断它们是：
    - 已形成独立阻塞，需要继续留在当前阶段。
    - 仅属于运行期观察或后续运维关注，不再阻塞阶段归档。
3. 若该阶段仍缺少 `todo.md` 清理、回归记录落盘、阶段总结或 `todo-archive.md` 归档块，则视为仍在当前阶段收口中，不得开启下一阶段正式规划。

#### Step 2. 证据链检查

1. 必须核对本阶段是否已有足够的回归或验证证据；缺失时先补到 `docs/reports/regression/current.md` 或等价证据载体。
2. 至少确认以下信息已经存在且可引用：
    - 本阶段核心实现或治理结论。
    - 已执行验证与质量门结果。
    - Review Gate 结论。
    - 未覆盖边界与后续观察项。
3. 若阶段收口依赖历史回归记录、归档日志或专项报告，应先补齐索引或引用路径，再进入归档。

#### Step 3. 文档同步检查

1. 中文事实源必须先同步完成：`todo.md`、`todo-archive.md`、`roadmap.md`。
2. 若本次阶段切换涉及 backlog 条目上收、长期主线切片或短期候选去重，必须同步更新 `docs/plan/backlog.md`，不能把 backlog 视为“可选补记”。
3. 若阶段状态发生变化（例如“规划中”变为“已审计归档”），必须同步检查 `docs/i18n/*/plan/roadmap.md` 的摘要是否需要更新。
4. `todo.md` 与 `backlog.md` 默认保持中文，不为阶段归档单独创建翻译件。
5. 若本阶段伴随规范、指南、README 或模块设计文档漂移，必须在归档前一并判断是否需要同步，不能把“文档补写”留作隐性尾项。

#### Step 4. 文件更新顺序

1. 先更新 `docs/plan/todo.md`：清理当前进行中事项，移除已完成阶段的正文，仅保留仍在执行的阶段。
2. 再更新 `docs/plan/todo-archive.md`：追加完整归档块，包含审计结论、主线条目、验收摘要与关键验证结论。
3. 然后更新 `docs/plan/roadmap.md`：把该阶段状态改为“已审计归档”或等价完成状态，并补一句阶段收口结论。
4. 若本次切换涉及 backlog 条目变化，再更新 `docs/plan/backlog.md`：长期主线补记最新状态、最近一次上收阶段与下一次切片方向；短期 / 一次性候选在正式上收后去重或改写为历史说明。
5. 最后按需更新 `docs/i18n/*/plan/roadmap.md`：只同步阶段状态与摘要，不强制翻译 `todo.md` / `backlog.md`。

#### Step 5. 最低验证矩阵

阶段归档本身属于规划文档正式变更，最低必须满足 `V0 + V1 + RG`，且至少补齐以下检查：

1. 文档结构检查：对本次改动的 Markdown 文件执行 `lint-md` 或等价检查。
2. 文档目录检查：执行 `pnpm docs:check:i18n`，确认没有旧目录回流或重复翻译页。
3. 质量状态确认：核对本阶段最近一次 `lint`、`typecheck`、定向测试或专项验证结论是否已落盘；若无法证明当前阶段通过最低质量门，不得直接归档。
4. 错误面检查：确认本次归档涉及文件没有新的编辑器诊断错误。

#### Step 6. Review Gate 与放行结论

1. 归档前必须至少完成一轮 review；可由 `@code-auditor` 或等效审查流程给出 `Pass / Reject` 结论。
2. 若仍存在 blocker、缺失关键证据、或翻译 / 路线图状态明显漂移，则归档结论必须为 `Reject`。
3. `Pass` 时仍可保留非阻塞观察项，但必须显式写入“未覆盖边界”或“后续观察项”，不得伪装为已关闭。

#### Step 7. 下一阶段规划闸门

1. 只有在当前阶段已完成上述归档流程后，才允许把 backlog 条目正式上收为下一阶段规划。
2. 下一阶段候选方案在未落盘前，应以“评估结论 / 候选主线 / ROI 与风险”形式存在，而不是提前写入 `todo.md` 或 `roadmap.md`；候选来源应显式区分是来自 backlog 的“长期主线切片”还是“短期 / 一次性候选”。
3. 若用户只要求“先分析下一阶段”，则必须停留在候选方案层，不得偷跑正式规划落盘。

#### backlog 双轨模板

`docs/plan/backlog.md` 至少应维护以下两类区域：

1. **长期主线任务（可跨阶段保留）**
    - 适用对象: 覆盖率治理、测试有效性、ESLint / 类型债、重复代码 / 纯函数复用、周期性回归、安全 / 性能基线等可反复切片的治理主线。
    - 最少字段: 目标、状态（建议使用“进行中 / 观察中 / 暂停 / 已关闭”之一）、最近一次上收阶段、下一次可切片方向。
    - 管理规则: 某一阶段只提取当期切片进入 roadmap / todo，主线卡片持续留在 backlog，直到明确关闭、取消或被新的长期主线替代。
2. **短期 / 一次性候选任务（上收后去重）**
    - 适用对象: 单次评估、单阶段功能、一次性迁移、独立预研或完成后即可关闭的事项。
    - 最少字段: 验收目标、主要约束、是否满足上收前置条件；若同一区块使用统一共享说明，必须在区块标题下显式声明当前默认状态与上收前置条件口径。
    - 管理规则: 一旦被正式上收到 roadmap / todo，必须从 backlog 删除或改写为历史说明，避免与正式阶段正文重复。

#### 阶段归档最小清单

每次归档至少逐项确认以下清单：

- 当前阶段核心条目已完成或已明确转为非阻塞观察项。
- `todo.md` 已清理当前阶段正文。
- `todo-archive.md` 已追加归档块。
- `roadmap.md` 已同步阶段状态与结论。
- 需要同步的 `docs/i18n/*/plan/roadmap.md` 已更新。
- 回归记录 / Review Gate 证据可追溯。
- `lint-md`、`docs:check:i18n` 与最小质量门确认已完成。
- 已形成 `Pass / Reject` 结论。

## 5. 跨模块变更评估流程

1.  **提交倡议**: 描述新功能或变更的动机。
2.  **自我评估**: 按照 “第 3 章” 的矩阵进行初排分。
3.  **影响范围审计**:
    -   是否涉及数据库 Scheme 变更？
    -   是否影响 i18n 多语言映射？
    -   是否增加前端打包体积？
4.  **决策备案**: 将评估结果记录于相关设计文档中。
